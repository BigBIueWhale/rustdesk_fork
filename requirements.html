Security Requirements for a Hardened RustDesk Fork

Every connection method, every session type, and every byte of content — encrypted and mutually password-authenticated from the first byte, by the only path the program offers, and secure by assertion, not merely by default. The bar is SSH.

Normative specification for a hardened fork of upstream RustDesk pinned at 1.4.7 (release commit 0c86d46) with its protocol submodule libs/hbb_common at df6badc. Every architectural claim here was verified by direct read of that exact tree (279 Rust files), cited by file and symbol name. RFC 2119 key words (MUST, MUST NOT, SHALL, SHOULD) are normative.

The thesis, in one paragraph

RustDesk's serverless "Direct IP" mode applies no encryption: the login challenge, the salted password hash, keystrokes, screen, files and clipboard cross the wire as plaintext, so the hash is brute-forceable offline on a GPU farm and an active attacker can inject input into an authenticated session or man-in-the-middle it on first contact — without the password. Upstream declined to fix this. This fork keeps RustDesk's excellent Rust + Flutter engine and replaces its cryptographic judgement: it inserts one mandatory, mutually-authenticated PAKE at the single choke point through which all six inbound transports flow, so that all five session types and every auxiliary content channel are keyed before a single application message is parsed; it makes those properties runtime invariants the program refuses to operate without; and it excises the fetch-and-run updater, the unvalidated native-plugin loader, the remote server-list rewrite, and the trust-anchor overrides that have no place on an internet-exposed host. Decisively, the exposed box ships a controlled-only build — the viewer/initiator role, the rustdesk:// deep-link, and the in-process video decoders are compiled out, not merely keyed (R-R2b) — so its reachable mode set is literally one and its only inbound decode is post-PAKE from the already-authenticated peer; the operator's own devices run the full viewer build, where the connecting side is held to the same keyed-before-any-message bar as the responder (R-S13). The result is meant to be, on the wire, as defensible as SSH — the one residual where SSH still leads, a viewer build decoding attacker-reachable peer media in-process, is documented (Appendix C #2b) and kept off the exposed box by the split.

1. Purpose, scope & the SSH bar

The fork's objective: reaching a controlled machine by its own address is encrypted and mutually password-authenticated from the first byte, on every connection method and every session type, by the only code path the program offers — and the program asserts these properties and aborts rather than operate without them.

The bar is SSH. Concretely, the fork must match SSH on the channel and authentication properties that matter and may exceed it. One honest calibration up front: because the fork authenticates by password, the apples-to-apples comparison is SSH password authentication — SSH public-key auth carries far more credential entropy, which R-S9's strong-password requirement only partly offsets.

Three scoping decisions are load-bearing and non-negotiable:

  • All connection methods and all session types. Six inbound transports, five session kinds, and every auxiliary content channel (§4) — not just screen control.
  • Secure by assertion. Every property below is backed by a runtime, build-time, or test-time check that fails closed (§9).
  • Both roles, split by build profile. Upstream ships one cross-platform binary that is viewer and controlled side, selected at runtime by argv. The fork keeps that binary for the operator's own devices (the full viewer build) but ships the internet-exposed box a separate controlled-only build with the viewer/initiator role compiled out (R-R2b) — so both roles stay in scope, but as two artifacts, and the exposed box's reachable mode set is one. All shipped platforms are in scope (§3, §12, §16).

Licensed, like upstream, under AGPLv3; corresponding source for every published binary MUST be published.

2. Threat model

Adversary: an active attacker on the network path (café/transit Wi-Fi, ISP, any transited AS, or a host already on the LAN) who can read, drop, modify, replay and inject packets and can initiate connections to the controlled machine's listening port; and, separately, a malicious or compromised rendezvous server the operator might otherwise rely on. The adversary does not know the password. The controlled machine is reachable from the internet (home-router DMZ on a static address). We defend the channel and the host's exposed surface; we do not defend a compromised endpoint or an adversary who knows the password. Both roles are in scope (§1), but split across two builds: the viewer build (the operator's own devices) also initiates connections, so the adversary additionally includes a malicious peer the operator is induced to connect to — a poisoned or mistyped address, a QR code, or a rustdesk:// deep link — against which the viewer build holds the initiator to the same keyed-before-any-message bar as the responder, and treats every byte a connected peer returns as attacker-controlled (§4.4). The internet-exposed box runs the controlled-only build (R-R2b), which never initiates — so this initiator adversary does not reach the exposed host at all; that surface is absent by construction there, not merely keyed.

Failure mode A — offline-crackable credentials

Upstream sends salt+challenge in cleartext and the answer is a fast, unstretched SHA-256(SHA-256(pw‖salt)‖challenge); captured once, brute-forced offline on a GPU farm. Online limits are irrelevant. No credential material may be recoverable from the transcript.

Failure mode B — passwordless MITM & injection

No channel encryption ⇒ no message authentication. Upstream checks the password once, flips self.authorized=true (connection.rs), then trusts every subsequent frame; an active attacker injects input into the authenticated stream, or MITMs first contact, without the password. Authorization must be a property of a mutually-authenticated, encrypted channel.

Failure mode C — host expansion (pivot, RCE, re-homing)

An exposed controlled host must not be a launch point. This fork additionally treats the updater, the plugin loader, the server-pushed config rewrite, the rustdesk:// deep-link writes, and the os_login → PAM desktop-session-start as parts of the attack surface to be excised (§8) — these are reachable by the network or a malicious rendezvous server without the password, so they are squarely in scope. (The file system, reachable only by an already-authenticated peer, is deliberately a single unconfined, full-filesystem mode per R-S8 and §2 — confining it would mean defending against a password-knower, which we do not, and a confinement toggle is itself forbidden by R-S12.)

The single choke point: all six session transports construct a Stream and hand it to create_tcp_connection (src/server.rs), which today calls stream.set_key() for the secure case before Connection::start. Default serverless exposure is one port (21118), and only if direct-server=Y. The hardened box's reachable surface is therefore exactly this single v4-only port (R-D3, R-D5) — understood and accountable, not merely firewalled.

4.2 The pre-authentication message surface

"Pre-auth" = processed while self.authorized==false. The gate is the single } else if self.authorized { at connection.rs; only five message arms execute before it (one benign), plus per-connection setup in on_open:

  • on_open: check_whitelist is default-OPEN (empty list ⇒ every source IP passes); emits Hash{salt,challenge} unconditionally (a pre-auth oracle, by design).
  • Misc/CloseReason: deliberately placed ahead of the gate so a peer can close pre-auth — benign: logs the reason and tears down the session, parses no other attacker data, grants nothing.
  • LoginRequest: the highest-value pre-auth surface — stores the full request, parses OptionMessage, performs a trusted-device 2FA bypass keyed on attacker-supplied hwid/id, and on some branches starts the desktop/CM session — notably a PAM-authenticated X-session start driven by the attacker's os_login credentials (linux_desktop_manager.rs; a second, root-context credential path excised by R-X14) — and sends a Telegram 2FA push before the password is checked.
  • Auth2fa: TOTP check; persists a trusted device from attacker fields on success.
  • TestDelay: a free pre-auth echo/reflector.
  • SwitchSidesResponse: an authorization path that bypasses the password on a matching 10-second UUID, and sets from_switch which also skips 2FA.

Direct-path note: the SignedId/PublicKey key-bootstrap parse is gated if secure (server.rs) and the direct listener passes secure=false, so on the direct path that parse — and AddrMangle::decode — do not run pre-auth (both are mediator/KCP/relay-only); the only attacker-byte parsers before keying are the frame codec and the CPace protobuf (R-P14). Separately, lan.rs parses unauthenticated UDP-broadcast datagrams and replies with host/user info — a distinct pre-auth parser + info-leak, excised by R-X5.

Pre-auth DoS — unbounded frame length

The frame codec's max_packet_length defaults to usize::MAX and set_max_packet_length is called nowhere in the tree (verified). The length header allows ~1 GiB frames; buffering grows with the declared length. Any peer that reaches a listener can force ~1 GiB/connection memory amplification before authentication.

4.3 What rides the keyed stream — and the two escapes

Verified: every byte of peer-bound content, for all five session types, rides the single keyed conn.stream as a protobuf Message — remote control, file transfer, view-camera, terminal, clipboard text and CLIPRDR file bytes, whiteboard input, remote-printer jobs (streamed as file Blocks), and audio/voice. Auxiliary "IPC" channels (CM, audio, whiteboard, portable-service shared memory) are localhost privilege-separation only and never touch the network. So one PAKE at the choke point covers all of it — with exactly two exceptions:

  1. Port-forward / RDP tunnel payload. After login, both ends call set_raw() (connection.rs server; port_forward.rs client), which drops the secretbox (tcp.rs) — so the tunnelled RDP/VNC/SSH bytes cross the link in plaintext, even though the auth was keyed.
  2. File-transfer path confinement. File names are well-validated (fs.rs rejects .., null, absolute, UNC, and symlink components), but the destination base directory is attacker-supplied and unconfined — an authenticated peer may write/overwrite/rename/recursively-delete anywhere the service user can reach (~/.ssh, /etc, C:\Windows). There is also an acknowledged symlink TOCTOU (path-validated, then opened by path; the openat/O_NOFOLLOW fix is deferred upstream).

4.4 The initiator's (viewer's) outbound surface

The same binary is also the viewer, and §1 puts both roles in scope — so the connecting side is audited symmetrically, not assumed safe. These surfaces live in the viewer build (the operator's own devices); the internet-exposed box runs the controlled-only build (R-R2b) in which the entire initiator role — mod client, the rustdesk:// deep-link, and the in-process video decoders — is compiled out, so on the exposed host they are absent, not merely keyed. Verified in the cloned tree, the initiator carries four surfaces the responder analysis (§4.1–4.3) does not cover, and the spec's keying requirements (§7) act on each in the viewer build:

  1. An unkeyed direct-IP path — the initiator twin of failure modes A & B. When the viewer dials a bare address, Client::_start returns a live stream with no keying for both the IP case (client.rs, is_ip_str) and the domain:port case — both return before secure_connection is ever reached, and the only client-side is_secured() read (io_loop.rs) merely sets a UI lock-icon, never aborts. So the flagship serverless mode (§17: a client reaches <static-IPv4>:21118) runs plaintext on the connecting side: the login hash is captured for offline cracking and first contact is MITM-able, exactly as on the controlled side. Keying the responder (R-S1) alone does not close this.
  2. A wide pre-key decoder. On the rendezvous/relay path the viewer calls Message::parse_from_bytes over the full application union before keying (client.rs, ahead of conn.set_key), on frames whose length is unbounded (set_max_packet_length is never called on the client path) — the initiator-side twin of the responder's wide-decoder + ~1 GiB DoS surface (§4.2).
  3. The kcp-sys C parser, linked for the viewer. mod kcp_stream is ungated (lib.rs) and the viewer reaches the C reassembly parser pre-key via KcpStream::connect (client.rs, from udp_nat_connect) — so excising the mediator (R-D4) does not by itself remove KCP from the connecting role.
  4. The deep-link auto-connect. rustdesk://<id>?password=…&/r@<server>?key=… initiates an outbound session with no confirmation, honoring an embedded password, relay, and — through the Rust initialize (client.rs, written into other_server) — an embedded trust-anchor key, delivered over four un-authenticated transports (Linux D-Bus, Windows WM_USER+2, macOS _url IPC, Android intent).

Beyond keying, a viewer induced to connect to a hostile peer feeds attacker-controlled bytes into the in-process native decoders — libvpx/aom/FFmpeg/libyuv (video), libopus (audio), zstd (cursor/clipboard/file blocks), and the Windows CLIPRDR C parser (the io_loop.rs dispatch). The PAKE authenticates the peer but cannot make a password-correct-but-hostile box safe; this residual is bounded operationally (deliberate-connect + channel restriction, R-S13/R-X6), not cryptographically, and is recorded as such (Appendix C #2b). This residual is confined to the viewer build — the exposed controlled-only build (R-R2b) compiles out mod client and the video decoders, so the broad attacker-reachable decode surface is absent on the exposed host; its only inbound decode is post-PAKE opus/zstd from the already-authenticated peer (§13), the narrow surface §2 already accepts.

5. Definitions

PAKE. A protocol by which two parties sharing only a password derive a strong, mutually-authenticated session key with no certificates or pre-shared keys; a network attacker without the password cannot complete it, cannot MITM it, and learns nothing offline-crackable — reduced to one online guess per attempt. Balanced = symmetric (both hold the same password) = the "me and my own machine" case.

Mutual key confirmation. After deriving the shared key, each side sends a MAC (keyed by that shared key, over its own handshake message) and verifies the peer's — proving both derived the same key, i.e. both knew the password. One-sided proof is not authentication; it is exactly the hole that sank the prior community fork (a constant b"direct-ok" reply). Exact construction: R-P3.

Transcript binding. Tying the derived keys to everything negotiated — both ephemeral public values, the session id sid, and the protocol/version label carried in the channel identifier CI — so no part can be spliced, downgraded, or replayed cross-protocol. (There are no long-term identities to bind; see R-P5.)

The five session types. The protobuf LoginRequest.oneof union (and the server enum AuthConnType{Remote,FileTransfer,PortForward,ViewCamera,Terminal}): remote control, file transfer, port-forwarding (incl. tunnelled RDP as PortForward{host:"RDP",port:0}), view-camera, remote terminal. The kind is read after login — which is why the stream must be keyed before it.

Excise. Remove from the build so the path cannot be reached, configured back on, or accidentally re-enabled — stronger than "disabled by default."

6. Functional requirements (preserve)

R-F1MUST Preserve all five session types (terminal excised from the exposed default — R-X8)
Remote control, file transfer, port-forward (incl. RDP), view-camera, terminal — all preserved as code and all secured (§7). Caveat (terminal): post-PAKE the terminal spawns a shell as the service user — root on the exposed box (R-D3) — with no Linux re-auth (should_use_terminal_os_login_scope is Windows-only, connection.rs; the Linux token path is a stub), so one PAKE password yields a direct root shell — more permissive than the bar this spec holds itself to. Defaulting enable-terminal OFF is not enough (it is a runtime-writable option a password-knower can flip back, exactly what R-S12 forbids — and it defaults ON upstream): the shipped direct-only default build MUST excise the terminal entirely (R-X8). Preserving the code satisfies "all five types" for an attended, non-exposed build; the exposed deployment ships it absent — the same split R-D4 applies to the mediator.
R-F2MUST Preserve GUI file transfer, session persistence, parallel & multi-machine connections
Drag-and-drop transfer; reconnect to the same live desktop; phone + laptop at once; more than one controlled machine.
R-F3MUST Preserve "connect by address, from anywhere, any device"
No server, account, VPN, or pre-shared key may become required to connect.
R-F4MUST The direct port is a single pinned compile-time constant (21118)
The direct listener binds one TCP port, pinned at compile time to the literal 21118 (a const DIRECT_PORT: i32 = 21118 — not the inherited RENDEZVOUS_PORT + 2 derivation at rendezvous_mediator.rs, which would silently shift the port and desync the §10.4 CI KAT be16(21118)=527e if RENDEZVOUS_PORT ever changed) — not a runtime option, env var, or config key. It folds into the PAKE channel binding CI (R-P1), so it is a one-mode constant, not a knob; an operator who needs a different port changes the constant and rebuilds (a build-time choice, never a runtime mode). So the host's "one audited port" discipline holds, and no dynamic UDP range is needed for the serverless path.

7. Security requirements (the core)

R-S1MUST An unconditional mutual PAKE at the single choke point is the only way to connect
The PAKE (§10) MUST run inside create_tcp_connection (server.rs) before Connection::start, keying the stream for all six transports of §4.1. The secure parameter MUST be deleted (not honored) — note the direct path passes secure=false, so a naïve "encrypt if secure" would skip the very path we care about; and the relay path's secure is attacker-controlled (rr.secure) and MUST be ignored. In the viewer build, the same bar binds the initiator (R-S13): the viewer's unkeyed direct-IP and domain:port early-returns (client.rs, §4.4) MUST be deleted and every viewer initiation routed through the CPace initiator handshake (R-P14) before any Message is parsed — keying the responder alone leaves the connecting side plaintext. The exposed controlled-only build (R-R2b) compiles the initiator role out entirely, so there the connecting side is absent, not merely keyed. The plaintext path, the unverified-trust escape hatch, and the manual fingerprint ceremony MUST be deleted.
R-S2MUST Authorization collapses into the PAKE — one keyed fact, not a re-gated legacy gate
Since the CPace PRS is the permanent password (R-P1, R-S9), a successful mutual key-confirmation (R-P3) already proves both sides know the password before Connection::start. The fork therefore collapses the inherited authorization state machine into the CPace one rather than re-gating it: self.authorized=true (connection.rs) MUST be set only on the CPace KEYED success edge (R-P14a), and the inherited post-keying second authentication — the Hash{salt,challenge} emission (on_open), the validate_password/verify_h1 challenge-response, the 30-s is_recent_session re-admit, the SwitchSidesResponse UUID password-bypass with its 2FA-skip, and the CM operator-accept edge ipc::Data::Authorize → send_logon_response_and_keep_alive (arm, call, a grant with no LoginRequest password check) — MUST be deleted or, where retained on the attended viewer build's controlled side, sound only because it is downstream of the PAKE (the headless controlled-only build compiles the CM out, R-R2b, so the arm is absent there), never re-gated. Re-gating would leave several redundant grant paths into, each a separate gate to implement correctly on a 6,000-line file whose on_message chain is pre-auth by default; the collapse leaves one. The challenge-response proves the same secret CPace already proved — two locks keyed alike are one lock — so deleting it loses no assurance (R-S6). What survives: LoginRequest is kept only for its non-credential payload (the session-type oneof → AuthConnType, OptionMessage, version); its password/os_login/hwid fields are dropped (R-X7/R-X14). TOTP is excised, not kept: the second factor (Auth2fa) is enrollable only through the GUI ceremony (generate2fa/verify2fa, auth_2fa.rs), which the headless controlled-only build compiles out (R-R2b), and R-D6 pins the 2fa option empty — so require_2fa is permanently None and the entire Auth2fa + trusted-device subsystem is un-instantiable dead surface (the OTP/terminal pattern, §8). Keeping it would also re-admit a KEYED-but-not-yet-authorized sub-state — a second-factor gate sits between keying and authorization — contradicting this requirement's own single edge into authorized (R-A2). So TOTP, require_2fa, the Auth2fa arm, the trusted-device store, and the totp-rs dependency are excised with the rest (R-X7) — leaving authorized == KEYED literally true. A standalone second factor is out of scope for this single-owner box; CPace + the strong permanent password (R-S9) is the bar.
R-S3MUST Every post-handshake byte is encrypted + integrity-protected + anti-replay
The PAKE-derived per-direction keys (R-P2) MUST drive the secretbox layer for the whole connection; each frame carries a Poly1305 tag and a monotonic sequence; any auth/sequence failure aborts (fail-closed).
R-S4MUST All five types + all auxiliary content channels are covered
Keying the single stream before the oneof type is parsed covers remote/file/port-forward/camera/terminal and the clipboard-text, CLIPRDR file bytes, whiteboard input, printer, and audio channels (all verified to ride conn.stream). The fork MUST NOT implement the handshake per-feature.
R-S5MUST Seal the port-forward / RDP set_raw() escape
The key-strip at connection.rs and port_forward.rs MUST be removed so tunnelled bytes stay inside the secretbox — or the forward MUST be refused. There is no opt-in plaintext mode — a default-off-but-reachable raw tunnel is exactly what the §1 bar ("the plaintext path is deleted, not defaulted off"), R-S12, and acceptance criterion 3 ("the set_raw escape is sealed") forbid. Initiator note (viewer build): removing the client-side strip (port_forward.rs) is necessary but not sufficient — the viewer MUST also assert the stream it tunnels was PAKE-keyed (is_secured(), R-S13), since the inherited direct-IP initiator path produced an unkeyed stream in the first place (§4.4); a sealed strip over an unkeyed stream still tunnels plaintext. (The exposed controlled-only build has no initiator — R-R2b — so this note is moot there; the server-side seal above still applies.) This one in-band plaintext escape is closed, not toggled.
R-S6MUST No offline-crackable credential material on the wire
The balanced PAKE provides this structurally — the transcript reveals nothing brute-forceable and the attacker is reduced to one online guess per interaction, bounded further by the per-IP rate limiter (R-S10 — which the fork MUST re-key onto the PAKE outcome, since the existing limiter sits on the now-removed password path) and the required strong password (R-S9). The upstream fast SHA-256 challenge/response (the Hash{salt,challenge} emission and validate_password/verify_h1) MUST be deleted outright — R-S2 collapses authorization into the PAKE, so this redundant second proof of the same secret is removed, not merely demoted from the security boundary (keeping it would leave dead pre-auth code and a live pre-auth Hash oracle). No memory-hard password pre-hashing is used or needed: a balanced PAKE has no stored verifier for an Argon2/scrypt to protect (that is its only job), and a per-user salt cannot be agreed in a serverless symmetric setting — so the PRS is the normalized password directly (R-P1). Closes failure mode A.
R-S7MUST Cap frame length before authentication
The fork MUST call set_max_packet_length to a small bound — ≤ 4 KiB pre-PAKE (only the ≤120-B CPace messages of R-P14 flow before keying) — on every inbound connection before the PAKE completes, and a sane ceiling after — ≈ 32 MiB (far below the variable-length header's 1 GiB ceiling, well above the 128 KiB file block at fs.rs and any realistic video keyframe; the inherited 256 KiB speculative-prealloc cap at bytes_codec.rs bounds only the initial reserve, not the declared length), closing the ~1 GiB/connection pre-auth memory-amplification DoS (§4.2). Pre-PAKE protobuf is the only attacker-reachable parser before keying — and it MUST decode only the dedicated Cpace message (R-P14), via parse_from_bytes::<Cpace>, never the full application Message union — so the reachable decoder is the handshake's fixed fields, not the whole schema (minimize what is decoded, not merely what is acted on). The kcp-sys KCP packet parser is dropped in the direct-only default build: KCP is driven solely by the rendezvous mediator (rendezvous_mediator.rs), which R-D4 compiles out — and the direct-only build gates mod kcp_stream out entirely (no KCP in either role, since both connect over TCP), so that C parser is absent for both roles, not merely minimized. This cap is the load-bearing pre-auth DoS control: because set_max_packet_length is currently never called anywhere in the tree, the ~1 GiB/connection amplification is the dominant pre-auth memory-DoS — a larger exposure than the per-IP limiter (R-S10) addresses, and it fires per-connection regardless of source IP, so the frame cap is primary and the limiter only a secondary flood-shed. Post-key inflation twin: the receive-path decompress is zstd::decode_all with no output bound (hbb_common/src/compress.rs) — the frame cap bounds the input block but not the decompressed size, so a small compressed file-block, clipboard, or cursor payload from a keyed peer amplifies to an unbounded allocation/disk-write on either role. The fork MUST cap decompressed output (a streaming bound or set_max_output_size) on every receive decompress — the post-key twin of this pre-auth frame cap.
R-S8MUST Close the symlink TOCTOU; file transfer is a single full-filesystem mode (no confinement toggle)
File names are already well-validated (fs.rs rejects .., null, absolute, UNC, symlink components); only the destination base is attacker-supplied. Per §2 the authenticated peer holds the password and is the trusted owner — and remote control already drives the box at the service privilege — so file transfer legitimately operates at that privilege (root on the exposed box) with full filesystem reach, as a single mode; this capability MUST be documented. The fork MUST NOT add a confinement option (a configurable allowed-root): a security-relevant toggle is exactly what R-S12 forbids — it is state that can be misconfigured, it would regress the legitimate "put a file anywhere on my own machine" use, and confining file transfer while the root desktop stays reachable is pointless (the peer has root either way). What is not optional is write-path safety: every write MUST use handle/descriptor-based no-follow opens (openat + O_NOFOLLOW / Windows equivalent) to close the acknowledged symlink TOCTOU (§4.3) — defending the peer's own writes against a local symlink swap, a correctness guarantee asserted on every write (R-A5), not a confinement feature. Consistency with R-X8: the discipline that excises the terminal is toggle-and-surface removal (a default-on flippable path + redundant code), not capability denial — so keeping file transfer as one full-FS mode while adding no confinement knob is the same principle, not an exception to it. Both build profiles: the no-follow write guarantee binds the viewer build's receive writes too — a hostile peer the viewer connected to can race a local symlink on the viewer's download directory (the same TOCTOU, symmetric), so R-A5's per-write O_NOFOLLOW assert applies to the viewer build's receive path as well as the controlled-only build's, not the controlled side alone.
R-S9MUST Default-deny the source whitelist; strong permanent password, permanent-only + password approve-mode
check_whitelist MUST be inverted to default-deny on an exposed host (an unset or all-unparseable whitelist blocks, not passes — a compile-time const at connection.rs, with an explicit 0.0.0.0/0 entry the auditable connect-from-anywhere opt-out; R-S16(e)). The fork MUST require a non-empty permanent password — non-empty after NFC normalization (R-P1), since CPace has no empty-PRS guard (it derives a valid generator even from b"") — and pin verification-method = use-permanent-password (the one-time-password path is excised — R-X7) and approve-mode = password (not click-to-accept) in the controlled-side policy table (R-S16). The PAKE's PRS is this permanent password (R-P1). Because CPace authenticates from the password directly (the PRS is the normalized permanent password, R-P1) and the legacy validate_password salted-hash check is deleted (R-S2/R-S6), the controlled side MUST hold the password in a form usable as the PRS — i.e. password-equivalent material at rest, not a one-way salted hash. Rotation & liveness: making the credential PRS-usable is a real storage-format change to set_permanent_password (upstream stores h1 = SHA-256(pw‖salt) in two places — the local store and a signed-custom.txt preset in HARD_SETTINGS that validate_password falls through to, R-S16(d)(v); the preset is empty-by-signed-gate on a fork build, so the fork holds one PRS-usable local credential); the responder MUST derive the CPace PRS by a live read of the current at-rest credential on every handshake — no cached PRS, generator input, or derived scalar may outlive a password change — and a change (via --password, R-D2) MUST take effect on the next connection with the old value rejected promptly. The config file MUST therefore be treated as secret by filesystem permissions (its at-rest wrapper is keyed by the machine UUID — obfuscation, not secrecy against a local reader). Balanced-PAKE consequence: the controlled side must hold a password-equivalent value (the PRS) to run CPace, so a config read yields password-equivalent material — inherent to a balanced PAKE and acceptable here because endpoint compromise is out of scope (§2). An augmented PAKE (e.g. OPAQUE) would let the controlled side store a non-password-equivalent verifier instead, but reintroduces a stored-verifier and asymmetric initiator/responder model — deliberately rejected for this symmetric, single-user, sovereign design.
R-S10MUST Constant-time secrets; a DoS-safe rate limiter, re-keyed onto the PAKE
Every secret-touching comparison MUST be constant-time (upstream already uses constant_time_eq for the password — MUST NOT regress to ==). The per-IP failure limiter (LOGIN_FAILURES, connection.rs) MUST be kept and re-keyed onto the PAKE, but billed honestly: it is cheap CPU-flood shedding, never credential defense (the strong password + one-guess-per-PAKE is that), so it MUST be tuned loose. Cardinal principle — the limiter must never become the attacker's weapon against the owner: it MUST key on the connecting source and MUST NOT be global or connection-count-wide, so a flood from one source can never lock the legitimate owner (a different source) out of their own machine; and every block MUST decay (see (c)) into a self-healing tarpit, never a permanent ban — a limiter that denies the owner service is doing the attacker's job for them. Direct-IP is the clean case for this: the box keys on the peer's real TCP source address (the accepted socket's addr in the lifted direct_server), not a relay-collapsed one. As inherited the limiter is not DoS-safe, so three coupled fixes are mandatory:
  • (a) Re-key onto the PAKE; gate before the scalar-mult. Inherited, the counter was bumped/checked only inside the LoginRequest password path (around validate_password — checked, bumped); R-S2 deletes that whole path, so the limiter no longer lives there at all — it moves wholesale to the choke point, keyed on a CPace key-confirmation failure (R-P14c). A failed key-confirmation MUST increment the entry, and the choke point (server.rs, peer IP in hand) MUST check the threshold and reject before running the PAKE, shedding floods ahead of the scalar-mult. This is the limiter's real job — cheap CPU-flood shedding, not credential defense: the strong password (R-S9) plus one-guess-per-PAKE is the structural brute-force defense, so the limiter may be tuned loose. Two implementation notes: the limiter is a Connection method keyed on self.ip, but the choke point has no Connection yet — it MUST be lifted to a free function taking the raw peer IP; and only a CPace key-confirmation failure may count (a wrong password or a MITM), never a transport/parse abort, so handshake noise cannot inflate the counter.
  • (b) Bound the memory. Verified: LOGIN_FAILURES only ever grows — an entry is removed only on a successful login; no size cap, no LRU, no time eviction — and it stores a per-full-address entry even for IPv6 (it blocks by /64–/48 prefix yet still inserts the full address). An attacker owning one IPv6 /64 (264 addresses) thus grows it without bound — a memory-exhaustion DoS — and counting pre-PAKE widens that to cheap pre-auth garbage. The fork MUST bound the structure (a hard size cap — e.g. ≤ 8192 entries — plus LRU/time eviction; the inherited store is a never-evicted [Arc<Mutex<HashMap<String,(i32,i32,i32)>>>; 2] — a pair of maps, password and 2FA paths — at connection.rs) and key IPv6 by prefix (≤ /64), never the full address. On the v4-only box (R-D5) this entire IPv6-prefix subsystem is dead code — get_ipv6_prefixes returns None for v4 peers, and try_into_v4 (called at connection.rs, defined in hbb_common/src/lib.rs) normalizes v4-mapped addresses first — so the limiter collapses to a bounded, decaying, per-v4-address counter and the /64 amplification vector cannot occur; drop the prefix machinery on that build.
  • (c) Decay the block; never lock out permanently. Verified: the cumulative block (failure.2 > 30 per IP, > 100 per /48) is a lifetime counter that never decays — once tripped, that source/prefix is blocked until a successful login or a restart. That is a permanent-lockout weapon against any shared egress (CGNAT or other shared mobile/NAT egress) or a shared /48. The fork MUST window/decay the counter (a tarpit that expires) so legitimate and collateral sources self-heal. The inherited thresholds are cumulative, never-decaying lifetime counts — >30 per IP, >60 per /64, >80 per /56, >100 per /48 (connection.rs) — atop a per-minute burst shed (>6/min); keep the per-minute shed, but make the cumulative count decay (e.g. clear an entry after ~60 min with no new failure) so it is a tarpit, not a lifetime ban.
Topology note: the controlled box keys on the connecting source IP, and the legitimate viewer almost always connects from a different source than an attacker (and TCP source cannot be spoofed through a full handshake) — so the common case is an attacker throttling only itself; (b) and (c) defang the residual shared-prefix/CGNAT collateral. Direct-IP mode is the clean case for per-IP limiting — the peer's real address is visible (no relay IP-collapsing), so a bounded per-IP counter is genuinely meaningful there; conversely, if ever deployed behind IP-collapsing infrastructure (a relay), add an application-level identity to the failure key. Observability gap to close: upstream emits no log line on a failed login or a rate-limit block (the wrong-password branch only bumps the counter and sends a protobuf error; post_alarm_audit is dead with no api-server), so the fork MUST emit a structured key-confirmation-failure record (source IP + outcome) at the choke point — the only audit signal on a serverless box, and the prerequisite for an optional host-level fail2ban backstop, which (being post-hoc and log-driven) complements but cannot replace the in-process pre-PAKE shed.
R-S11MUST The local IPC and service are not bypasses; least privilege
The UI⇄service IPC MUST stay local-only with its parent-dir hardening intact; no network peer may reach session functionality without the PAKE. The controlled-side service MUST run least-privilege (failure mode C). The daemon IPC handler is itself a config-integrity boundary: the main channel today writes any config key via Data::Options→set_options, the permanent password via Data::Config, and the whole config via Data::SyncConfig with no allowlist (ipc.rs) — so a local process could reverse the §8 excisions and §9 hard-pins from inside. The fork MUST allowlist the keys writable over IPC and reject trust-anchor / credential / hard-pin writes there (the R-S16-pinned keys are already rejected at is_option_can_save per R-S16(c), so the allowlist need only cover the residual non-pinned keys; the world-facing _service socket already restricts writes to only SyncConfig; the main channel restricts nothing). On Windows that main pipe is created allow_everyone_create and accepts any same-session process (ipc.rs, auth.rs) — the fork MUST also tighten it to a same-user DACL. The allowlist MUST be applied per write-arm, not just to the set_options key set: the same main channel also exposes Data::Socks → set_socks (ipc.rs, a proxy that re-routes all traffic — a local MITM primitive an Options-key allowlist would miss), the salt / unlock-pin / id+set_key_confirmed(false) sub-keys of Data::Config, and Data::RemoveTrustedDevices/ClearTrustedDevices. Make it a positive chokepoint, not a denylist: route every IPC config write through a single allowlist function and CI/AST-assert that no Config::set*/set_socks/set_permanent_password is reachable from the main-channel handler except through it — so the handler cannot write a key that is not on the list, rather than the spec having to enumerate every dangerous arm. Beyond config writes, two action classes on the main channel escape a key-allowlist and MUST be handled separately: (i) the mediator-control arms Data::TestRendezvousServer (ipc.rs → test_rendezvous_server, an outbound probe) and Data::Deployed (RendezvousMediator::restart) write no config key, so the allowlist never sees them — they MUST be removed with the mediator (R-D4) or gated out of the handler; and (ii) the RAII Drop guards CheckTestNatType (constructed at ipc.rs, and in set_socks/set_options) and CheckIfRestart, whose Drop fires test_nat_type() (binding an ephemeral STUN UDP socket — an egress IOC, R-D6) / restart() at arm entry, before any allowlist check runs. An allowlist cannot stop a Drop that constructs on entry, so the fork MUST make the trigger unreachable instead: pin socks and allow-websocket immutable (R-D6) so CheckTestNatType's is_direct can never change between construction and drop, and excise CheckTestNatType/CheckIfRestart with the mediator (R-D4). R-A6's reachability guard MUST traverse Drop impls reachable from the handler — a name-only grep misses them. (Upstream 1.4.7 already gates the IPC transport — per-uid 0600 sockets + SO_PEERCRED — so this is the residual write-authorization gap, not the socket itself.)
R-S11aMUST Executable-match is never the sole IPC authorization; fresh credentials; no /tmp adoption
Verified: the cross-uid IPC accept gate combines a kernel-attested peer credential (SO_PEERCRED uid: root-or-active-user, auth.rs) with an executable-path match (/proc/<pid>/exe equals the service binary, auth.rs). The exe-match alone is not an authorization — any local user may run the genuine root-owned /usr/bin/rustdesk and satisfy it — so it is only sound behind the uid gate, and CI MUST AST-assert that no accept path reaches the exe-match without a preceding uid check (individually weak, jointly sound; a refactor must not drop the uid gate and leave exe-match standing). Two further hardenings the inherited code leaves open: (a) the _service channel MUST authorize against a fresh active-user lookup (as the _uinput_* path already does, uinput.rs), not the service-loop cache, so a just-switched-out user cannot pass in the cache-lag window; and (b) the service MUST NOT adopt a pre-existing world-traversable /tmp/{app}-service it did not create — on owner mismatch it MUST reject-and-recreate (no fchown-adopt), and a startup assertion (R-A4) MUST confirm the parent dir is 0o0711 root-owned with each socket at its expected mode before listening. (The world-mode 0o0666 on the _service/_uinput_* sockets is a deliberate consequence of the root-service ↔ user---server cross-uid design, mediated by these gates — not a permission bug; Appendix B. On the §17 Xorg build the _uinput_* sockets are compiled out entirely — R-X13, since uinput is the Wayland input path — leaving only _service world-mode there.)
R-S12MUST No insecure build/runtime configuration is reachable
No feature flag, env var, config key, or CLI flag may disable/weaken the PAKE, the encryption, the no-follow write guarantee (R-S8), or the §9 assertions. Such controls MUST be removed, not defaulted off. For the controlled side's security-policy options specifically, R-S16 is the realization of this principle: the policy is a compile-time table the Config::get_option funnel returns and the is_option_can_save guard rejects, so no key can be defaulted-permissive or written back on.
R-S13MUST The initiator (viewer) is held to the responder's bar
§1 puts both roles in scope; this requirement governs the viewer build (the operator's own devices), whose connecting side must meet the same keyed-before-any-message guarantee as the responder (§4.4) — the inherited initiator path does not, and keying the responder (R-S1) does not fix it. (The exposed box runs the controlled-only build, R-R2b, which compiles the initiator role out entirely — so on the exposed host this surface is absent, not keyed; (a)–(d) are how the viewer build closes it.) The viewer build MUST: (a) delete the unkeyed direct-IP / domain:port early-returns (client.rs) and route every initiation — direct, rendezvous, relay — through the CPace initiator handshake (R-P14), so a usable Stream is never returned without a confirmed PAKE; (b) assert is_secured() before the viewer dispatches or tunnels any byte — promoting the inherited informational check (io_loop.rs, today only a UI lock-icon) to a fail-closed abort, the initiator analog of R-A1; (c) decode only the dedicated Cpace message pre-key on the client path (parse_from_bytes::<Cpace>, replacing the full-union Message::parse_from_bytes at client.rs), and call set_max_packet_length to the small pre-PAKE bound before the first read (R-S7); and (d) connect TCP-only — gating mod kcp_stream out (R-D4) and removing the udp_nat_connect punch path (client.rs) so the kcp-sys C parser is absent in this role too. Residual (viewer build only; documented, not closable by keying): a viewer deliberately connected to a hostile but password-correct peer still feeds attacker bytes to the in-process native decoders (§4.4); this is bounded by the deep-link confirmation and channel restriction of R-X6, not by the PAKE (Appendix C #2b). It does not exist on the exposed controlled-only build, which carries no video decoders (R-R2b). The TCP-only choice costs UDP NAT traversal — acceptable for the static-IP direct deployment (§17), where connecting-by-address (R-F3) is preserved over TCP.
R-S14MUST Screen capture is bound to a PAKE-authenticated session — a reused grant must not capture outside one
The security boundary on screen capture is the PAKE (R-S1/R-A1), not the OS consent dialog — a distinction that matters because this fork targets unattended access (R-S9). On Android, MediaProjection consent is a system dialog the user must physically tap and which cannot be auto-granted, so the inherited token reuse (MainService keeps the projection alive across session end via reuseVirtualDisplay/setSurface(null), "to avoid media projection confirmation dialog every connection", MainService.kt) is precisely what makes unattended reconnection possible — mandating a fresh per-session consent would break the use case. The requirement is therefore not re-consent but binding: capture MUST begin only inside a confirmed PAKE session — no held grant (the Android projection token, or the Wayland xdg-portal restore-token at libs/scrap/…/wayland/pipewire.rs) may drive capture pre-authentication, and the held grant MUST NOT be reachable by any non-authenticated path. The persisted Wayland restore-token is a local at-rest exposure (a local reader could lift it) in the same §2 endpoint-compromise-out-of-scope class as the UUID-obfuscated config (Appendix C #14); the fork SHOULD encrypt it at rest and release the projection when the service is not running (and on the §17 Xorg build, R-X12 compiles the Wayland capture path out entirely, making this half absent by construction). (Re-prompting per session MUST NOT be added where it would break unattended access. The §17 Xorg box has no capture-consent gate at all — it captures for the authenticated peer by design, the intended unattended behavior — so this requirement is about where the gate is: the PAKE, not an OS dialog.) macOS source: the held-grant twin is the persistent TCC.db Screen-Recording/Accessibility grant — there is no fork-stored token (so the "encrypt at rest" half is N/A), but the grant is app-wide because Release.entitlements ships the App Sandbox off (with an unjustified cs.allow-jit), held by a KeepAlive, login-window-resident --server. Capture is already PAKE-bound on macOS through the shared lazy-on-subscribe path, so the obligations are to keep that binding, minimize the entitlement set (the full per-key allow-list spans Release.entitlements and DebugProfile.entitlements — the latter adds com.apple.security.network.server — enumerated by R-A6's Apple pass), and ensure a --server restart re-enters capture only through a fresh PAKE session (the macOS analog of Android's START_NOT_STICKY). iOS: no capture, so N/A.
R-S15MUST The viewer's in-session message handling is a config-integrity boundary — a keyed hostile peer may not write the viewer's PeerConfig
§1 puts the initiator in scope and R-S13 keys it (in the viewer build; the exposed controlled-only build has no mod client/io_loop client dispatch — R-R2b — so this surface is absent there) — but keying is where the spec stopped: a correctly-keyed but hostile peer (one the viewer was induced to connect to, §2/§4.4) then sends ordinary in-session messages that reach the viewer's persistent PeerConfig. Verified: Misc::BackNotification→PrivacyModeState writes the peer's unbounded impl_key string under privacy-mode-impl-key (io_loop.rs) with no user click; TerminalResponse.service_id and the peer's PeerInfo username/hostname/platform are persisted on every connect (client.rs) — all via save_config. This is the initiator twin of R-S11, whose config-write allowlist is written IPC-only and never inspects the client dispatch. The fork MUST extend R-S11's single-allowlist chokepoint to io_loop.rs: route every peer-message-driven config write through a validated allowlist (privacy-mode-impl-key rejected unless it is in the compile-time get_supported_privacy_mode_impl() set; PeerInfo strings length-bounded), and CI/AST-assert (R-A6) that no Config::set*/save_config is reachable from a peer-message arm except through it. (Severity is config-poisoning, not RCE — the impl map is compile-time — but it is the one persistent-write class the §4.4 audit found that survives a correctly-keyed peer, so it is closed, not accepted as the §2 native-decoder residual is.)
R-S16MUST The controlled side enforces ONE compile-time policy at the config funnel — never a permissive default, a peer/server field, or a runtime write
Principle. The controlled side draws every security decision from exactly two input classes: config options — almost all resolved through the funnel Config::get_option (hbb_common/src/config.rs → get_or; precedence OVERWRITE_SETTINGS → config-file → DEFAULT_SETTINGS → ""), with a handful in a second namespace read via get_builtin_option (closed in (d)(iv)) — and peer/rendezvous protobuf (the control_permissions blob and the LoginRequest credential fields). RustDesk makes every one of these fail open: option2bool returns true for any unset enable-* key (config.rs), verification_method() → UseBothPasswords and approve_mode() → Both on empty (password_security.rs), and permission() honors a rendezvous-server-supplied bit before any local policy (connection.rs). Policy MUST therefore be a property of the artifact — a compile-time constant the program enforces unconditionally — not a default that can be unset, wrong, peer-supplied, or locally overwritten. This requirement supersedes the per-resolver approach earlier requirements implied ("make verification_method()/approve_mode()/Permission::terminal return a constant and never read the option"): those resolvers are left unmodified — the funnel pin makes them return the policy transparently, the single-edit closure rather than a per-call-site rewrite.
  • (a) One source of truth. A #[cfg(feature = "lockdown")] pub const PINNED_SETTINGS: &[(&str, &str)] in config.rs — the idiom RustDesk already uses for RS_PUB_KEY/RENDEZVOUS_PORT (config.rs) — empty when the feature is off. lockdown is orthogonal to R-R2b's viewer feature and is on for every shipped controlled-side profile (the headless controlled-only build always; the viewer build's controlled side per R-D7). It pins only Config server-settings keys (KEYS_SETTINGS); the parallel LocalConfig map (viewer UI prefs) is untouched. The table — opinionated and exact — is the box's entire security policy in one auditable place: 
    verification-method = use-permanent-password   # was: unset → UseBothPasswords (accept either credential)
approve-mode        = password                 # was: unset → Both (silent click-to-accept)
access-mode         = custom                   # neutralizes the "full" → grant-every-capability shortcut
enable-keyboard=Y  enable-clipboard=Y  enable-file-transfer=Y  enable-audio=Y  enable-camera=Y   # the content channels a remote-control box needs (§4.3)
enable-terminal=N  enable-tunnel=N             # escalation + outbound-proxy vectors (R-X8, R-D6)
enable-remote-restart=N  enable-record-session=N  enable-block-input=N  enable-privacy-mode=N  enable-remote-printer=N
                                               # meaningless on a headless root box, or better done out-of-band
enable-trusted-devices=N  2fa=  bot=           # no 2FA-bypass, no TOTP, no Telegram push (R-X7, R-D6)
api-server=  custom-rendezvous-server=  relay-server=  proxy-url=          # egress-silent; the proxy pin MUST use the proxy-url key (R-D6)
allow-websocket=N  allow-insecure-tls-fallback=N  allow-linux-headless=N  # R-D6, R-X14
stop-service=N  allow-only-conn-window-open=                              # service un-killable by a local write (R-X9); never self-DoS a headless box
    The only Ys are the content capabilities a remote-control + file-transfer box needs; every escalation, egress, or headless-meaningless capability is an explicit N. An operator who needs a different policy edits the const and rebuilds (the R-F4 build-time-choice discipline) — never a runtime knob. (direct-server/direct-access-port are not here: R-D4 makes the listener unconditional and R-F4 pins the port as a code literal, not an option.)
  • (b) Pin the read funnel. At the head of Config::get_option (config.rs), before the get_or lookup, return the pinned value if the key is in PINNED_SETTINGS. This makes every config-driven resolver return the policy with no edit to any of them — verification_method(), approve_mode(), option2bool("enable-*") inside is_permission_enabled_locally (connection.rs), is_headless_allowed(), the egress reads — and covers every un-enumerated KEYS_SETTINGS security key for free, because that funnel is singular for the KEYS_SETTINGS namespace — a second option-read funnel (get_builtin_option, the KEYS_BUILDIN_SETTINGS namespace) exists and is closed in (d)(iv). Strictly stronger than RustDesk's own OVERWRITE_SETTINGS, which is a runtime RwLock populated only if a RustDesk-Ed25519-signed custom.txt loads (common.rs) — a fork cannot sign it, and it can be left empty or mutated in-process; a const can be neither.
  • (c) Pin the write funnel. In is_option_can_save (config.rs — the single guard already gating both set_option and set_options/purify_options), return false for any pinned key, reusing the existing drop-and-purge-from-the-persisted-file behavior. A local process, an IPC Data::Options/SyncConfig write (R-S11), or a server-pushed config-merge (R-X3) thus cannot set a pinned option key — structurally, with no allowlist lookup for this subset. Scope: this guards only the options map; the credential and identity are struct fields, not options — set_permanent_password/set_salt/set_id/set_unlock_pin via IPC Data::Config (ipc.rs) and whole-struct Data::SyncConfig bypass is_option_can_save entirely, so those writes are R-S11's per-arm allowlist job, not this guard's.
  • (d) Close the two gates that bypass get_option. (i) permission() (connection.rs) consults the rendezvous-server control_permissions blob before local policy. On the controlled-only build the mediator is excised (R-D4) so it is always None (rendezvous_mediator.rs) and (b) governs the fall-through — but the fork MUST also add, under lockdown, an early return Self::is_permission_enabled_locally(opt) at the head of permission() that skips the control_permissions branch entirely, closing the server-push vector by construction, not merely by the mediator's absence. (On the controlled-only build the #[cfg(feature="flutter")] access-mode shortcut in is_permission_enabled_locally is itself compiled out — flutter is off, R-R2b — so capability resolution is purely option2bool(enable-*, pinned); the access-mode = custom pin is belt for any build where flutter is on.) (ii) The local CM ipc::Data::SwitchPermission handler (connection.rs) reassigns conn.keyboard/clipboard/audio/file/… at runtime, bypassing permission(); once the policy is fixed a runtime re-widener has no place, so the handler arm MUST be compiled out under lockdown (the headless controlled-only build has no CM to drive it anyway; structural, not merely covered by R-S11's allowlist). The peer's inbound disable_* overlays (connection.rs) only ever restrict the cached booleans, so they need no change. (iii) The SOCKS/proxy state is the one structured config that bypasses the string-option funnel: get_socks() reads OVERWRITE_SETTINGS/CONFIG2.socks/DEFAULT_SETTINGS directly (config.rs) and get_network_type() likewise, while set_socks writes the structured CONFIG2.socks field — so neither (b)'s read pin nor (c)'s write guard reaches it. Under lockdown the fork MUST therefore make get_socks()/get_network_type() return None/Direct and set_socks() inert at those accessors directly; pinning proxy-url in the table is belt for any get_option reader but is not sufficient alone. This keeps CheckTestNatType's is_direct constant so the IPC Drop STUN-probe never fires (R-D6/R-S11). (iv) A second option-read funnel exists outside Config::get_option: get_builtin_option (common.rs) reads the BUILTIN_SETTINGS map for the KEYS_BUILDIN_SETTINGS namespace (config.rs), which the (b) pin does not reach. The security-relevant builtin keys read on the controlled path are one-way-file-transfer (connection.rs — the directionality of an authenticated peer's file pull), allow-logon-screen-password (feeding the approve_mode==Click decision), one-way-clipboard-redirection, and enable-perm-change-in-accept-window (ui_cm_interface.rs). BUILTIN_SETTINGS is populated only from the same RustDesk-signed custom.txt a fork cannot produce, so on a fork build it is empty and these take their safe defaults — not a fail-open reachable by a non-password-knower — but it is a real second funnel the pin must account for. The fork MUST (1) assert BUILTIN_SETTINGS is empty at startup (R-A4 — no managed override leaked in), and (2) if it ever pins a KEYS_BUILDIN_SETTINGS value (e.g. one-way-file-transfer=Y to make the root box push-only), mirror the PINNED_SETTINGS consult into get_builtin_option too — the Config::get_option pin alone does nothing for these keys. (v) A third non-get_option store closes the set: HARD_SETTINGS (config.rs), read by is_incoming_only/is_outgoing_only (the conn-type key, gating real behavior at client.rs/rendezvous_mediator.rs) and get_hard_option (exposed to Dart via main_get_hard_option) — and, more pointedly, a second at-rest credential store: get_preset_password_storage_and_salt reads HARD_SETTINGS's password/salt, and validate_password (connection.rs)/has_permanent_password (config.rs) fall through to that preset when the local store is empty. Like BUILTIN_SETTINGS, it is written only by the RustDesk-signed read_custom_client (common.rs, gated on sign::verify with the embedded key), so a fork build leaves it empty — but the fork MUST assert it empty at startup (R-A4) for the same reason as BUILTIN_SETTINGS, which removes the conn-type funnel and the preset-password credential vector by construction (and reconciles R-S9's single-h1 at-rest model, which this preset branch otherwise contradicts). (This is also why reusing RustDesk's existing OVERWRITE_SETTINGS precedence would be insufficient: it too sits only on the get_option path — the decisive requirement is intercepting every read funnel — Config::get_option, get_builtin_option, get_socks, permission()'s control_permissions, the LocalConfig namespace (R-SV10), and HARD_SETTINGS ((v)) — not just the get_option path.)
  • (e) The whitelist is a semantic fix, not a pinned value. Unlike the fail-open options above, the source whitelist can only ever narrow the accepted set (a bad or empty value fails closed), so it safely stays the one operator-settable input besides the password — a deliberate, narrow relaxation of R-D2 justified because it cannot open access beyond the PAKE. Its only defect is the silent default: check_whitelist (connection.rs) currently passes all on an empty list. The fork MUST invert that empty/all-unparseable case to deny via a compile-time const (R-S9), with an explicit 0.0.0.0/0 entry the auditable opt-out — reconciling default-deny with R-F3's connect-from-anywhere: the PAKE is the gate, so wide reach becomes a deliberate, visible choice, never a silent default-open.
The credential storage format (the password-equivalent PRS the controlled side holds) is R-S9/R-P1's concern; (a)'s verification-method pin fixes only the policy that selects it. The single self.authorized = true edge (connection.rs) collapses onto the PAKE KEYED success (R-S2). R-A6 proves the whole mechanism complete — that the keys are pinned, that the funnels consult the pin, and that the resolvers reach config only through Config::get_option (so none is rewritten to a divergent constant).

8. Paths to EXCISE (block / remove)

The audit found several inherited paths that cannot be made safe on an internet-exposed host and add nothing to a serverless fork. These MUST be compiled out / removed so they cannot be re-enabled.

R-X1MUST Excise the auto-updater (fetch-and-run RCE)
updater.rs's check_update downloads a binary from a server-supplied URL (the version-check JSON's url field) with no signature or hash check and executes it privileged (--update / MSI). A second identical path exists in the Flutter UI (the flutter_ffi update-me/download-new-version keys). Whoever controls that response — or DNS/TLS for the version host, or a re-pointed fork — gets privileged code execution. MUST remove src/updater.rs (its mod in lib.rs, desktop-cfg-gated), the --update handler in core_main, and the Flutter update keys. The fork ships its own releases (§12); if update-checking is ever re-added it MUST verify a pinned SHA-256 of the artifact before execution (R-B2) — never fetch-and-run. macOS source twin: the sharpest form is the root DMG path — macos.rs's update_me/update_from_dmg run an osascript … with administrator privileges shell that strips com.apple.quarantine and re-installs as root, and mod updater compiles on macOS — so it MUST be excised on the Apple source too. The version-check egress and the universal cross-platform form are §18 (R-SV2/R-SV3).
R-X2MUST Excise the native-plugin loader
plugin/plugins.rs dlopens any plugin_*.{so,dll,dylib} from a system path and calls its C entrypoints — with the validation left as a literal // to-do; install_plugin_with_url unzips a download with no verification. That is arbitrary native code in-process. It is already double-feature-gated and not in the default build, and is not remotely installable (the PluginRequest message is post-auth and only invokes already-loaded plugins) — but it is dead weight on a hardened fork. MUST never compile plugin_framework, and MUST drop the src/plugin/ tree and the --plugin-install/--plugin-uninstall args so it cannot be re-enabled by accident.
R-X3MUST Excise the remote ConfigureUpdate server-list rewrite
rendezvous_mediator.rs lets the rendezvous server rewrite the client's rendezvous-servers list at runtime and restart — re-homing the client to arbitrary infrastructure. A serverless fork's server list MUST be fixed by local config; make the ConfigureUpdate arm a no-op. Note the twin — and that it is not the mediator's: a malicious rendezvous server can also rewrite rendezvous-servers from inside an unauthenticated TestNatResponse, but the cu field → set_option("rendezvous-servers") + set_serial handler lives inside test_nat_type_() (common.rs), reached by the startup test_nat_type() probe — not by RendezvousMediator. So excising the mediator (R-D4) does not remove it: on the viewer/Android/iOS builds the running app's startup probe still TCP-calls Config::get_rendezvous_server() (default rs-ny.rustdesk.com) and would adopt an attacker-supplied rewrite from the reply. It is removed only when the test_nat_type subtree is made cfg-absent and its startup callers deleted (R-SV4(d)). And a third twin: hbbs_http::sync::handle_config_options (sync.rs) writes arbitrary config keys — including the trust-anchor key, custom-rendezvous-server, api-server, even direct-access-port — from a /api/heartbeat strategy.config_options response, with no key allowlist (inert once mod sync is gated out per R-D4 and api-server is pinned empty per R-D6, but uncaught otherwise). R-A6 MUST grep all three re-home paths (ConfigureUpdate · TestNatResponse/common.rs · handle_config_options/sync.rs) so a partial fix — or a server-assisted rebuild that keeps hbbs_http::sync — cannot leave a twin live.
R-X4MUST Pin the trust anchor; remove the key/server overrides
The public key (trust anchor of the rendezvous-secured channel) and the server set are overridable via the "key" config option, a license/"config string" (which even accepts unsigned JSON — custom_server.rs), a renamed executable (or the RUSTDESK_APPNAME env var that spoofs the exe-name license without renaming, windows.rs), and the registry (common.rs, custom_server.rs, core_main.rs; on Windows the HKLM license read at windows.rs). A malicious installer name or shared config string silently points the client at attacker infrastructure with the attacker's key. The fork MUST bake its own values, make get_key() ignore the runtime override, and remove custom_server.rs + the --config path; it MUST NOT accept the unsigned-JSON branch. The same class includes three ungated CLI paths beyond --config: --import-config <path> overwrites the entire config (incl. trust anchor + servers) from an attacker-suppliable file with no is_root gate (core_main.rs); --remove <path> is an ungated file-delete gadget; and --assign/--deploy/--set-id presuppose a server account this fork removes and leak tokens/passwords on argv. All MUST be excised and CI-grepped (R-A6). macOS · iOS source: the trust-anchor override — get_key() reading the "key" option (the rendezvous anchor, distinct from the per-device get_key_pair()) and overriding RS_PUB_KEY — is live on both Apple platforms (the deep-link ?key= reaches it via mod client); custom_server, --config and --import-config exist on macOS but are absent on iOS (no core_main); the RUSTDESK_APPNAME/renamed-exe spoof is Windows-only, hence absent on Apple. So get_key() MUST ignore the override on macOS and iOS, and the macOS config-import paths MUST be excised on the Apple source.
R-X5MUST Excise LAN discovery; the recent-session re-admit is deleted with the FSM collapse
LAN discovery (lan.rs, UDP 21119) discloses MAC, RustDesk ID, hostname, active username, and platform to any LAN pinger. Per §8's "compiled out / removed" bar (not merely "disabled"), the direct-only build MUST excise the lan.rs UDP listener — absent, not firewalled (cf. R-D4). On Android this is acute: start_lan_listening is hard-forced true (rendezvous_mediator.rs), so the excision MUST extend to the Android build (R-D7a), not Linux alone. The is_recent_session 30-s auto-re-admit (connection.rs) is not a separate weaker path: R-S2's collapse deletes it outright — authorization is the CPace KEYED edge, established fresh per connection, so there is no re-admit path to gate and the SESSION_TIMEOUT window (connection.rs) goes with it. Reconnect simply re-runs the sub-second handshake; there is never an unauthenticated re-admit.
R-X6MUST Excise the trust-anchor/password deep-link writes; confirm the connect deep-link
A completeness sweep of the platform/UI layer found the one cross-trust-boundary surface the Rust-core audits could not see: the rustdesk:// URL-scheme handler — a malicious web page, QR code, or co-installed app can fire it, pre-authentication. rustdesk://config/<b64> writes the rendezvous/relay/api server and the public key (trust anchor); rustdesk://password/<pw> sets the permanent password — both default-OFF behind compile-time flags but with no install+root gate (unlike the CLI), and both are the same trust-anchor / credential-injection class as R-X4. The fork MUST remove the config/server/key/password-writing deep-link authorities entirely. The third path, rustdesk://<id>?password=…&/r@<server>, silently auto-initiates an outbound session to an attacker-chosen peer/relay on all platforms; the fork MUST require explicit user confirmation for any deep-link-initiated connection and MUST NOT honor an embedded password/relay/key (incl. the ?key= trust-anchor injected into the outbound session) without it. The strip MUST be enforced in both layers: the Dart parser (flutter/lib/common.dart handleUriLink) and the Rust core, where the embedded ?key= is adopted into other_server (client.rs) and where the raw URI also reaches the core via bind.sendUrlScheme (the desktop_multi_window child-window path) — a Dart-only fix is bypassed, since the trust-anchor adoption lives in the Rust initialize (client.rs). (flutter/lib/common.dart; client.rs.) The Linux delivery transport is a D-Bus session-bus method (src/server/dbus.rs: bus org.rustdesk.rustdesk, method NewConnection(id)) that ignores the caller (no peer check) and forwards the URI verbatim into the same handler — any co-installed same-session app can invoke it, and it claims the bus name with replace_existing=true (a name-hijack that lets a malicious process intercept legitimate links). The fork MUST remove or caller-restrict NewConnection and drop replace_existing (CI-grep org.rustdesk.rustdesk · NewConnection · start_dbus_server). The Windows and macOS delivery transports are the same un-authenticated local-IPC class — Windows relays the URI by a WM_USER+2 window message (core_main.rs → send_message_to_hnwd), macOS by the _url IPC socket (ipc::send_url_scheme) — and neither authenticates the sender; the fork MUST apply the same caller-restriction where the OS permits and route all of them through the explicit-confirmation gate above, which is the load-bearing control across all four transports. On Android the deep-link is the manifest's rustdesk:// VIEW/BROWSABLE intent-filter on an exported MainActivity, and the manifest also ships an exported BootReceiver with a DEBUG_BOOT_COMPLETED broadcast any installed app can fire (AndroidManifest.xml: MainActivity, BootReceiver); the fork MUST remove the debug broadcast, set those components exported=false where possible, scope the intent-filter, set android:allowBackup="false" (else the UUID-obfuscated config holding the password-equivalent PRS is exfiltratable via adb backup) + a cleartext-deny network-security-config, and minimize the permission set to the dual viewer+controlled minimum (the app is one binary in both roles, R-R2, so it cannot be pared to "viewer-only"). Drop the legacy READ_EXTERNAL_STORAGE/WRITE_EXTERNAL_STORAGE (superseded by MANAGE_EXTERNAL_STORAGE on the pinned targetSdk 33) and requestLegacyExternalStorage. Keep (each maps to a preserved feature) INTERNET, ACCESS_NETWORK_STATE, FOREGROUND_SERVICE (+ the typed FOREGROUND_SERVICE_MEDIA_PROJECTION if targetSdk is bumped to ≥ 34), BIND_ACCESSIBILITY_SERVICE (controlled-side input injection), RECORD_AUDIO, MANAGE_EXTERNAL_STORAGE, POST_NOTIFICATIONS, WAKE_LOCK, REQUEST_IGNORE_BATTERY_OPTIMIZATIONS and RECEIVE_BOOT_COMPLETED (controlled auto-start). SYSTEM_ALERT_WINDOW is the one high-risk permission to drop, and this profile decides it: cut the FloatingWindowService and drop SYSTEM_ALERT_WINDOW — the controlled-status surface is already carried by the mandatory foreground-service notification (MainService.kt), and a TYPE_APPLICATION_OVERLAY window is an overlay/tapjacking primitive a single-purpose box does not need. The inherited floating window is moreover default-on (disable-floating-window is opt-out, == "Y", MainActivity.kt) with cosmetic size/transparency/untouchable/SVG sub-options — all removed with it. Cutting the permission MUST sever, not silently disable, the boot-auto-start dependency that currently requires it (BootReceiver.kt; canStartOnBoot). (If a status affordance is ever retained, it MUST be a single non-configurable minimal form — no runtime variability.) iOS source (the transport this list omits): iOS has no Rust IPC — the natively-registered rustdesk:// CFBundleURLSchemes (ios/Runner/Info.plist) flow OS → FlutterAppDelegate → the uni_links plugin (uriLinkStream/getInitialLink, which does run on iOS — only isLinux is excluded) into the Dart handleUriLink, and the Rust ?key=→other_server adoption (in mod client's LoginConfigHandler::initialize) is reachable there too — so the strip MUST cover the iOS transport in both layers, and the iOS-only config/password deep-link write authorities MUST be removed, not left flag-gated. macOS detail: start_ipc_url_server is a separate listener that bypasses the is_service_ipc_postfix accept gate, so any same-uid process can inject (the cross-uid hole is already closed by the per-uid 0o0700/0o0600 hardening); it MUST authenticate its sender (peer-uid + peer-exe, as the _service channel already does).
R-X7MUST Excise the one-time-password mechanism and the TOTP second factor; pin to the permanent password
RustDesk carries two credentials: the permanent password and an in-memory, rotating one-time password (TEMPORARY_PASSWORD, hbb_common/src/password_security.rs), and the default verification-method accepts either (UseBothPasswords; validate_password tries temporary first, then permanent, connection.rs). A balanced PAKE authenticates from one shared secret, so a second live credential is dead weight at best and a non-PAKE bypass at worst; the one-time password is also attended-only (screen-shown, rotating) and useless for the unattended access this fork targets. The fork MUST excise the temporary-password code (TEMPORARY_PASSWORD, get_auto_password, update_temporary_password, check_update_temporary_password) and hard-pin verification-method = use-permanent-password, making the permanent password the sole credential and sole PAKE PRS (R-P1, R-S9). (With R-S2's collapse, validate_password is itself deleted, so verification-method no longer feeds any auth decision — the hard-pin and its CI-grep persist purely to keep the OTP/temporary-password code absent, not because the resolver is still consulted.) The TOTP second factor is excised on the same logic (R-S2): Auth2fa (connection.rs), the require_2fa state, generate2fa/verify2fa/get_2fa (auth_2fa.rs), the trusted-device store (add_trusted_device, the RemoveTrustedDevices/ClearTrustedDevices IPC arms), and the totp-rs dependency (Cargo.toml) MUST be removed: TOTP is enrollable only via the GUI (compiled out of the headless build, R-R2b) and R-D6 pins the 2fa option empty, so it is un-instantiable dead surface, and a post-keying second factor would re-admit the multi-step pre-authorized state the R-S2 collapse eliminates. Dropping totp-rs also retires a puller of the idna 0.5 RUSTSEC-2024-0421 vuln (Appendix D, R-R3). This also deletes the consecutive-failure temporary-password rotation (connection.rs) — a remotely-triggerable lockout where 10 failed attempts rotate the password out from under a legitimate user, already gated on temporary_enabled(), so it vanishes with the excision. Enforcement MUST be fail-closed and immutable: verification_method()/approve_mode() today read a runtime-writable option and fail OPEN — an unset/unknown value yields UseBothPasswords/Both, re-enabling OTP and click-to-accept (password_security.rs) — and a DEFAULT pin can be shadowed by a written option (the get_or read-precedence at config.rs returns a set value over the DEFAULT; set_option writes) — which is precisely why the policy is pinned at the funnel, not in a DEFAULT tier. The fork pins verification-method and approve-mode in the controlled-side policy table (R-S16): the resolvers stay unmodified, and the Config::get_option funnel returns the policy while is_option_can_save rejects every runtime write — so the funnel pin wins over any written value, the failure these resolvers' DEFAULT-only pin could not prevent. R-S16/R-A6 own the enforcement and its CI proof.
R-X7aMUST Excise the Android approve-mode / verification-method selector UI — not merely fix it off
R-X7/R-S9 pin verification-method = use-permanent-password and approve-mode = password in the controlled-side policy table (R-S16), so the Config::get_option funnel returns the policy and is_option_can_save rejects any write to those keys (R-A6 AST-checks the resolvers reach config only through that funnel). But the Android server-settings UI still presents and writes the multiplicity — selectors for password / click / both and one-time / permanent / both, plus the one-time-password length and the numeric-OTP toggle, via bind.mainSetOption (flutter/lib/mobile/pages/server_page.dart; writers) — gated only by the soft isOptionFixed managed-config lock (config.rs), the exact "defaulted off, not removed" pattern R-S12 forbids. Once the R-S16 pin lands, the funnel overrides any written value and the write guard rejects it, so the UI is a second, divergent presentation of a setting it can no longer change. The fork MUST remove these selectors (leaving only Set permanent password), not disable them — the same toggle-and-surface removal R-X8 applies to the terminal.
R-X8MUST Excise the terminal root-PTY from the direct-only build — not merely default it off
Post-PAKE the Terminal session type spawns a shell as the service user — root on the exposed box (R-D3) — with no second credential on Linux: the OS-login/PAM re-auth (should_use_terminal_os_login_scope, connection.rs) is Windows-only by construction (cfg!(target_os = "windows")), and the Linux fill_terminal_user_token is a stub that yields a root PTY via spawn_command (terminal_service.rs). So one PAKE password buys a direct root shell — strictly more permissive than the bar this spec sets for itself (which authenticates, then escalates, rather than handing out root in one step). Worse, upstream's enable-terminal defaults ON (option2bool treats an unset key as enabled) and the terminal modules compile unconditionally (no feature gate). Defaulting it OFF is insufficient: it is a runtime-writable option a password-knower can flip back — the exact fail-open pattern R-X7/R-S12 forbid. Every other more-permissive-than-the-bar path is excised (the plaintext set_raw tunnel R-S5, the one-time password R-X7, the mediator R-D4); the terminal root-PTY MUST be treated the same. The shipped direct-only default build MUST compile out terminal_service/terminal_helper, refuse the LoginRequest.Terminal arm, and pin enable-terminal = N in the controlled-side policy (R-S16) so the Self::permission(OPTION_ENABLE_TERMINAL, …) gate (connection.rs) resolves false through the Config::get_option funnel — the module excision is the primary closure, the pin the belt — CI-proven per R-A6. The terminal stays a buildable capability for an attended, non-internet-exposed build (its code existence is a non-goal to remove, §14) — but it is absent from the exposed default, the same split R-D4 applies to the mediator.
R-X9MUST Excise the Windows run-mode, elevation, and service-disable plurality — one mode, one privilege transition
The Windows binary is a multi-tool selected by argv, executable name, install state, elevation state, and runtime config — the opposite of "one mode, the correct one." The controlled host needs exactly one: an installed, auto-start LocalSystem service (--service → start_os_service, windows.rs) that launches the session --server via the winlogon-token CreateProcessAsUserW (launch_privileged_process, windows.rs) — the single privilege transition — which launches the same-user --tray. Everything else MUST be compiled out, not gated:
  • Non-service run modes. The portable / un-installed in-process service (start_portable_service, StartPara, portable_service.rs; core_main.rs) and quick-support (set_quick_support, is_quick_support_exe, --quick_support, core_main.rs) are removed — they run the controlled side without the installed-service privilege model.
  • The stop-service runtime toggle. get_create_service honors a runtime-writable Config::get_option("stop-service") (windows.rs) that suppresses the service entirely — a local process (via --option/IPC) can disable the controlled host's service. This is the textbook R-S12 "security-relevant toggle, default-off-but-reachable": the branch MUST be excised and the key removed from the IPC-writable set (R-S11). The installed service is always present and always auto-start.

R-A10MUST PAKE known-answer & transcript-binding tests
The draft's published test vectors (which cover the protocol through ISK), plus a self-consistency KAT for the R-P3 confirmation MAC (both sides compute equal tags; any mutated input — including CI's role/port (§10.4) — flips the tag; no published vector exists for this step), plus negative tests that mutate each transcript element (ephemerals, sid, the version label / CI, AD) and assert the handshake aborts. Add a KAT that the two derived directional keys satisfy k_c2s ≠ k_s2c (R-P2/R-P10) — but this is necessary, not sufficient: add a two-peer end-to-end test driving an actual controlled↔viewer handshake and asserting each direction decrypts only under the mirrored key (responder seals k_s2c/opens k_c2s; initiator the mirror), which catches a role-binding inversion or a single-key regression that the derived-value KAT and the wire-capture test (R-A9) cannot (R-A5). Add a project KAT over the fork-specific CI byte-encoding (no draft vector covers it, R-P1). Add a second project KAT over a full 32-byte production sid (sid_a‖sid_b, R-P1/R-P14): the §10.4 anchor reuses the draft vector's 16-byte sid for reproducibility, but a live session always carries 32 bytes, so the 16-byte KAT never exercises the production sid length. The construction is parametric in sid length (length-prefixed in the generator and ISK; bare-concatenated under the fixed-width 64-byte ISK trailer in mac_key, which stays unambiguous), so a conformant implementation passes both — but the 32-byte KAT is what stops an implementer hard-coding a 16-byte sid buffer (§10.4 pins both anchors). Add a non-ASCII PRS KAT exercising the password normalization (R-P1): a combining-character password (e.g. "Café" with a decomposed e+U+0301) MUST normalize to its NFC form and yield the expected g/ISK, and an empty-after-normalization input MUST be rejected (R-S9) — the ASCII "Password" anchors never exercise normalization. (The fork's NFC-only normalization deliberately does not perform the OpaqueString non-ASCII-space map or FreeformClass rejection, R-P1; a non-ASCII-space password is therefore expected to differ from a full-OpaqueString peer — acceptable because both ends run the same build and any mismatch fails closed.) The negative AD test MUST confirm the abort is taken by the explicit ADa/ADb equality check (R-P5), not only by incidental transcript divergence. The suite MUST further include wire-choreography negative tests against the R-P14a state machine: a step delivered out of order (③ before ①), a duplicated step (two ① frames), a partial/dribbled frame that never completes (asserting the R-P14b timeout fires and the connection drops), a frame exceeding the ≤ 4 KiB pre-PAKE cap, and a replayed prior-connection transcript (asserting it aborts at the R-P3 tag, not via any sid cache). Each MUST assert fail-closed abort with no set_key, and that only the tag-mismatch case increments the R-S10 limiter — extending R-A8's owner-safe property to the pre-key path.

10. The PAKE specification

The handshake is small but unforgiving. Every step below is normative to the byte — each is a place real PAKE deployments have failed — and the construction follows the CPace draft and its published security proof, not intuition.

10.1 Primitive, suite & implementation

Primitive — balanced CPace

CPace, a balanced composable PAKE that authenticates from the shared password alone, with a published formal proof (Abdalla–Haase–Hesse 2021, adaptive UC). Suite: ristretto255 + SHA-512. This is a deliberate, documented deviation: the CFRG draft's recommended suites are CPACE-X25519-SHA512 (first) and the NIST-P curves; ristretto255 is a permitted "can also be implemented" option, chosen here because it is a prime-order group — eliminating by construction the cofactor / small-subgroup / twist pitfalls the draft enumerates for X25519.

Implementation reality — implement the draft; the crate is non-conformant

Settled empirically (and independently re-verified). A from-scratch draft-21 CPACE-RISTR255-SHA512 reference passes all of the CFRG draft's published ristretto255 test vectors (from github.com/cfrg/draft-irtf-cfrg-cpace): 18/18 protocol checks + 13/13 encoding checks, both abort cases included. Those published values are the derived CPace generator g = 222b6b19… (not the ristretto255 base point) and the symmetric-mode ISK ISK_SY = 544199d7… — the canonical ristretto255 vector, carried in testvectors.json under the group codename G_Coffee25519 and matching the draft body's worked example (the order-independent o_cat transcript reproduces it, R-P6). (testvectors.md lists the same vector — identical PRS/sid/scalars/AD — under a different example CI, giving a6fc82c3…/980dcc5a…; that is a channel-identifier difference, not a second vector, so do not conflate the two.) Computing jedisct1's pake-cpace on the same inputs diverges: the crate is on-the-wire non-conformant — it implements the 2019 draft-haase-cpace-01, unaudited, last released 2023. Its security-relevant deltas (no sid in the ISK, initiator-chosen sid, no key confirmation) live inside the hashed transcript and cannot be patched at the application layer — fixing them means forking lib.rs, i.e. rewriting draft-21 minus the vectors. There is no viable "use the crate" path.

Therefore the fork MUST implement draft-21 CPACE-RISTR255-SHA512 in-tree over curve25519-dalek — zero novel curve math (dalek supplies the audited, constant-time from_uniform_bytes map, scalar×point, decompress, is_identity; you write lv_cat/o_cat, the 64-byte generator string, the ISK transcript, and the R-P3 MACs). The construction core (encoding + generator/DH + ISK/MAC) is ~150–270 lines; counting everything §10 makes mandatory and security-relevant — the R-P14a handshake state machine, the two-key secretbox rewrite (R-P2/R-P10, which ripples through tcp.rs/websocket.rs/stream.rs), bounded reads (R-P14b), the dedicated Cpace protobuf, and the dual-role integration — the security-relevant diff is ~450–850 lines, with a comparably-sized KAT/negative-test harness on top. CI MUST gate on the draft's published test vectors — they catch the real failure modes (encoding off-by-ones, the zero-pad length prefix, 32-vs-64 hash length, transcript ordering, the aborts). Pin curve25519-dalek ≥ 4.1.3 (RUSTSEC-2024-0344). This adds no new external crate: every primitive the handshake needs is already in the 1.4.7 lock — curve25519-dalek 4.1.3, sha2 0.10.8, hkdf 0.12.4, hmac 0.12.1, subtle 2.6.1, zeroize 1.8.1, rand 0.8.5 (CSPRNG; getrandom 0.2.15 also present), and unicode-normalization (NFC, present via idna) for the PRS — which holds only because the PRS normalization is NFC-only, a deliberate subset of RFC 8265 OpaqueString (R-P1); a fully-conformant OpaqueString would need a new precis-profiles crate, which this fork does not add, and the inherited session layer is sodiumoxide 0.2.7 (libsodium: secretbox + the Ed25519 device key to be removed per R-P5 — there is no ed25519-dalek) — so the PAKE adds no new dependency, over crates already compiled in (exact lock versions in §3.2). The §11 review is vector-checkable for the construction core (encoding, generator derivation, ISK), but its scope is not bounded by that small line count: the R-P14a state machine, the two-key secretbox rewrite, the constant-time/identity guarantees, and the R-P3 confirmation-MAC composition (which has no published vector — its self-consistency KAT cannot catch a self-consistently-wrong composition) require review by someone fluent in protocol/nonce-misuse analysis. The independent expert audit R-V3 mandates is therefore necessary, not merely tractable-because-small.

No registry, no stored keys — the most sovereign anchor there is

Because a balanced PAKE authenticates from the shared password, this fork uses no long-term public keys, no signatures, no certificates, and no key registry of any kind. TLS anchors trust in CA registries; SSH anchors it in known_hosts (trust-on-first-use — a local registry). The PAKE anchors it in the password you already share — nothing to store, distribute, pin, or look up. One credential, one primitive, one path; no second PAKE to select, negotiate, or fall back to. (The per-connection ephemeral group elements on the wire are key-exchange values, not identities, and are kept nowhere.)

10.2 The handshake

R-P1MUST Inputs & channel binding
PRS = the NFC-normalized, no-case-fold UTF-8 of the permanent password (§10.4), used directly — the sole credential, since the one-time-password path is excised (R-X7); no memory-hard pre-hash (R-S6 explains why one would add nothing here). This is a deliberate NFC-only subset of the RFC 8265 OpaqueString profile: it applies NFC (the only normalizer in the lock — unicode-normalization via idna) but not OpaqueString's non-ASCII-space→U+0020 map or FreeformClass disallowed-codepoint rejection — chosen so the PAKE adds no new crate (§10.1), and safe because both peers run the same fork build (R-R2b), so any normalization divergence fails closed at the R-P3 tag (wrong PRS → divergent generator → divergent ISK → abort) rather than weakening; a balanced PAKE compares by exact PRS equality, so there is no collision surface. The password MUST be non-empty after normalization (R-S9) — CPace has no empty-PRS guard — and a non-ASCII KAT exercises this (R-A10). Because CPace is balanced, the controlled side must hold this PRS — a password-equivalent value — to run the handshake; protect it as R-S9 requires. The channel identifier CI MUST carry the protocol/version label and any per-connection channel binding, and MUST be identical and pre-agreed on both sides (CI is never sent on the wire). The session id sid MUST be a unique per-connection value agreed by both parties — e.g. a fresh 16-byte random nonce contributed by each side, concatenated initiator-then-responder (a 32-byte sid). That contribution exchange rides the still-unkeyed wire pre-PAKE, so its integrity rests solely on the R-P3 confirmation abort: sid MUST enter the ISK transcript (draft-21 symmetric — the ISK_SY vector), not merely the generator, so a tampered nonce yields a divergent ISK/mac_key and aborts. CI MUST be a fixed both-sides-identical string (fork tag · suite · version · port binding) — the per-side role instead lives in AD (ADa=b"viewer" / ADb=b"server"), since the draft requires both sides to hold the same CI; no per-connection value that would need a tamperable exchange. The port folded into CI MUST be the pinned compile-time direct-access port (21118, R-F4) — the same constant compiled into both roles, never a runtime option and never a value read from the live socket (NAT/port-forwarding would otherwise desync CI between peers and silently abort the handshake) — and its byte-encoding MUST be pinned (§10.4); since no published draft vector exercises a fork-specific CI, the build MUST add a project KAT over CI and the derived generator (R-A10). No identity keys (R-P5).
R-P2MUST Exchange & key derivation
Each side sends its CPace message (ephemeral group element); both derive the Intermediate Session Key ISK. The raw DH output K MUST NOT be exposed, logged, or used directly (a leaked K re-enables offline attack). From ISK, derive two directional secretbox keys (controlled→viewer and viewer→controlled) by HKDF-SHA-512 with distinct info labels; derive the confirmation key separately by the draft's own construction (mac_key = H("CPaceMac" ‖ sid ‖ ISK), R-P3). All three are domain-separated by construction — MUST NOT slice one hash for several keys, nor reuse one key across both directions (R-P10). Role→direction binding (load-bearing): both peers derive the same k_c2s and k_s2c, so correctness depends on each installing them in mirrored slots — the controlled side (responder) seals with k_s2c and opens with k_c2s; the viewer (initiator) the mirror. The inherited symmetric set_key(key) (one key both directions) MUST be replaced by a keying call that carries the role/direction, and the symmetric form MUST NOT survive (R-A6); a mis-binding that engages one key both ways re-creates the catastrophic reuse and is not caught by the derived-value k_c2s ≠ k_s2c KAT — only by R-A5's engaged-key assertion and R-A10's two-peer test.
R-P3MUST Explicit mutual key confirmation (the proven construction, not a transcript MAC)
After ISK: mac_key = H("CPaceMac" ‖ sid ‖ ISK); each party sends MAC(mac_key, lv_cat(its-own element, its-AD)); each MUST verify the peer's tag in constant time and abort before authorizing. The MAC MUST be HMAC-SHA-512 (the draft offers HMAC or CMAC; HMAC-SHA-512 matches the suite hash and carries its own RFC 4231 test vectors). This is the CPace draft's own Key confirmation construction, verbatim — specifically the variant it defines for the parallel setting without message ordering, which is exactly our symmetric mode (R-P6); it is not an ad-hoc transcript MAC. The reply MUST NOT be a constant — it is a function of ISK by construction (the fix for the prior fork's b"direct-ok"). Do not MAC "the whole transcript and both identities" — there are no identities, and that diverges from the construction the proof covers. Explicit confirmation also upgrades CPace's forward secrecy from weak to perfect. In the symmetric mode these tags are sender-bound only once R-P5's role/AD discipline is applied (otherwise a tag is reflectable in a loopback) — R-P3 MUST NOT be implemented without R-P5.
R-P4MUST One handshake at the choke point, before the type is parsed
Run once where the single stream is created (server.rs), keying all six transports and all five types before any application message.
R-P5MUST No PKI/registry; defend the loopback
No identity keys, signatures, certificates, host-key pinning, or registry; the existing SignedId/PublicKey/Ed25519 session-key bootstrap in the connection handshake is removed (§8) — but that is the device-identity keypair only: the rendezvous trust anchor (RS_PUB_KEY, R-X4) stays only as a baked constant. A server-assisted build would have needed the register_pk/key_confirmed registration-liveness gate (rendezvous_mediator.rs) preserved or the box goes unreachable — but §18 (R-SV4) compiles the mediator out of every shipped artifact, so there is no server-assisted build to preserve it for: register_pk is absent (R-SV10 greps it out), not preserved, and this carve-out is now dead for everything the fork ships (the direct-only deployment already sidestepped it). Consequence of having no identities: CPace will key with anyone presenting the same PRS+sid — including the host looping back to itself. The fork MUST add a loopback/relay defense: assign fixed initiator vs responder roles per (PRS,sid) and forbid one host playing both, or enforce unique ADa/ADb and reject a received AD that matches a locally-active session. Resolved (§10.4): pin fixed roles — the controlled side is always responder, the viewer always initiator — and carry the role as the per-side associated data ADa = b"viewer" (initiator) / ADb = b"server" (responder), with only the shared suite/version/port in CI (the draft requires both sides to hold the same CI, so the role cannot live there). Because AD is per-side, sent, and folded into both the transcript and the R-P3 tag, each tag is sender-bound and a host cannot run both ends for one (PRS,sid); a loopback/reflection fails by role/AD mismatch. The AD check MUST be explicit: the responder MUST verify the received ADa == b"viewer" and the initiator MUST verify ADb == b"server", aborting on mismatch — folding the received AD into the transcript is necessary but not sufficient; the literal equality check is what makes each confirmation tag sender-bound.

10.3 Construction details (each is a place real PAKEs have failed)

R-P6MUST Symmetric-mode ordering
Both ends are peers, so message order is undefined ⇒ the symmetric setting: combine with the draft's lexicographic o_cat(lv_cat(Ya,ADa), lv_cat(Yb,ADb)), OR impose fixed initiator/responder roles (which also satisfies R-P5). Never a fixed Ya‖Yb order without roles. Note: the fork pins fixed roles and uses the symmetric o_cat transcript — a deliberate superset of the draft's initiator-responder transcript_ir (safe: o_cat binds both elements + ADs). The ISK_SY vector is therefore the gating KAT; an implementer MUST NOT substitute transcript_ir (it fails the vector and breaks interoperability with a conformant fork peer).
R-P7MUST Abort on invalid / identity point
If the shared point is the group identity, or a received element fails ristretto255 decode(), abort — never continue with a degenerate key. The trap: the all-zeros encoding decodes successfully to the identity, so decode-success alone is insufficient — the K.is_identity() test after the multiply is what catches a malicious identity input (§10.4).
R-P8MUST sid uniqueness; no secret reuse
sid unique per connection (e.g. random_a ‖ random_b); ephemeral secret scalars and sid MUST NOT be reused.
R-P9MUST Pinned domain separation
Pin the exact DSI strings for the chosen draft version and a distinct HKDF label per key; a DSI mismatch MUST make keys not agree, never silently cross-confuse.
R-P10MUST secretbox nonce discipline
The 24-byte XSalsa20-Poly1305 nonce MUST be unique per key and never reused — including across directions. The inherited layer derives the nonce from a per-direction sequence counter alone (tcp.rs get_nonce = LE64(seq), no direction tag) and installs one key with both counters at 0 (set_key Encrypt(key,0,0)) — so a single shared key makes each side's frame n seal a different plaintext under the same (key, nonce): catastrophic XSalsa20-Poly1305 reuse. Per-direction keys (R-P2) close it — the invariant is key-uniqueness, not nonce-uniqueness, so identical per-direction counters from 0 are safe and one MUST NOT "optimize" back to a single key; also specify a derived-counter generation and an abort/rekey policy before counter wraparound. A rekey MUST install fresh per-direction keys — never reset the counters under the existing keys, which would re-create the very (key, nonce) reuse this requirement closes. (The u64 counters make wraparound unreachable in practice — millennia at any real frame rate — so the rekey is defensive belt-and-suspenders against a reset bug, not the load-bearing closure; the closure is key-distinctness, which is why R-A5 asserts the engaged send/recv keys differ, not merely that the derived values do. The pre-increment makes the first nonce LE64(1), not 0 — immaterial to the argument.) Ripple — not a one-file change: the two-key fix reshapes the public Encrypt tuple (tcp.rs) and changes the set_key signature to carry the role/direction (R-P2) across Stream::set_key's three variants (stream.rs) and the WsFramedStream that reuses tcp::Encrypt (websocket.rs). The peer↔peer relay session is keyed at the same choke point (server.rs) so R-P14 covers it; the rendezvous-signaling secure_tcp path (common.rs) is removed by R-D4 on the direct-only build (touched by the set_key signature change only in a server-assisted viewer build).
R-P11MUST No downgrade, no suite negotiation
Hard-pin one suite, one draft version, one DSI set; carry the version label in CI so any mismatch yields non-agreeing keys. No negotiation handshake exists to downgrade. The pinned draft is draft-irtf-cfrg-cpace-21 — the current CFRG revision as of 2026-06 (latest revision 2026-04-22, in IESG review; not yet an RFC), and its published ristretto255 vectors are exactly the ones §10.4 reproduces — which MUST be taken from the draft-irtf-cfrg-cpace-21 tag (or vendored), not the live CFRG master, whose later example-CI change produces different vector values (R-A10). If CPace later reaches RFC, adopt it only as a deliberate, re-vectored change — never an automatic version bump.
R-P12MUST K-secrecy & constant-time
K MUST NOT be exposed or logged (only ISK leaves the PAKE layer). Generator computation, scalar sampling, scalar multiplication, and every tag comparison MUST be constant-time (curve25519-dalek provides this for the curve ops — assert it). Scalar sampling MUST use Scalar::from_bytes_mod_order_wide / Scalar::random, never the deprecated Edwards-only Scalar::from_bits (§10.4); and the tag comparison is the one CT op not inside dalek — use a constant-time comparator (subtle / Mac::verify), never ==. Constant-timeness MUST be machine-checked — a dudect-style statistical timing test in CI over the sampling and the tag compare, plus the type-level guarantees of subtle/curve25519-dalek — so it is verified, not reviewed by eye — removing the timing axis from the review burden (but not the state-machine, nonce-misuse, or MAC-composition axes, which still need expert review — §11/R-V3).
R-P13MUST Quantum-annoying, not PQ — record it
CPace is "quantum-annoying" (a quantum attacker solves one CDH per password guess), not post-quantum. Role/port already fold into CI (R-P5/§10.4), raising per-guess work; the fork MUST document that long-term confidentiality rests on password entropy — a recorded transcript is offline-recoverable by a future quantum attacker against a weak password. A PQ-hybrid is out of scope for a single-user box unless that threat is.

Estimated change size: the CPace construction core is ~150–270 lines, but the full security-relevant diff — the handshake state machine, the two-key secretbox rewrite, the §10.3 invariants, the §8 excisions, and removing the SignedId/Ed25519 exchange — is ~450–850 lines (§10.1), with a comparably-sized test harness. The code diff is small; the work around it is not — every line in §10.3 is a place a real PAKE deployment has failed, and the verification (§11), independent audit (R-V3), and per-platform build pipeline (§12) dwarf the diff. That asymmetry is the point, not a reassurance.

R-P14MUST On-wire choreography: where the handshake runs and the messages it replaces
The CPace exchange runs inside create_tcp_connection (server.rs) on the unkeyed Stream, replacing the current SignedId ↔ PublicKey key bootstrap verbatim (controlled sends SignedId at server.rs; peer replies PublicKey{symmetric_value,asymmetric_value}; set_key(Encrypt::decode(…))). That block (server.rs), its client counterpart, the box_ seal/open path (tcp.rs), and the SignedId/PublicKey protobuf messages are removed (R-P5, §8). The new flow is pinned:
  • Message — a dedicated top-level type, not a Message-union variant. Add Cpace as its own top-level protobuf message in libs/hbb_common/protos/message.proto (a oneof over the steps below), framed through the existing bytes_codec but parsed and serialized on its own on the unkeyed stream — parse_from_bytes::<Cpace>, never Message::parse_from_bytes. Routing it through the application Message union (the obvious shortcut) would make the pre-auth decoder reach the entire Message schema — every VideoFrame/file/clipboard field an attacker could populate — whereas the dedicated type keeps the reachable pre-key decoder surface to the handshake's ~4 fixed fields (R-S7). Both roles change: the responder (the controlled-only build, and the viewer build when acting as responder) parses only Cpace pre-key, and the initiator (viewer build only, R-R2b) sends/parses only Cpace pre-key — which also protects a connecting viewer from a malicious server's malformed pre-key frames. Both sides switch to the normal Message framing only after keying. No other message type is accepted, or even decoded, on the unkeyed stream.
  • Fixed roles & sequence (viewer = initiator, controlled = responder). The generator g needs the full sid = sid_a‖sid_b and CI, so the sid contributions precede the group elements: ① initiator → responder { sid_a(16B), ADa=b"viewer" }; ② responder → initiator { sid_b(16B), ADb=b"server", Yb(32B) }; ③ initiator → responder { Ya(32B), Ta }; ④ responder → initiator { Tb }. Ta/Tb are the R-P3 HMAC-SHA-512 tags (64 B). Two round-trips.
  • Gate. Each side verifies the peer's tag in constant time, derives the two directional keys (R-P2), and calls set_key only on success; no set_key, no Connection::start, no application message, no authorized before both tags verify (R-A1/R-A2). Any decode/length/timeout/tag failure aborts fail-closed.
  • Framing. set_max_packet_length is capped small before the first byte (R-S7); each CPace message is ≤ ~120 B. With KCP dropped (R-S7/R-D4), this dedicated, fixed-shape Cpace message (parsed on its own, never via the Message union — R-P14) is the only attacker-reachable parser before keying.
This choreography is in scope for the §11 adversarial review (R-V1).
R-P14aMUST Strict step-ordering and no-duplicate handshake state machine
§10.2 gives the message sequence; this pins the state machine that enforces it. The handshake MUST be driven by an explicit four-state machine per role (responder: WAIT_1 → INIT2 → WAIT_3 → SEND_4 → KEYED; initiator: INIT → WAIT_2 → SEND_3 → WAIT_4 → KEYED). In each receiving state the implementation MUST accept exactly one Cpace oneof variant and MUST abort fail-closed on any other — a future-step variant, an own-side variant, or a repeat of an already-consumed step (no peer may send two step-① frames). A step arriving before its prerequisites (e.g. step-③ before step-① at the responder, which then holds no sid_b/g/yb) MUST be a wrong-variant abort, never buffered; no state may silently ignore an unexpected frame and keep waiting (that re-opens the half-open / duplicate surface). Each step is consumed exactly once, and set_key is reachable only from the tag-verify success edge (R-P14, R-A1).
R-P14bMUST Bounded reads — the timeout is the half-open resource bound
set_max_packet_length(≤ 4 KiB) MUST be set before the first read (R-S7), and every inbound handshake step MUST be read under a finite timeout (per-step, or one whole-handshake deadline). Verified: the framing codec returns Ok(None) on a partial/dribbled frame (bytes_codec.rs) — it never completes and never errors — so the timeout is the sole bound on a peer that opens a connection and stalls (the inherited code wrapped only its single next(): server.rs / client.rs, CONNECT_TIMEOUT/READ_TIMEOUT = 18 s; four steps need four bounded waits, or one deadline). A timeout, a peer EOF (Stream::next() → None, tcp.rs), an oversize frame (bytes_codec.rs "Too big packet"), or a parse_from_bytes::<Cpace> decode error MUST abort fail-closed. The timeout (stall bound), the ≤ 4 KiB cap (size bound, R-S7), and the per-source limiter (CPU-flood shedding, R-S10) together bound half-open exhaustion to ≤ 4 KiB + one task per connection until the deadline; no per-connection state beyond the in-flight handshake is created.
R-P14cMUST Only key-confirmation failure feeds the limiter; replay needs no state; nothing on the wire to downgrade
Of all pre-key aborts, only an R-P3 key-confirmation tag mismatch MUST increment the per-source limiter (R-S10) — it is the sole event that is an online password guess; decode / length / out-of-order / duplicate / decompress / identity / AD-mismatch / timeout / EOF aborts MUST NOT increment it, or a malformed-frame flood would trip the owner's own block (R-S10's cardinal principle; the limiter is checked before the scalar-mult). Whole-handshake replay is harmless and MUST NOT be defended by sid-tracking state: each side contributes a fresh 16-byte sid nonce and a fresh ephemeral scalar per connection (R-P8), so a replayed sid_a/Ya cannot yield a tag valid under the responder's fresh sid_b/Yb (the mac_key/ISK diverge) and aborts at R-P3. sid freshness is enforced unilaterally and locally by CSPRNG sampling; a "seen-sid" cache is forbidden — unnecessary, and itself a memory-DoS / stateful-config surface (R-S12). The Cpace oneof carries no version / suite / capability field — those live only in the never-sent CI (R-P1, R-P11) — so there is no on-wire field to downgrade, and an unknown/unset oneof case is a decode abort, not a fallback.

Honest caveat — stated every time

The in-tree CPace implementation (§10.1) is not yet independently security-audited. The fork trades upstream's no cryptography for a principled, standardized, test-vector-conformant one — strictly better — but the implementation MUST clear the review below before it guards an internet-exposed root-capable service. The construction core (encoding, generator derivation, ISK) is vector-checkable; but the review scope is not bounded by that small line count — the R-P14a state machine, the two-key secretbox rewrite, the constant-time/identity paths, and the R-P3 confirmation-MAC composition (which has no published vector, so its self-consistency KAT cannot catch a self-consistently-wrong composition) need a reviewer fluent in protocol/nonce-misuse analysis. 12. Build, release & version pinning

R-B1MUST Build & publish the full client matrix
From the one forked codebase, the build MUST produce Android aarch64 (.apk), Windows x86_64 (.exe + .msi), and Debian x86_64 (.deb) — the full viewer build, each usable as viewer and, via --server, controlled side — plus a headless controlled-only Linux .deb and Windows service with the viewer feature off (R-R2b) for the internet-exposed box (§17). It MUST run to completion off GitHub-hosted runners (locally or a self-hosted runner; R-B2); macOS/iOS are dropped (R-R2).
R-B2MUST No code-signing; the build cannot live on GitHub-hosted CI
Upstream's release model signs artifacts with secrets injected into GitHub Actions — unusable here on two counts. (1) GitHub-hosted runners are too unreliable to be the build path: the heavy RustDesk matrix (vcpkg built from source, multi-GB toolchains, long Windows/Android jobs) fails on the order of one run in five, so you cannot depend on getting a green build out of it — flutter-build.yml MUST be runnable to completion off GitHub-hosted runners (locally or a self-hosted runner; R-B1). (2) Artifacts ship unsigned: code-signing is a CA/registry dependency (the model R-P5 already rejects), it adds nothing once the binary's hash is verified directly, and with the build off GitHub there are no CI secrets to hold a signing key anyway. Exception — Android: the OS refuses to install an APK with no signature, so it uses a self-generated local key (apksigner/keytool — no CA, no Play Store), a forced local keypair, not CA code-signing — Android refuses to install an unsigned APK (the v2 scheme is mandatory since Android 11), so this signature is not optional. Pin it: one stable RSA-4096 keystore (SHA256withRSA; validity ≥ 10000 days; fixed alias), generated once and reused for every release — Android ties app identity to the signing key, so a stable key gives clean in-place upgrades whereas a changed/absent key forces uninstall+reinstall (wiping app data). The keystore is a secret: kept out of the repo and the build image, fed to build-android.sh only at sign time (R-B11) — mounted read-only, never baked into an image layer, with its password supplied via file/agent, not env or argv (both leak via /proc and container inspect). Because one stable key signs every release for clean in-place upgrades, its theft lets an attacker ship malicious upgrades and rotation forces a data-wiping reinstall, so the fork MUST document a key-compromise / break-glass procedure and hold the keystore offline (ideally HSM-backed) — the residual is real and currently unaddressed. The signature is Android's install gate, not the trust anchor — the pinned SHA-256 still is. Integrity from build to install is a pinned SHA-256 per artifact, verified on the target over the operator's existing trusted out-of-band channel — the same pinned-SHA check used for any third-party .deb — over the reproducible build from the committed lockfiles (R-R1). Tamper-evidence without a signature. One determinism prerequisite the recorded-SHA bar depends on: upstream bakes a wall-clock, local-timezone build timestamp into every binary (hbb_common::gen_version emits BUILD_DATE via chrono::Local::now(), hbb_common/src/lib.rs, also writing into the source tree at ./src/version.rs), so two builds of identical source on the identical pinned image otherwise yield different bytes and a different SHA-256 — turning the operator's rebuild-and-compare into a guaranteed mismatch and making a malicious build host indistinguishable from a benign rebuild. The fork MUST neutralize it — honor SOURCE_DATE_EPOCH (patch gen_version to read it, or hard-pin BUILD_DATE to the release commit's date) — and CI MUST assert it by a double build: build the same source twice and require byte-identical SHA-256. Without that assertion the recorded-SHA bar is unfalsifiable.
R-B3MUST Publish AGPLv3 source per binary
Each release links the exact source revision (both repos + submodule commit) it was built from.
R-B4MUST Smoke-test every platform; assume nothing builds until watched
Per platform, connect + authenticate + drive each session type must pass before tagging. (Upstream's fork pipeline emits nothing until upload-artifact is wired and the build is watched to green — real work, the more so on the flaky runners R-B2 keeps the build off of.) Plus a socket-surface smoke-test on the running --service: ss -lntup + strace -f -e trace=bind,listen,connect MUST show exactly one v4 TCP listener on the direct port, zero UDP listeners, and no unsolicited outbound connect — the build-time empirical confirmation that the binary binds what Appendix B claims (R-D3/R-D5/R-D6), with R-A4 the permanent runtime backstop.

R-B9MUST One build script per target — environment-validated, fail-loud, no fallbacks, one mode
The build is driven by checked-in scripts, one per target (build-debian.sh, build-android.sh, build-windows.ps1), held to the same engineering bar as the rest of this spec — the "one option, the good option" discipline applied to the build:
  • Validate the environment, then abort. Assert the exact pinned versions — not mere presence — of Rust / Flutter / NDK / the vcpkg baseline / LLVM / JDK, plus the OS + arch, the required system libraries, and the submodule + lockfile state, before compiling a byte.
  • Fail loud; no fallbacks. set -euo pipefail (PowerShell $ErrorActionPreference = 'Stop'); every check dies with a precise message; no "install latest if missing", "retry another way", or silent default — a single mode of operation, the correct one.
  • Pin from one manifest. Every version comes from a single committed pins file; no command resolves "latest" or a moving channel (R-B5a, R-R1).
  • Self-verifying + idempotent. Re-running is safe; on success the script verifies the artifact (it exists, is the right type, and matches its recorded SHA-256, R-B2) and prints that — it never reports success it has not proven (the full per-script behavioral contract is §12.3).
R-B10MUST One networked step: online-fetch.sh → a git-ignored, SHA-256-verified ./online
The repository is build-oriented and offline-by-construction. A single checked-in scripts/online-fetch.sh is the only script permitted to touch the network; it materializes every resource the repo does not embed — cargo vendor of the committed lockfile (incl. the 44 git deps), the pub/Gradle caches, vcpkg @120deac3 and its asset cache, the NDK r28c zip, the Flutter SDK, LLVM 15.0.6, the Android SDK (cmdline-tools / platform-34 / build-tools-34.0.0), the pinned base-image digests, and the Windows installers/ISOs — into ./online/, which is git-ignored. Every artifact is checked against its pinned SHA-256 in the single scripts/pins.env manifest (the machine-readable §3.2); any mismatch MUST abort fail-closed. This reconciles R-R1's "pinning ≠ vendoring" with the offline build: the bulky pinned world is cached, not committed — re-creatable from pins.env and re-verifiable, never trusted from the network at build time. The build scripts (R-B8/B9, §12.1–§12.2) then run with the network namespace removed (--network=none) and MUST refuse to run — never silently fetch — if ./online is incomplete or any SHA fails. The isolation MUST be of the compile container itself, not merely "no published ports": its own network namespace removed, no Docker socket bind-mounted in, and the warm caches mounted read-only — with a canary build.rs in CI that attempts an outbound connect and MUST fail the compile stage, so "no fetch at compile time" is proven, not trusted. Run order: host-provision.sh (once) → online-fetch.sh (once, or on a pins.env change) → build-* (per build, offline) → cleanup.sh.
R-B11MUST Additive host provisioning; reversible-only cleanup that never touches what pre-existed
scripts/host-provision.sh installs only the host-level runtimes the ephemeral envs need — the container engine and, for the Windows VM, qemu-kvm + libvirt + swtpm + OVMF — plus the few tools the scripts call. The pinned toolchains never land on the host; they live in the ephemeral image/VM (R-B8). Provisioning is additive and idempotent: for each package it checks whether the host already has it and installs only what is absent, recording exactly what it added to ./online/.provisioned. scripts/cleanup.sh is reversible-only: by default it removes just the artifacts the harness created — the ephemeral containers, qcow2 overlays and transient libvirt domains, and the built images — which is always safe because we made them. Reversing the host install is a separate, explicit invocation that removes only the packages recorded in ./online/.provisioned as installed-by-us; it MUST NOT remove or downgrade any package that pre-existed on the build machine, and MUST refuse (fail-closed) if that manifest is absent rather than guess. No --force, no heuristics — the cleanup surface is exactly the inverse of what the scripts created. (The .provisioned manifest is build state, not cache: it MUST live outside the disposable ./online so a cache wipe cannot strand the reversal.)
R-B12MUST Pin provenance, not just a hash — close the bootstrap TOFU and the codegen/distfile gaps
R-B10 verifies every ./online artifact against a SHA-256 in pins.env, but a hash is only as trustworthy as its origin. The fork MUST establish each first pin by an audited, dual-sourced bootstrap — cross-checking the publisher's own published hash/signature (Rustup, Flutter, LLVM, the NDK, and the Windows toolchain all publish them) against a second path — and record per-pin provenance, so a compromised mirror/CDN at first-fetch cannot silently poison the anchor (the distinction is pinned vs pinned-and-provenance-verified). 12.1 The containerized build flow — Debian x86_64 and Android aarch64, in turn

This is the concrete realization of R-B7–R-B9 for the two artifacts that build on the Linux x86_64 host in Docker — the Debian x86_64 .deb and the Android aarch64 .apk, built in turn; the Windows .exe/.msi follows the same discipline in a KVM virtual machine on the same box (§12.2). Both Docker builds run inside a digest-pinned image — the image is the versioned build environment R-B8 demands ("provisioned to the pinned toolchain and nothing more"). This does not contradict §17's "no Docker": that rule governs the controlled host's runtime, never the build. Every toolchain version, vcpkg triplet, feature flag and build command below is taken verbatim from upstream's 1.4.7 build (R-B7, R-B5a) — the fork makes no independent version choices; its only deltas are to pin what upstream uses, run off GitHub on one box, drop code-signing (R-B2), build offline (R-B5b), and bind loopback-only (R-D3).

Build environments are ephemeral instances of an immutable, pinned template

Every target follows the same disposable-instance discipline ("cattle, not pets"). The persistent, version-controlled artifact is the template — the Docker image pinned by digest (Debian/Android) or the golden Windows VM image (§12.2). Each build spins up a fresh, throwaway instance of that template (a docker run --rm container; a transient libvirt domain over a copy-on-write qcow2 overlay), runs the validated flow, copies out the artifact and its SHA-256, and is then stopped and deleted — no instance survives the task, and no state carries between builds. That is what makes R-B9's "one mode, no drift" structural rather than aspirational: every build starts from the byte-identical pinned baseline, so the recorded SHA-256 (R-B2) is reproducible, and nothing mutable accumulates on the host.

Why Docker is the faithful path, not a deviation (R-B7)

Upstream already builds the .deb inside a container: flutter-build.yml's build-rustdesk-linux runs every step through rustdesk-org/run-on-arch-action in an ubuntu18.04 container — a deliberately old glibc baseline that keeps the .deb portable. The Android job cross-compiles natively on ubuntu-24.04. So a pinned ubuntu:18.04 image for the .deb and a pinned ubuntu:24.04 image for the .apk reproduce upstream's own environments off GitHub-hosted runners — the "exactly two deltas (no signing, no GitHub runner) and no others" bar of R-B7.

The shared two-stage shape. Each script MUST split a network-enabled fetch stage from a network-isolated compile stage, so "no build.rs / Gradle / vcpkg fetch at compile time" (R-B5b) is proven structurally, not trusted:

  1. Validate (R-B9) — assert the exact pinned versions, the base-image digest, and the submodule + lockfile state; abort on any mismatch before a byte compiles.
  2. Fetch (network on) — cargo vendor the lockfile (incl. the 34 rustdesk-org git deps, R-B5b), build/restore the vcpkg deps from the pinned baseline, warm the Flutter/pub/Gradle caches.
  3. Compile (--network=none) — build --locked --offline; with the network namespace gone, any forgotten fetch fails loudly here instead of silently succeeding.
  4. Package + self-verify (R-B2, R-B9) — produce the artifact, record its SHA-256, re-check it; never claim success unproven.

No new exposure — loopback only. No build stage opens an externally-bound (0.0.0.0) listener: dockerd is reached over its unix socket with no published ports, the network-on fetch stage is outbound only, and the Windows VM (§12.2) is driven over a loopback-bound SSH forward (127.0.0.1). The build adds nothing to the host's audited ports (R-D3) and is safe to run even where only loopback binding is permitted.

The integrity bar for this flow

The target is same committed lockfiles + same pinned image → the same recorded SHA-256, re-verified on the controlled host over SSH (R-B2) — not byte-identical-for-any-third-party reproducibility, a harder bar this spec does not claim. Docker supplies the environmental determinism that keeps the SHA-256 stable across rebuilds on this one host; the committed lockfiles plus --locked --offline under --network=none supply the rest.

12.2 Windows x86_64 — the ephemeral KVM Windows 11 VM

Windows is the one target that cannot be cross-built from Linux: flutter build windows runs only on a Windows host (the desktop embedder needs Visual Studio's MSVC "Desktop development with C++"), the vcpkg x64-windows-static codecs need cl.exe, and the installer needs WiX — all Windows-only. So the .exe/.msi build in a Windows 11 guest, realized here as an ephemeral KVM virtual machine on the same bare-metal Linux x86_64 box (any x86_64 host with hardware-virtualization / KVM support) — so all three targets live on one machine with no second physical host and no GitHub-hosted runner. (Where the build shares the §17 host, all build tooling binds loopback only — no 0.0.0.0 listener beyond R-D3's audited ports — and runs only during a build window, never during exposed operation.)

The template — built once. Provision a Windows 11 guest with the pinned toolchain and nothing more (R-B5a): VS 2022 Build Tools + "Desktop development with C++", Windows 11 SDK, LLVM 15.0.6, Flutter 3.24.5, Rust 1.75 (x86_64-pc-windows-msvc), vcpkg @120deac3 (x64-windows-static), WiX, and OpenSSH Server for headless control — then freeze it read-only as the golden backing image. Windows 11's requirements are routine on KVM today: TPM 2.0 via swtpm (in the default Ubuntu repos), Secure Boot via OVMF, virtio storage/net. This golden image is the Windows analog of a digest-pinned Docker image — version-stamped, and the reproducibility anchor. Three integrity steps are mandatory: the golden image MUST be verified by the SHA-256 of its qcow2 before each build (a version-stamp string is forgeable; a content hash is not); the VM's SSH host key MUST be pinned in known_hosts (no first-connect TOFU, even on loopback — the spec rejects TOFU everywhere else); and the Windows toolchain installers that seed the image are partly evergreen, so the fork MUST apply R-B12's captured-layout pinning rather than assume a stable upstream URL. swtpm/OVMF here only satisfy Win11's installer — they are not part of the integrity story (no guest attestation is claimed).

# TEMPLATE (built once): golden Win11 image, pinned toolchain, frozen read-only
golden=win11-rustdesk-golden.qcow2           # the pinned "image" == read-only backing file

# PER BUILD: a disposable instance — the exact analog of `docker run --rm`
qemu-img create -f qcow2 -b $golden -F qcow2 build.qcow2      # copy-on-write overlay, throwaway
virsh create build-transient.xml                             # TRANSIENT domain; QEMU user-mode (SLIRP) net — only listener is a loopback SSH forward (hostfwd tcp:127.0.0.1:2222-:22); build-vm = that alias
ssh build-vm "powershell -File C:\rustdesk_fork\build-windows.ps1"   # validate -> vcpkg -> cargo -> flutter -> WiX
scp build-vm:C:/out/rustdesk-${VERSION}-x86_64.exe ./        # pull .exe + .msi + SHA256SUMS
scp build-vm:C:/out/rustdesk-${VERSION}-x86_64.msi ./
scp build-vm:C:/out/SHA256SUMS ./
sha256sum -c SHA256SUMS                                       # verify on the way out (R-B2)

# TEARDOWN: stop + delete — no instance survives; the golden image is untouched
virsh destroy build-transient
rm -f build.qcow2

build-windows.ps1 mirrors upstream's build-for-windows-flutter — $ErrorActionPreference = 'Stop', assert the pinned versions, then vcpkg install --triplet x64-windows-static → build.py --portable --hwcodec --flutter --vram (cargo features hwcodec,vram,flutter; runs flutter build windows --release) → WiX .msi → record the SHA-256 (R-B9). No cross-compile, no code-signing — integrity is the recorded SHA-256, verified on the target over the trusted channel (R-B2).

12.3 The script contract — idempotent, defensive, fail-closed

Every script of §12 (R-B9/B10/B11) obeys one behavioral contract, so the harness is predictable regardless of the host's prior state. Each is a convergent, idempotent step: run it when the system already matches pins.env and it is a no-op (it verifies, says "already satisfied," exits 0); run it on a fresh host and it establishes only what is absent. One rule makes that safe.

Defensive means: complain loudly and die — not "deal with it"

A script may create what is missing; it MUST NOT mutate what is present-but-wrong to force a match. A wrong-version tool, a SHA that doesn't match the pin, a conflicting artifact it did not create — each is an anomaly, and the correct response is a precise, non-zero, loud abort naming expected vs found — never a silent upgrade, downgrade, overwrite, retry, or skip. Reconciling a genuine change (a bumped pin) is an explicit operator action — cleanup.sh the stale state, then re-run — not something a script does behind your back. Every component distrusts both the environment and the outputs of its sibling scripts and re-verifies them before acting (set -euo pipefail / $ErrorActionPreference = 'Stop'; every value from pins.env; success proven, never assumed). "Establish-the-absent" is the only mutation any script performs.

14. Non-goals

  • The rendezvous/relay servers (hbbs/hbbr) — serverless by design. (Self-hosting that pair is the upstream-sanctioned road to sovereign-and-encrypted, but it adds internet-facing ports and key-managed services and surrenders the "connect by address, nothing pre-configured" property the PAKE fork keeps.)

  • Defending a compromised endpoint or a password-knowing adversary.
  • Host network segmentation — the fork must not worsen pivot risk but does not implement the host firewall/VLAN.

Sovereign by construction (§18): every shipped artifact — viewer and controlled, on all three platforms — makes zero connections to any global resource and carries no auto-update, version-check, or self-install path; CI proves the §18 token set compiled-absent (sound grep) or call-graph-unreachable (the dual-use STUN/NAT probes) per artifact (R-SV10), the runtime self-check covers the viewer build too (zero UDP, one TCP listener), and the build smoke-test (R-B4) confirms no unsolicited outbound connect. The only surviving network capability is the direct, PAKE-gated peer connection by IP — connect-by-ID is removed on every platform, so R-F2/R-F3 hold for peers that present a reachable address; the relay-only NAT-traversal case is a deliberate, accepted drop (R-SV5, R-F3).
  11. Apple source-conformance (R-R2): iOS and macOS are not built, but a cargo check --target {x86_64,aarch64}-apple-darwin + aarch64-apple-ios pass proves the philosophies hold on the Apple code paths — the shared wire/auth/config/fs core inherited with no Apple #[cfg], and the Apple platform surfaces (the osascript root updater/install and DMG fetch-and-run, the two in-process root-exec primitives, the unauthenticated _url IPC, the deep-link incl. the iOS transport and the Rust ?key= adoption, the key/get_key() override, the sandbox-off/persistent-TCC entitlements, and the §18 egress) excised or hardened in source — so any future Apple build on a real Mac is the hardened fork, never the un-hardened upstream.
The one-sentence bar

Connecting by address is encrypted and mutually password-authenticated across every transport, type, and content channel, with no plaintext escape and no fetch-and-run or override path left in the building; the program refuses to run if any of that is untrue; and a build exists for every target device — making it, on the wire, as defensible as SSH.

17. Deployment: serverless direct-IP on Linux (no central server, no Docker)

The controlled Ubuntu box runs the fork's own binary — the headless controlled-only build (R-R2b), viewer/initiator role compiled out — as a native systemd service and listens on one TCP port for inbound direct-IP connections, each gated by the PAKE (§10). There is no rendezvous account, no hbbs/hbbr, no relay, and no Docker — the "connect by address, nothing pre-configured" property (R-F3) with the sovereignty of a single owned port.

R-D1MUST Native systemd service; no container, no rendezvous dependency
Ship the verified unit (res/rustdesk.service: ExecStart=/usr/bin/rustdesk --service, Type=simple); the .deb postinst already systemctl enables and starts it. The controlled side MUST NOT require any container runtime or reachable external server to accept a direct connection.
R-D2MUST The only runtime policy input is the permanent password — all else compile-time pinned
The hardened box has exactly one runtime security-policy input — the permanent password (the PAKE PRS, R-P1). (The source whitelist is the one other operator-settable value, but it is fail-closed — it can only narrow the accepted source set, never widen it, R-S16(e) — a deny-only filter, not the permissive policy knob this requirement eliminates; the PAKE is the gate regardless.) Everything upstream exposes as a runtime --option is here a compile-time pin, not a knob: the direct listener is unconditional (it is the mode — start_direct_only, R-D4), the port is fixed (21118, R-F4), and the whole controlled-side security policy — verification-method = use-permanent-password, approve-mode = password, the capability set, egress-empty — is a compile-time table the Config::get_option funnel returns and the is_option_can_save guard rejects every write to (R-S16), so no resolver reads a permissive default and no local write can change it (R-X7, R-S9, AST-proven by R-A6). None of these is an effective runtime knob, so none can be flipped by a local process or an IPC write (R-S11). The single headless step uses --password (not --config, the excised key/server path, R-X4): 
rustdesk --password <strong-permanent-password>   # the sole runtime input = the PAKE PRS (R-P1)
A client then reaches <static-IPv4>:21118 with that password → the mutual PAKE → a keyed session. No second credential exists (R-X7); per-IP rate-limiting is in its clean, real-address case (R-S10).
R-D3MUST Account for every open port — one inbound TCP port, zero UDP, understood not merely allowed
The hardened box's entire inbound surface MUST be exactly one TCP port — the direct-server port (21118, R-D2) — and zero UDP; there is no LAN-discovery (21119, R-X5), no relay range, no ephemeral UDP (R-D4), and no second listener of any kind. Every socket the process binds MUST be enumerable and individually justified (asserted at startup, R-A4). "Secure even on a DMZ" is not a firewall claim: it holds because the binary itself cannot open anything else — R-D4 removes the rendezvous/LAN/KCP sockets, R-D5 binds the listener v4-only, R-D6 silences egress — so an operator's firewall merely agrees with a surface that is already minimal, rather than being the thing that makes it safe. The service runs as root (input injection, virtual display, and login-screen capture require it), so R-S11 least-privilege is bounded by that reality — which is precisely why the failure-mode-C excisions (§8: updater, plugin loader, config rewrite, deep-link writes, and the root-PTY terminal R-X8) are load-bearing: a privileged, internet-exposed service must carry no fetch-and-run, re-home, or shell-spawning path. The host firewall/VLAN and any out-of-band management channel are the environment (a non-goal, §14); this requirement governs the fork's own surface, which must be self-evidently minimal whether or not a firewall is present. The complete socket-by-socket enumeration — every bind site, its original purpose, and the exact mechanism that neutralizes it — is Appendix B.
R-D3aSHOULD Confine the root service with the systemd sandbox the unit currently lacks
R-D3 accepts that the service runs as root (input injection, virtual display, login-screen capture need it) and that least-privilege is "bounded by that reality" — but the shipped unit (res/rustdesk.service) today adds zero kernel-level confinement, so a memory-corruption bug in the root service has the whole machine. The §8 excisions remove the reachable escalation paths; this shrinks the blast radius of anything missed, without touching the root requirement. NoNewPrivileges cannot be set (it would break the sudo -u drop to the --server child), but the unit SHOULD add: a CapabilityBoundingSet dropping caps the service never uses (CAP_SYS_MODULE, CAP_SYS_BOOT, CAP_NET_ADMIN, CAP_AUDIT_*, …); RestrictAddressFamilies=AF_UNIX AF_INET — dropping AF_INET6 (the kernel-policy twin of R-D5's v4-only bind) and AF_NETLINK; ProtectKernelModules, ProtectKernelTunables, RestrictRealtime, LockPersonality, SystemCallArchitectures=native, and SystemCallFilter=@system-service minus @mount/@reboot/@swap; and MemoryDenyWriteExecute if validated against the JIT-free codec path. Each is compatible with the root + input/display/capture reality and is the realistic least-privilege available beyond the excisions.
R-D4MUST The direct-only build excises the rendezvous mediator — no phoning home, no pre-PAKE signaling, no ephemeral UDP
Verified gap: the R-D2 config does not by itself make the box serverless. With direct-server=Y and no stop-service, RendezvousMediator::start_all (rendezvous_mediator.rs) still runs its registration loop, calls test_nat_type() and lan::start_listening(), and get_rendezvous_servers() falls back to the hardcoded rs-ny.rustdesk.com (config.rs); stop-service disables the direct listener too, so no config makes the box serverless. The mediator keeps a persistent outbound connection, registers its pk, processes PunchHole/RequestRelay/FetchLocalAddr/ConfigureUpdate and the kcp-sys UDP parser pre-PAKE, and opens ephemeral UDP (the registration socket, STUN probes via test_ipv6→test_bind_ipv6, and KCP) — exactly the malicious-rendezvous adversary of §2. The fork MUST ship a direct-only build as the default that compiles the mediator out, so this surface is absent, not firewalled. Implementation precision (do not get this wrong): the direct listener direct_server/get_direct_port you must keep is nested inside the very start_all you are removing (spawned; defined) — a blunt #[cfg] over the module or over start_all deletes port 21118 with it. Instead lift direct_server — plus the ServerPtr it takes (from new_server(), server.rs), the genuinely-shared startup calls it relies on (check_zombie(), server.rs; scrap::codec::test_av1(), codec.rs — neither rendezvous-only), and on Linux linux_desktop_manager::start_xdesktop() (which start_all spawns when is_server() — the seat0/greeter capture-session discovery of R-S14/R-X14, not rendezvous; omit it and Linux capture never initializes) — into a minimal start_direct_only(), and compile-feature-gate only the registration loop, test_nat_type, lan, hbbs_http::sync, the KCP accept path, and the mediator's ~6 cross-module callers (ipc.rs, flutter_ffi.rs, ui_interface.rs, server.rs, the lib.rs re-export). Two verified subtleties: (a) removing start_all drops its probe calls, but it is not their only reach: the IPC handler also fires test_rendezvous_server (Data::TestRendezvousServer, ipc.rs) and test_nat_type (via the CheckTestNatType Drop guard at ipc.rs), and on the viewer/Android/iOS builds the flutter process entry fires both at startup (main.rs, flutter_ffi::initialize, ui_interface::set_option) — all independent of start_all. R-SV4(d) resolves this for every shipped artifact by making the entire test_nat_type/STUN subtree cfg-absent and deleting those callers, which turns R-A6's "absent" guard into a sound symbol-grep rather than the call-graph-only check a linked-but-unreachable probe would have demanded (a function that stays linked cannot call a cfg-absent one); on the controlled-only build (R-R2b) the viewer-side probe callers (main.rs, cli, flutter_ffi, mod client) are already compiled out, so even before R-SV4(d) only the IPC reach (removed with R-S11, the socks/ws pin of R-D6 making the Drop trigger unreachable) remained. (b) kcp_stream is imported unconditionally by the viewer path (client.rs, KcpStream::connect), so removing the mediator alone makes "the kcp-sys parser is absent" true for the inbound accept surface but leaves the C parser linked for the viewer. The direct-only build MUST therefore gate mod kcp_stream out entirely — the direct-only deployment connects by address over TCP in both roles, so KCP is needed nowhere — making the kcp-sys C parser absent for both roles, not merely on the inbound path (this resolves the one-binary-both-roles question of R-R2 for this build, rather than leaving it open). (Keeping the rendezvous code for a server-assisted build remains a non-goal to remove, §14 — this governs the direct-only deployment, not the code's existence.) This also removes the register_pk/key_confirmed liveness dependency, making R-P5's identity-key removal clean.
R-D5MUST Bind the direct listener v4-only; make IPv6 unreachability structural, not configorial
Upstream's tcp::listen_any (hbb_common/src/tcp.rs) builds a dual-stack socket (TcpSocket::new_v6() + set_only_v6(false) → binds [::]:21118, accepting v4 and v6), falling back to a v4 socket only if the v6 bind fails. On a static-IPv4 box that v6 face is attack surface with no purpose. The fork MUST bind the direct listener v4-only — add a listen_any_v4 (the existing v4-fallback body, ~6 lines) and call it from the lifted direct_server (R-D4) — so IPv6 unreachability is a property of the binary, not of a host sysctl or an ip6tables rule that can drift. This is "structural > config" applied to address families. Disabling host IPv6 and an ip6tables default-DROP remain sensible outer layers (environment, §14), but the innermost guarantee is the bind. Bonus: a v4-only box makes R-S10's entire IPv6-prefix limiter (connection.rs) dead code — retiring its hardest, highest-collateral-lockout subsystem and its /64 memory-amplification vector by construction.
R-D6MUST Outbound-silent by construction; the firewall is a tripwire, not the gate
A correct direct-only build makes zero unsolicited outbound connections: the rendezvous registration/heartbeat, NAT/STUN probes, and hbbs_http::sync go with the mediator (R-D4); the fetch-and-run updater and its version-check go with R-X1; ConfigureUpdate/custom_server with R-X3/R-X4; and there is no live telemetry (Firebase is compiled out). Steady state: accept inbound on 21118, dial nobody. This inverts the usual posture — the binary is the primary control (it cannot phone home because the code is gone), and an operator's egress firewall is a defense-in-depth tripwire: a uid/cgroup-scoped OUTPUT default-DROP (v4 and v6) on the service account makes any outbound packet from the RustDesk process an IOC — a regression, or a supply-chain dependency that phoned home (the §3.1/§13 chain is large and not cargo audit-clean today, so the backstop is earned). Verified hole to close so the invariant is unconditional: four audit/record POSTs survive R-D4 because they are config-gated, not compiled out — post_conn_audit (connection.rs, fired in on_open pre-authorization: after the fork keys the stream via the PAKE but before any password check, so it triggers on the mere keyed reach of any peer), post_alarm_audit (connection.rs, on whitelist-reject / rate-limit), post_file_audit, and session-record upload (record_upload.rs) — each silent only while the api-server option is empty/public, and each routable through an attacker set_socks proxy were api-server ever written locally (R-S11). The fork MUST hard-pin the api-server/audit/record endpoints empty (or compile these POSTs out) on the controlled build, so "the process makes no egress" is a structural fact, not a configuration accident. Two further egress paths the api-server pin does not cover: (1) the 2FA Telegram push POSTs a hardcoded https://api.telegram.org/bot…/sendMessage (auth_2fa.rs, leaking the box id + peer IP), fired on a 2FA-enabled login and gated on the bot/2fa options — not api-server, and not riding the mediator — so the fork MUST hard-pin bot/2fa empty and assert TelegramBot::get()==None (or excise send_2fa_code_to_telegram); and (2) enable-tunnel turns an authenticated peer into an outbound TCP proxy — the box dials any host:port the peer's PortForward names (connection.rs) — so "dial nobody" holds only with enable-tunnel pinned OFF on the controlled build (the same posture as the R-S5/R-X8 port-forward/terminal hardening). And the local probe/MITM trigger: socks and allow-websocket MUST be hard-pinned (empty/off) and immutable — a written socks would re-route any surviving egress (R-S11), and toggling either is what flips CheckTestNatType's is_direct and fires the STUN ephemeral-UDP probe from the IPC Drop path (R-S11); pin them in the controlled-side policy table (R-S16) — proxy-url and allow-websocket are in PINNED_SETTINGS, so the read funnel returns them empty/off and is_option_can_save (config.rs) rejects every local override, including the set_socks write whose own guard already keys on proxy-url (config.rs) — so the pin MUST use the proxy-url key, not a key literally named "socks", or it will not block. This is strictly stronger than the signed-custom.txt/OVERWRITE_SETTINGS route, which a fork cannot sign. Either way the probe is then unreachable. The telegram push, enable-tunnel, and the socks/ws pin MUST all be in R-A6's egress grep so the invariant is structural, not api-server-conditional.
R-D7MUST The direct-only hardening is per-target — Android and Windows ship it too, not just Linux
R-D2–R-D6 and §17 are written for the Linux .deb, and R-R2 says only "viewer and --server in each" — leaving silent whether the controlled-side hardening reaches the other two shipped targets. It MUST: the Android .apk and Windows .exe/.msi ship the same direct-only profile as the .deb — mediator excised (R-D4), v4-only (R-D5), egress-silent (R-D6), LAN discovery and terminal absent (R-X5/R-X8). A server-assisted build on either would re-import the §2 malicious-rendezvous adversary (pre-PAKE signaling, ephemeral UDP, the ConfigureUpdate + TestNatResponse re-home twins, phone-home) on the platforms with the weakest sandbox guarantees — a split posture where two of three targets default to the un-hardened surface. Fit with the hardened host

This is the rebuilt DMZ box: a static IPv4, the fork's RustDesk on a single v4-only TCP port (21118), zero UDP, IPv6 off, no Docker, no rendezvous, and no unsolicited egress (R-D6). The fork is the application; the host firewall, static IP, IPv6 toggle, and any out-of-band management channel are the environment (a non-goal of this spec, §14). The load-bearing claim is that the box is defensible even with no firewall at all, because the binary's reachable surface is that one PAKE-gated port — the firewall is a redundant tripwire, not the thing keeping it safe.

18. Sovereignty: no phone-home, no auto-update (universal — every artifact, every platform, every code path)

§17 makes the exposed controlled box egress-silent (R-D6) and updater-free (R-X1), but for that one artifact and largely by config-pin — "silent only while the api-server option is empty," "pin bot/2fa empty" — leaving the code compiled and one local write away (R-S11's Data::Socks proxy shows that write is locally reachable). The full viewer build the operator runs, and the Apple code paths (§16, R-R2), retain the lot. This section raises the bar to sovereignty: every running client or server reaches no global resource and carries no auto-update — by compile-out, not config-pin, on every artifact, and (per R-R2) enforced on the macOS/iOS source too. The only surviving network capability is the direct, PAKE-gated peer connection by IP (R-F3 as direct-IP, R-S13). Sovereignty becomes a property of the artifact and the tree, not a setting one trusts.

Two conclusions follow, and the reader is meant to draw both.

The pinned base is unrepaired. The RustDesk 1.4.7 tree this fork builds on was committed on 2 June 2026 — nearly three months after that disclosure — and still ships the vulnerable code for every client-side item in this table. Where upstream touched a finding at all, it did so by halves: the fixed all-zero nonce behind CVE-2026-30785 became a random per-record nonce in the symmetric_crypt routine of hbb_common's password_security module, yet the encryption key is still the raw machine UUID zero-padded to length with no key-derivation function (#14); and the invalid-certificate-accepting outbound path behind CVE-2026-30794 is now gated behind an allow-insecure-tls-fallback option that is off by default, so it is no longer a live exposure on a stock build and earns no row of its own. The only defensible reading is that upstream RustDesk does not treat the confidentiality and authentication of an internet-exposed session as a release-blocking concern — which is precisely the premise this fork exists to reverse.

The single most consequential finding is absent from that record. #1 — the live session transport reusing one libsodium secretbox key in both directions — is, to the limit of an exhaustive search of the CVE batch, the disclosure's own proof-of-concept dossier, RustDesk's SECURITY.md, and every upstream encryption discussion thread, unreported as of this writing. It must not be conflated with the widely-cited CVE-2026-30785. That identifier is the configuration file at rest — the password_security module, a literal all-zero nonce, a machine-UUID key — exploited by a local reader of the file. #1 is the live wire — the Encrypt type in the tcp module of hbb_common, sequence-counter nonces, one key in both directions — exploited by a passive network eavesdropper who never touches the host. They are different defects, in different code, with different victims. Any disclosure of #1 must make that distinction its first sentence, or be dismissed as a duplicate of CVE-2026-30785.

Finding #1 in full — key establishment, blast radius & proof of concept

Why one key serves both directions. The session key is generated once, by the connecting viewer. In create_symmetric_key_msg (the common module of rustdesk/rustdesk) the viewer calls secretbox::gen_key() a single time, seals that one key to the controlled side inside an ephemeral Curve25519 box (NaCl crypto_box — a fresh key-pair is generated per handshake, so that layer's own all-zero nonce is harmless), sends it as the PublicKey.symmetric_value field, and installs it locally through set_key. The controlled side recovers the identical key with Encrypt::decode and installs it through its own set_key. Both endpoints now hold a single Encrypt value wrapping that one key, and each seals its outbound frames and opens the peer's inbound frames under that same key with sequence-counter nonces that both start at zero. The two-time pad is therefore structural — not that the two directions might collide, but that frame N in each direction is guaranteed to.

Blast radius — this is not a direct-IP vulnerability. The defective code lives in hbb_common, vendored verbatim by both rustdesk/rustdesk (the client, pinned here at 1.4.7) and rustdesk/rustdesk-server (which builds both the rendezvous server hbbs and the relay server hbbr), and it is engaged on every transport that keys the secretbox: NAT-hole-punched TCP, intranet TCP, KCP-over-UDP, and relayed TCP — that is, the ordinary identifier-based connection that the overwhelming majority of RustDesk users make through a rendezvous server and, on relay fallback, a relay server. The serverless direct-IP listener is the one path this defect does not touch, precisely because it never keys the secretbox at all — it transmits in cleartext, the separate finding #2. The relayed case is the most exposed: hbbr is a blind byte-pipe carrying the end-to-end stream, so the relay operator — including whoever runs RustDesk's default public relay infrastructure — sees both reused-nonce ciphertext streams with no additional network position.

Proof of concept — passive capture suffices. Record one active session's frames in both directions (from a shared local network, any transit hop, or simply as the relay operator). For each sequence index N, exclusive-or the controlled-side ciphertext with the viewer-side ciphertext: the XSalsa20 keystream cancels, leaving the exclusive-or of the two plaintexts. Seed crib-dragging with the fixed leading protobuf tag bytes that prefix nearly every frame (for instance the video_frame and key_event field numbers) and with structurally predictable content such as video key-frame headers and cursor coordinates; recovering either direction's frame at index N immediately yields the other. No key recovery and no man-in-the-middle are required for disclosure. For an active attacker the reused Poly1305 one-time key additionally admits frame forgery — injected input or spoofed content that authenticates correctly.

The remedy is upstream's, and it is generic. Independently of this fork's PAKE, RustDesk must derive two distinct directional keys from the established session key — for instance HKDF with separate info labels for the controlled-to-viewer and viewer-to-controlled directions — or, at minimum, fold a direction byte into the nonce so the two directions can never share a (key, nonce). The change is confined to the Encrypt type in hbb_common and the handshake's set_key call sites, which would pass an initiator/responder role; it is wire-compatible and repairs every server-mediated connection at once. This fork applies the same correction as a by-product of R-P2/R-P10 (per-direction keys derived from the PAKE secret), but the defect and its fix belong to upstream and its entire hbbs/hbbr user base — not to the serverless direct-IP build.
