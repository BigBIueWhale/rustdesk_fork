Security Requirements for a Hardened RustDesk Fork

Every connection method, every session type, and every byte of content — encrypted and mutually password-authenticated from the first byte, by the only path the program offers, and secure by assertion, not merely by default. The bar is SSH.

Normative specification for github.com/BigBIueWhale/rustdesk_fork, a fork of upstream RustDesk pinned at 1.4.7 (release commit 0c86d46) with its protocol submodule libs/hbb_common at df6badc. Every architectural claim here was verified by direct read of that exact tree (279 Rust files), cited file:line. RFC 2119 key words (MUST, MUST NOT, SHALL, SHOULD) are normative.

The thesis, in one paragraph

RustDesk's serverless "Direct IP" mode applies no encryption: the login challenge, the salted password hash, keystrokes, screen, files and clipboard cross the wire as plaintext, so the hash is brute-forceable offline on a GPU farm and an active attacker can inject input into an authenticated session or man-in-the-middle it on first contact — without the password. Upstream declined to fix this. This fork keeps RustDesk's excellent Rust + Flutter engine and replaces its cryptographic judgement: it inserts one mandatory, mutually-authenticated PAKE at the single choke point through which all six inbound transports flow, so that all five session types and every auxiliary content channel are keyed before a single application message is parsed — on the connecting side as much as the accepting one, closing the viewer's own unkeyed direct-IP path (R-S13); it makes those properties runtime invariants the program refuses to operate without; and it excises the fetch-and-run updater, the unvalidated native-plugin loader, the remote server-list rewrite, and the trust-anchor overrides that have no place on an internet-exposed host. The result is meant to be, on the wire, as defensible as SSH.

1. Purpose, scope & the SSH bar

The fork's objective: reaching a controlled machine by its own address is encrypted and mutually password-authenticated from the first byte, on every connection method and every session type, by the only code path the program offers — and the program asserts these properties and aborts rather than operate without them.

The bar is SSH. Concretely, the fork must match SSH on the channel and authentication properties that matter and may exceed it. One honest calibration up front: because the fork authenticates by password, the apples-to-apples comparison is SSH password authentication — SSH public-key auth carries far more credential entropy, which R-S9's strong-password requirement only partly offsets.

Operator's note on man-in-the-middle. The operator considers active MITM a lesser concern in practice (a wrong desktop is noticed quickly; the password gate already raises the bar). The fork nonetheless closes MITM by construction via the mutual PAKE — so this requirement is met "for free" and is not relaxed; it simply is not the load-bearing reason for the design.

Three scoping decisions are load-bearing and non-negotiable:

  • All connection methods and all session types. Six inbound transports, five session kinds, and every auxiliary content channel (§4) — not just screen control.
  • Secure by assertion. Every property below is backed by a runtime, build-time, or test-time check that fails closed (§9).
  • The whole client, every platform. The controlled side and the viewer are one cross-platform binary selected by argv; both roles and all shipped platforms are in scope (§3, §12).

Licensed, like upstream, under AGPLv3; corresponding source for every published binary MUST be published.

2. Threat model

Adversary: an active attacker on the network path (café/transit Wi-Fi, ISP, any transited AS, or a host already on the LAN) who can read, drop, modify, replay and inject packets and can initiate connections to the controlled machine's listening port; and, separately, a malicious or compromised rendezvous server the operator might otherwise rely on. The adversary does not know the password. The controlled machine is reachable from the internet (home-router DMZ on a static address). We defend the channel and the host's exposed surface; we do not defend a compromised endpoint or an adversary who knows the password. Both roles are in scope (§1): the same binary also initiates connections, so the adversary additionally includes a malicious peer the operator is induced to connect to — a poisoned or mistyped address, a QR code, or a rustdesk:// deep link — against which the fork holds the initiator to the same keyed-before-any-message bar as the responder, and treats every byte a connected peer returns as attacker-controlled (§4.4).

Failure mode A — offline-crackable credentials

Upstream sends salt+challenge in cleartext and the answer is a fast, unstretched SHA-256(SHA-256(pw‖salt)‖challenge); captured once, brute-forced offline on a GPU farm. Online limits are irrelevant. No credential material may be recoverable from the transcript.

Failure mode B — passwordless MITM & injection

No channel encryption ⇒ no message authentication. Upstream checks the password once, flips self.authorized=true (connection.rs:1544), then trusts every subsequent frame; an active attacker injects input into the authenticated stream, or MITMs first contact, without the password. Authorization must be a property of a mutually-authenticated, encrypted channel.

Failure mode C — host expansion (pivot, RCE, re-homing)

An exposed controlled host must not be a launch point. This fork additionally treats the updater, the plugin loader, the server-pushed config rewrite, and the rustdesk:// deep-link writes as parts of the attack surface to be excised (§8) — these are reachable by the network or a malicious rendezvous server without the password, so they are squarely in scope. (The file system, reachable only by an already-authenticated peer, is deliberately a single unconfined, full-filesystem mode per R-S8 and §2 — confining it would mean defending against a password-knower, which we do not, and a confinement toggle is itself forbidden by R-S12.)

The single choke point: all six session transports construct a Stream and hand it to create_tcp_connection (src/server.rs:190), which today calls stream.set_key() for the secure case before Connection::start. Default serverless exposure is one port (21118), and only if direct-server=Y. The hardened box's reachable surface is therefore exactly this single v4-only port (R-D3, R-D5) — understood and accountable, not merely firewalled.

4.2 The pre-authentication message surface

"Pre-auth" = processed while self.authorized==false. The gate is the single } else if self.authorized { at connection.rs:2691; only five message arms execute before it (one benign), plus per-connection setup in on_open:

  • on_open (:1320): check_whitelist is default-OPEN (empty list ⇒ every source IP passes, :1291); emits Hash{salt,challenge} unconditionally (a pre-auth oracle, by design).
  • Misc/CloseReason (:2374): deliberately placed ahead of the gate so a peer can close pre-auth — benign: logs the reason and tears down the session, parses no other attacker data, grants nothing.
  • LoginRequest (:2382 → :2315): the highest-value pre-auth surface — stores the full request, parses OptionMessage, performs a trusted-device 2FA bypass keyed on attacker-supplied hwid/id (:2321), and on some branches starts the desktop/CM session and sends a Telegram 2FA push before the password is checked.
  • Auth2fa (:2612): TOTP check; persists a trusted device from attacker fields on success.
  • TestDelay (:2647): a free pre-auth echo/reflector.
  • SwitchSidesResponse (:2663): an authorization path that bypasses the password on a matching 10-second UUID, and sets from_switch which also skips 2FA.

Direct-path note: the SignedId/PublicKey key-bootstrap parse is gated if secure (server.rs:200) and the direct listener passes secure=false, so on the direct path that parse — and AddrMangle::decode — do not run pre-auth (both are mediator/KCP/relay-only); the only attacker-byte parsers before keying are the frame codec and the CPace protobuf (R-P14). Separately, lan.rs:35 parses unauthenticated UDP-broadcast datagrams and replies with host/user info — a distinct pre-auth parser + info-leak, excised by R-X5.

Pre-auth DoS — unbounded frame length

The frame codec's max_packet_length defaults to usize::MAX and set_max_packet_length is called nowhere in the tree (verified). The length header allows ~1 GiB frames; buffering grows with the declared length. Any peer that reaches a listener can force ~1 GiB/connection memory amplification before authentication.

4.3 What rides the keyed stream — and the two escapes

Verified: every byte of peer-bound content, for all five session types, rides the single keyed conn.stream as a protobuf Message — remote control, file transfer, view-camera, terminal, clipboard text and CLIPRDR file bytes, whiteboard input, remote-printer jobs (streamed as file Blocks), and audio/voice. Auxiliary "IPC" channels (CM, audio, whiteboard, portable-service shared memory) are localhost privilege-separation only and never touch the network. So one PAKE at the choke point covers all of it — with exactly two exceptions:

  1. Port-forward / RDP tunnel payload. After login, both ends call set_raw() (connection.rs:1220 server; port_forward.rs:190 client), which drops the secretbox (tcp.rs:143) — so the tunnelled RDP/VNC/SSH bytes cross the link in plaintext, even though the auth was keyed.
  2. File-transfer path confinement. File names are well-validated (fs.rs rejects .., null, absolute, UNC, and symlink components), but the destination base directory is attacker-supplied and unconfined — an authenticated peer may write/overwrite/rename/recursively-delete anywhere the service user can reach (~/.ssh, /etc, C:\Windows). There is also an acknowledged symlink TOCTOU (path-validated, then opened by path; the openat/O_NOFOLLOW fix is deferred upstream).

4.4 The initiator's (viewer's) outbound surface

The same binary is also the viewer, and §1 puts both roles in scope — so the connecting side is audited symmetrically, not assumed safe. Verified in the cloned tree, the initiator carries four surfaces the responder analysis (§4.1–4.3) does not cover, and the spec's keying requirements (§7) act on each:

  1. An unkeyed direct-IP path — the initiator twin of failure modes A & B. When the viewer dials a bare address, Client::_start returns a live stream with no keying for both the IP case (client.rs:257, is_ip_str) and the domain:port case (:272) — both return before secure_connection is ever reached, and the only client-side is_secured() read (io_loop.rs:184) merely sets a UI lock-icon, never aborts. So the flagship serverless mode (§17: a client reaches <static-IPv4>:21118) runs plaintext on the connecting side: the login hash is captured for offline cracking and first contact is MITM-able, exactly as on the controlled side. Keying the responder (R-S1) alone does not close this.
  2. A wide pre-key decoder. On the rendezvous/relay path the viewer calls Message::parse_from_bytes over the full application union before keying (client.rs:795, ahead of conn.set_key at :808), on frames whose length is unbounded (set_max_packet_length is never called on the client path) — the initiator-side twin of the responder's wide-decoder + ~1 GiB DoS surface (§4.2).
  3. The kcp-sys C parser, linked for the viewer. mod kcp_stream is ungated (lib.rs:79) and the viewer reaches the C reassembly parser pre-key via KcpStream::connect (client.rs:4257, from udp_nat_connect) — so excising the mediator (R-D4) does not by itself remove KCP from the connecting role.
  4. The deep-link auto-connect. rustdesk://<id>?password=…&/r@<server>?key=… initiates an outbound session with no confirmation, honoring an embedded password, relay, and — through the Rust initialize (client.rs:1818, written into other_server) — an embedded trust-anchor key, delivered over four un-authenticated transports (Linux D-Bus, Windows WM_USER+2, macOS _url IPC, Android intent).

Beyond keying, a viewer induced to connect to a hostile peer feeds attacker-controlled bytes into the in-process native decoders — libvpx/aom/FFmpeg/libyuv (video), libopus (audio), zstd (cursor/clipboard/file blocks), and the Windows CLIPRDR C parser (the io_loop.rs dispatch). The PAKE authenticates the peer but cannot make a password-correct-but-hostile box safe; this residual is bounded operationally (deliberate-connect + channel restriction, R-S13/R-X6), not cryptographically, and is recorded as such (Appendix C).

5. Definitions

PAKE. A protocol by which two parties sharing only a password derive a strong, mutually-authenticated session key with no certificates or pre-shared keys; a network attacker without the password cannot complete it, cannot MITM it, and learns nothing offline-crackable — reduced to one online guess per attempt. Balanced = symmetric (both hold the same password) = the "me and my own machine" case.

Mutual key confirmation. After deriving the shared key, each side sends a MAC (keyed by that shared key, over its own handshake message) and verifies the peer's — proving both derived the same key, i.e. both knew the password. One-sided proof is not authentication; it is exactly the hole that sank the prior community fork (a constant b"direct-ok" reply). Exact construction: R-P3.

Transcript binding. Tying the derived keys to everything negotiated — both ephemeral public values, the session id sid, and the protocol/version label carried in the channel identifier CI — so no part can be spliced, downgraded, or replayed cross-protocol. (There are no long-term identities to bind; see R-P5.)

The five session types. The protobuf LoginRequest.oneof union (and the server enum AuthConnType{Remote,FileTransfer,PortForward,ViewCamera,Terminal}): remote control, file transfer, port-forwarding (incl. tunnelled RDP as PortForward{host:"RDP",port:0}), view-camera, remote terminal. The kind is read after login — which is why the stream must be keyed before it.

Excise. Remove from the build so the path cannot be reached, configured back on, or accidentally re-enabled — stronger than "disabled by default."

6. Functional requirements (preserve)

R-F1MUST Preserve all five session types (terminal excised from the exposed default — R-X8)
Remote control, file transfer, port-forward (incl. RDP), view-camera, terminal — all preserved as code and all secured (§7). Caveat (terminal): post-PAKE the terminal spawns a shell as the service user — root on the exposed box (R-D3) — with no Linux re-auth (should_use_terminal_os_login_scope is Windows-only, connection.rs:130; the Linux token path is a stub, :3509), so one PAKE password yields a direct root shell — more permissive than the bar this spec holds itself to. Defaulting enable-terminal OFF is not enough (it is a runtime-writable option a password-knower can flip back, exactly what R-S12 forbids — and it defaults ON upstream): the shipped direct-only default build MUST excise the terminal entirely (R-X8). Preserving the code satisfies "all five types" for an attended, non-exposed build; the exposed deployment ships it absent — the same split R-D4 applies to the mediator.
R-F2MUST Preserve GUI file transfer, session persistence, parallel & multi-machine connections
Drag-and-drop transfer; reconnect to the same live desktop; phone + laptop at once; more than one controlled machine.
R-F3MUST Preserve "connect by address, from anywhere, any device"
No server, account, VPN, or pre-shared key may become required to connect.
R-F4MUST The direct port is a single pinned compile-time constant (21118)
The direct listener binds one TCP port, pinned at compile time to 21118 — not a runtime option, env var, or config key. It folds into the PAKE channel binding CI (R-P1), so it is a one-mode constant, not a knob; an operator who needs a different port changes the constant and rebuilds (a build-time choice, never a runtime mode). So the host's "one audited port" discipline holds, and no dynamic UDP range is needed for the serverless path.

7. Security requirements (the core)

R-S1MUST An unconditional mutual PAKE at the single choke point is the only way to connect
The PAKE (§10) MUST run inside create_tcp_connection (server.rs:190) before Connection::start, keying the stream for all six transports of §4.1. The secure parameter MUST be deleted (not honored) — note the direct path passes secure=false, so a naïve "encrypt if secure" would skip the very path we care about; and the relay path's secure is attacker-controlled (rr.secure) and MUST be ignored. The same bar binds the initiator: the viewer's unkeyed direct-IP and domain:port early-returns (client.rs:257/:272, §4.4) MUST be deleted and every viewer initiation routed through the CPace initiator handshake (R-P14) before any Message is parsed — keying the responder alone leaves the connecting side plaintext (R-S13). The plaintext path, the unverified-trust escape hatch, and the manual fingerprint ceremony MUST be deleted.
R-S2MUST Authorization is a property of the PAKE-keyed channel
self.authorized=true (:1544) MUST be reachable only after both sides' key-confirmation MACs verify. Every grant/resume path — password login (validate_password :2178), the 30-s reconnect (is_recent_session :2221), and SwitchSidesResponse (:2663) — MUST require a confirmed, keyed stream. The password-bypass via switch-sides UUID and the trusted-device 2FA bypass (:2321) MUST be re-gated behind the PAKE or removed.
R-S3MUST Every post-handshake byte is encrypted + integrity-protected + anti-replay
The PAKE-derived per-direction keys (R-P2) MUST drive the secretbox layer for the whole connection; each frame carries a Poly1305 tag and a monotonic sequence; any auth/sequence failure aborts (fail-closed).
R-S4MUST All five types + all auxiliary content channels are covered
Keying the single stream before the oneof type is parsed covers remote/file/port-forward/camera/terminal and the clipboard-text, CLIPRDR file bytes, whiteboard input, printer, and audio channels (all verified to ride conn.stream). The fork MUST NOT implement the handshake per-feature.
R-S5MUST Seal the port-forward / RDP set_raw() escape
The key-strip at connection.rs:1220 and port_forward.rs:190 MUST be removed so tunnelled bytes stay inside the secretbox — or the forward MUST be refused. There is no opt-in plaintext mode — a default-off-but-reachable raw tunnel is exactly what the §1 bar ("the plaintext path is deleted, not defaulted off"), R-S12, and acceptance criterion 3 ("the set_raw escape is sealed") forbid. Initiator note: removing the client-side strip (port_forward.rs:190) is necessary but not sufficient — the viewer MUST also assert the stream it tunnels was PAKE-keyed (is_secured(), R-S13), since the inherited direct-IP initiator path produced an unkeyed stream in the first place (§4.4); a sealed strip over an unkeyed stream still tunnels plaintext. This one in-band plaintext escape is closed, not toggled.
R-S6MUST No offline-crackable credential material on the wire
The balanced PAKE provides this structurally — the transcript reveals nothing brute-forceable and the attacker is reduced to one online guess per interaction, bounded further by the per-IP rate limiter (R-S10 — which the fork MUST re-key onto the PAKE outcome, since the existing limiter sits on the now-removed password path) and the required strong password (R-S9). The upstream fast SHA-256 challenge/response MUST be removed from the security boundary. No memory-hard password pre-hashing is used or needed: a balanced PAKE has no stored verifier for an Argon2/scrypt to protect (that is its only job), and a per-user salt cannot be agreed in a serverless symmetric setting — so the PRS is the normalized password directly (R-P1). Closes failure mode A.
R-S7MUST Cap frame length before authentication
The fork MUST call set_max_packet_length to a small bound — ≤ 4 KiB pre-PAKE (only the ≤120-B CPace messages of R-P14 flow before keying) — on every inbound connection before the PAKE completes, and a sane ceiling after — ≈ 32 MiB (far below the variable-length header's 1 GiB ceiling, well above the 128 KiB file block at fs.rs:952 and any realistic video keyframe; the inherited 256 KiB speculative-prealloc cap at bytes_codec.rs:6 bounds only the initial reserve, not the declared length), closing the ~1 GiB/connection pre-auth memory-amplification DoS (§4.2). Pre-PAKE protobuf is the only attacker-reachable parser before keying — and it MUST decode only the dedicated Cpace message (R-P14), via parse_from_bytes::<Cpace>, never the full application Message union — so the reachable decoder is the handshake's fixed fields, not the whole schema (minimize what is decoded, not merely what is acted on). The kcp-sys KCP packet parser is dropped in the direct-only default build: KCP is driven solely by the rendezvous mediator (rendezvous_mediator.rs:932/:976), which R-D4 compiles out — and the direct-only build gates mod kcp_stream out entirely (no KCP in either role, since both connect over TCP), so that C parser is absent for both roles, not merely minimized. This cap is the load-bearing pre-auth DoS control: because set_max_packet_length is currently never called anywhere in the tree, the ~1 GiB/connection amplification is the dominant pre-auth memory-DoS — a larger exposure than the per-IP limiter (R-S10) addresses, and it fires per-connection regardless of source IP, so the frame cap is primary and the limiter only a secondary flood-shed.
R-S8MUST Close the symlink TOCTOU; file transfer is a single full-filesystem mode (no confinement toggle)
File names are already well-validated (fs.rs rejects .., null, absolute, UNC, symlink components); only the destination base is attacker-supplied. Per §2 the authenticated peer holds the password and is the trusted owner — and remote control already drives the box at the service privilege — so file transfer legitimately operates at that privilege (root on the exposed box) with full filesystem reach, as a single mode; this capability MUST be documented. The fork MUST NOT add a confinement option (a configurable allowed-root): a security-relevant toggle is exactly what R-S12 forbids — it is state that can be misconfigured, it would regress the legitimate "put a file anywhere on my own machine" use, and confining file transfer while the root desktop stays reachable is pointless (the peer has root either way). What is not optional is write-path safety: every write MUST use handle/descriptor-based no-follow opens (openat + O_NOFOLLOW / Windows equivalent) to close the acknowledged symlink TOCTOU (§4.3) — defending the peer's own writes against a local symlink swap, a correctness guarantee asserted on every write (R-A5), not a confinement feature. Consistency with R-X8: the discipline that excises the terminal is toggle-and-surface removal (a default-on flippable path + redundant code), not capability denial — so keeping file transfer as one full-FS mode while adding no confinement knob is the same principle, not an exception to it.
R-S9MUST Default-deny the source whitelist; strong permanent password, permanent-only + password approve-mode
check_whitelist MUST be inverted to default-deny on an exposed host (an unset whitelist blocks, not passes). The fork MUST require a non-empty permanent password, hard-pin verification-method = use-permanent-password (the one-time-password path is excised — R-X7), and set approve-mode = password (not click-to-accept). The PAKE's PRS is this permanent password (R-P1). The permanent password is stored as a salted hash (good — keep); the config file MUST still be treated as secret by filesystem permissions (its at-rest wrapper is keyed by the machine UUID, i.e. obfuscation, not secrecy against a local reader). Balanced-PAKE consequence: the controlled side must hold a password-equivalent value (the PRS) to run CPace, so a config read yields password-equivalent material — inherent to a balanced PAKE and acceptable here because endpoint compromise is out of scope (§2). An augmented PAKE (e.g. OPAQUE) would let the controlled side store a non-password-equivalent verifier instead, but reintroduces a stored-verifier and asymmetric initiator/responder model — deliberately rejected for this symmetric, single-user, sovereign design.
R-S10MUST Constant-time secrets; a DoS-safe rate limiter, re-keyed onto the PAKE
Every secret-touching comparison MUST be constant-time (upstream already uses constant_time_eq for the password — MUST NOT regress to ==). The per-IP failure limiter (LOGIN_FAILURES, connection.rs:77) MUST be kept and re-keyed onto the PAKE, but billed honestly: it is cheap CPU-flood shedding, never credential defense (the strong password + one-guess-per-PAKE is that), so it MUST be tuned loose. Cardinal principle — the limiter must never become the attacker's weapon against the owner: it MUST key on the connecting source and MUST NOT be global or connection-count-wide, so a flood from one source can never lock the legitimate owner (a different source) out of their own machine; and every block MUST decay (see (c)) into a self-healing tarpit, never a permanent ban — a limiter that denies the owner service is doing the attacker's job for them. Direct-IP is the clean case for this: the box keys on the peer's real TCP source address (the accepted socket's addr in the lifted direct_server), not a relay-collapsed one. As inherited the limiter is not DoS-safe, so three coupled fixes are mandatory:
  • (a) Re-key onto the PAKE; gate before the scalar-mult. The counter is bumped/checked only inside the LoginRequest password path (validate_password, :2178), which the PAKE now precedes — a failed PAKE never reaches it. A failed key-confirmation MUST increment the entry, and the choke point (server.rs:190, peer IP in hand) MUST check the threshold and reject before running the PAKE, shedding floods ahead of the scalar-mult. This is the limiter's real job — cheap CPU-flood shedding, not credential defense: the strong password (R-S9) plus one-guess-per-PAKE is the structural brute-force defense, so the limiter may be tuned loose. Two implementation notes: the limiter is a Connection method keyed on self.ip, but the choke point has no Connection yet — it MUST be lifted to a free function taking the raw peer IP; and only a CPace key-confirmation failure may count (a wrong password or a MITM), never a transport/parse abort, so handshake noise cannot inflate the counter.
  • (b) Bound the memory. Verified: LOGIN_FAILURES only ever grows — an entry is removed only on a successful login; no size cap, no LRU, no time eviction — and it stores a per-full-address entry even for IPv6 (it blocks by /64–/48 prefix yet still inserts the full address). An attacker owning one IPv6 /64 (264 addresses) thus grows it without bound — a memory-exhaustion DoS — and counting pre-PAKE widens that to cheap pre-auth garbage. The fork MUST bound the structure (a hard size cap — e.g. ≤ 8192 entries — plus LRU/time eviction; the inherited map is a never-evicted HashMap<String,(i32,i32,i32)> at connection.rs:77) and key IPv6 by prefix (≤ /64), never the full address. On the v4-only box (R-D5) this entire IPv6-prefix subsystem is dead code — get_ipv6_prefixes (:3753) returns None for v4 peers, and try_into_v4 (connection.rs:540) normalizes v4-mapped addresses first — so the limiter collapses to a bounded, decaying, per-v4-address counter and the /64 amplification vector cannot occur; drop the prefix machinery on that build.
  • (c) Decay the block; never lock out permanently. Verified: the cumulative block (failure.2 > 30 per IP, > 100 per /48) is a lifetime counter that never decays — once tripped, that source/prefix is blocked until a successful login or a restart. That is a permanent-lockout weapon against any shared egress (CGNAT or other shared mobile/NAT egress) or a shared /48. The fork MUST window/decay the counter (a tarpit that expires) so legitimate and collateral sources self-heal. The inherited thresholds are cumulative, never-decaying lifetime counts — >30 per IP, >60 per /64, >80 per /56, >100 per /48 (connection.rs:3920-3939) — atop a per-minute burst shed (>6/min, :3950); keep the per-minute shed, but make the cumulative count decay (e.g. clear an entry after ~60 min with no new failure) so it is a tarpit, not a lifetime ban.
Topology note: the controlled box keys on the connecting source IP, and the legitimate viewer almost always connects from a different source than an attacker (and TCP source cannot be spoofed through a full handshake) — so the common case is an attacker throttling only itself; (b) and (c) defang the residual shared-prefix/CGNAT collateral. Direct-IP mode is the clean case for per-IP limiting — the peer's real address is visible (no relay IP-collapsing), so a bounded per-IP counter is genuinely meaningful there; conversely, if ever deployed behind IP-collapsing infrastructure (a relay), add an application-level identity to the failure key. Observability gap to close: upstream emits no log line on a failed login or a rate-limit block (the wrong-password branch only bumps the counter and sends a protobuf error; post_alarm_audit is dead with no api-server), so the fork MUST emit a structured key-confirmation-failure record (source IP + outcome) at the choke point — the only audit signal on a serverless box, and the prerequisite for an optional host-level fail2ban backstop, which (being post-hoc and log-driven) complements but cannot replace the in-process pre-PAKE shed.
R-S11MUST The local IPC and service are not bypasses; least privilege
The UI⇄service IPC MUST stay local-only with its parent-dir hardening intact; no network peer may reach session functionality without the PAKE. The controlled-side service MUST run least-privilege (failure mode C). The daemon IPC handler is itself a config-integrity boundary: the main channel today writes any config key via Data::Options→set_options, the permanent password via Data::Config, and the whole config via Data::SyncConfig with no allowlist (ipc.rs:914,:893,:933) — so a local process could reverse the §8 excisions and §9 hard-pins from inside. The fork MUST allowlist the keys writable over IPC and reject trust-anchor / credential / hard-pin writes there (the world-facing _service socket already restricts writes to only SyncConfig; the main channel restricts nothing). On Windows that main pipe is created allow_everyone_create and accepts any same-session process (ipc.rs:589, auth.rs) — the fork MUST also tighten it to a same-user DACL. The allowlist MUST be applied per write-arm, not just to the set_options key set: the same main channel also exposes Data::Socks → set_socks (ipc.rs:794, a proxy that re-routes all traffic — a local MITM primitive an Options-key allowlist would miss), the salt / unlock-pin / id+set_key_confirmed(false) sub-keys of Data::Config, and Data::RemoveTrustedDevices/ClearTrustedDevices. Make it a positive chokepoint, not a denylist: route every IPC config write through a single allowlist function and CI/AST-assert that no Config::set*/set_socks/set_permanent_password is reachable from the main-channel handler except through it — so the handler cannot write a key that is not on the list, rather than the spec having to enumerate every dangerous arm. (Upstream 1.4.7 already gates the IPC transport — per-uid 0600 sockets + SO_PEERCRED — so this is the residual write-authorization gap, not the socket itself.)
R-S11aMUST Executable-match is never the sole IPC authorization; fresh credentials; no /tmp adoption
Verified: the cross-uid IPC accept gate combines a kernel-attested peer credential (SO_PEERCRED uid: root-or-active-user, auth.rs:168) with an executable-path match (/proc/<pid>/exe equals the service binary, auth.rs:462). The exe-match alone is not an authorization — any local user may run the genuine root-owned /usr/bin/rustdesk and satisfy it — so it is only sound behind the uid gate, and CI MUST AST-assert that no accept path reaches the exe-match without a preceding uid check (individually weak, jointly sound; a refactor must not drop the uid gate and leave exe-match standing). Two further hardenings the inherited code leaves open: (a) the _service channel MUST authorize against a fresh active-user lookup (as the _uinput_* path already does, uinput.rs:928), not the service-loop cache, so a just-switched-out user cannot pass in the cache-lag window; and (b) the service MUST NOT adopt a pre-existing world-traversable /tmp/{app}-service it did not create — on owner mismatch it MUST reject-and-recreate (no fchown-adopt), and a startup assertion (R-A4) MUST confirm the parent dir is 0o0711 root-owned with each socket at its expected mode before listening. (The world-mode 0o0666 on the _service/_uinput_* sockets is a deliberate consequence of the root-service ↔ user---server cross-uid design, mediated by these gates — not a permission bug; Appendix B. On the §17 Xorg build the _uinput_* sockets are compiled out entirely — R-X13, since uinput is the Wayland input path — leaving only _service world-mode there.)
R-S12MUST No insecure build/runtime configuration is reachable
No feature flag, env var, config key, or CLI flag may disable/weaken the PAKE, the encryption, the no-follow write guarantee (R-S8), or the §9 assertions. Such controls MUST be removed, not defaulted off.
R-S13MUST The initiator (viewer) is held to the responder's bar
§1 puts both roles in scope, so the connecting side meets the same keyed-before-any-message guarantee (§4.4) — the inherited initiator path does not, and keying the responder (R-S1) does not fix it. The fork MUST: (a) delete the unkeyed direct-IP / domain:port early-returns (client.rs:257/:272) and route every initiation — direct, rendezvous, relay — through the CPace initiator handshake (R-P14), so a usable Stream is never returned without a confirmed PAKE; (b) assert is_secured() before the viewer dispatches or tunnels any byte — promoting the inherited informational check (io_loop.rs:184, today only a UI lock-icon) to a fail-closed abort, the initiator analog of R-A1; (c) decode only the dedicated Cpace message pre-key on the client path (parse_from_bytes::<Cpace>, replacing the full-union Message::parse_from_bytes at client.rs:795), and call set_max_packet_length to the small pre-PAKE bound before the first read (R-S7); and (d) connect TCP-only — gating mod kcp_stream out (R-D4) and removing the udp_nat_connect punch path (client.rs:703/:706/:547) so the kcp-sys C parser is absent in this role too. Residual (documented, not closable by keying): a viewer deliberately connected to a hostile but password-correct peer still feeds attacker bytes to the in-process native decoders (§4.4); this is bounded by the deep-link confirmation and channel restriction of R-X6, not by the PAKE (Appendix C #2b). The TCP-only choice costs UDP NAT traversal — acceptable for the static-IP direct deployment (§17), where connecting-by-address (R-F3) is preserved over TCP.
R-S14MUST Screen capture is bound to a PAKE-authenticated session — a reused grant must not capture outside one
The security boundary on screen capture is the PAKE (R-S1/R-A1), not the OS consent dialog — a distinction that matters because this fork targets unattended access (R-S9). On Android, MediaProjection consent is a system dialog the user must physically tap and which cannot be auto-granted, so the inherited token reuse (MainService keeps the projection alive across session end via reuseVirtualDisplay/setSurface(null), "to avoid media projection confirmation dialog every connection", MainService.kt:213,:447-451,:534-553) is precisely what makes unattended reconnection possible — mandating a fresh per-session consent would break the use case. The requirement is therefore not re-consent but binding: capture MUST begin only inside a confirmed PAKE session — no held grant (the Android projection token, or the Wayland xdg-portal restore-token at libs/scrap/…/wayland/pipewire.rs:746) may drive capture pre-authentication, and the held grant MUST NOT be reachable by any non-authenticated path. The persisted Wayland restore-token is a local at-rest exposure (a local reader could lift it) in the same §2 endpoint-compromise-out-of-scope class as the UUID-obfuscated config (Appendix C #14); the fork SHOULD encrypt it at rest and release the projection when the service is not running (and on the §17 Xorg build, R-X12 compiles the Wayland capture path out entirely, making this half absent by construction). (Re-prompting per session MUST NOT be added where it would break unattended access. The §17 Xorg box has no capture-consent gate at all — it captures for the authenticated peer by design, the intended unattended behavior — so this requirement is about where the gate is: the PAKE, not an OS dialog.)

8. Paths to EXCISE (block / remove)

The audit found several inherited paths that cannot be made safe on an internet-exposed host and add nothing to a serverless fork. These MUST be compiled out / removed so they cannot be re-enabled.

R-X1MUST Excise the auto-updater (fetch-and-run RCE)
updater.rs:120-285 downloads a binary from a server-supplied URL (the version-check JSON's url field) with no signature or hash check and executes it privileged (--update / MSI). A second identical path exists in the Flutter UI (flutter_ffi.rs:2920-2964). Whoever controls that response — or DNS/TLS for the version host, or a re-pointed fork — gets privileged code execution. MUST remove src/updater.rs (mod at lib.rs:63, desktop-cfg-gated), the --update handler (core_main.rs:224), and the Flutter update-me/download-new-version keys. The fork ships its own releases (§12); if update-checking is ever re-added it MUST verify a pinned SHA-256 of the artifact before execution (R-B2) — never fetch-and-run.
R-X2MUST Excise the native-plugin loader
plugin/plugins.rs:301-389 dlopens any plugin_*.{so,dll,dylib} from a system path and calls its C entrypoints — with the validation left as a literal // to-do (:361); install_plugin_with_url unzips a download with no verification. That is arbitrary native code in-process. It is already double-feature-gated and not in the default build, and is not remotely installable (the PluginRequest message is post-auth and only invokes already-loaded plugins) — but it is dead weight on a hardened fork. MUST never compile plugin_framework, and MUST drop the src/plugin/ tree and the --plugin-install/--plugin-uninstall args so it cannot be re-enabled by accident.
R-X3MUST Excise the remote ConfigureUpdate server-list rewrite
rendezvous_mediator.rs:397-407 lets the rendezvous server rewrite the client's rendezvous-servers list at runtime and restart — re-homing the client to arbitrary infrastructure. A serverless fork's server list MUST be fixed by local config; make the ConfigureUpdate arm a no-op. Note the twin: a malicious rendezvous server can also rewrite rendezvous-servers from inside an unauthenticated TestNatResponse (common.rs:662, the cu field → set_option("rendezvous-servers") + set_serial) — the same re-home class. R-D4 (mediator excised) removes both, but R-A6 MUST grep both so a partial fix cannot leave the twin live.
R-X4MUST Pin the trust anchor; remove the key/server overrides
The public key (trust anchor of the rendezvous-secured channel) and the server set are overridable via the "key" config option, a license/"config string" (which even accepts unsigned JSON — custom_server.rs:29), a renamed executable (or the RUSTDESK_APPNAME env var that spoofs the exe-name license without renaming, windows.rs:2109-2116), and the registry (common.rs:1805, custom_server.rs, core_main.rs:507; on Windows the HKLM license read at windows.rs:1943/:3819). A malicious installer name or shared config string silently points the client at attacker infrastructure with the attacker's key. The fork MUST bake its own values, make get_key() ignore the runtime override, and remove custom_server.rs + the --config path; it MUST NOT accept the unsigned-JSON branch. The same class includes three ungated CLI paths beyond --config: --import-config <path> overwrites the entire config (incl. trust anchor + servers) from an attacker-suppliable file with no is_root gate (core_main.rs:420); --remove <path> is an ungated file-delete gadget (:370); and --assign/--deploy/--set-id presuppose a server account this fork removes and leak tokens/passwords on argv (:539,:644,:476). All MUST be excised and CI-grepped (R-A6).
R-X5MUST Excise LAN discovery; the recent-session re-admit lives behind the PAKE
LAN discovery (lan.rs, UDP 21119) discloses MAC, RustDesk ID, hostname, active username, and platform to any LAN pinger. Per §8's "compiled out / removed" bar (not merely "disabled"), the direct-only build MUST excise the lan.rs UDP listener — absent, not firewalled (cf. R-D4). On Android this is acute: start_lan_listening is hard-forced true (rendezvous_mediator.rs:127-128), so the excision MUST extend to the Android build (R-D7a), not Linux alone. The is_recent_session 30-s auto-re-admit (connection.rs:2221) is not a separate weaker path: it MUST be re-gated behind a confirmed PAKE stream (R-S2) and its window (SESSION_TIMEOUT = Duration::from_secs(30), connection.rs:402) set to 0 on the exposed host — never an unauthenticated re-admit.
R-X6MUST Excise the trust-anchor/password deep-link writes; confirm the connect deep-link
A completeness sweep of the platform/UI layer found the one cross-trust-boundary surface the Rust-core audits could not see: the rustdesk:// URL-scheme handler — a malicious web page, QR code, or co-installed app can fire it, pre-authentication. rustdesk://config/<b64> writes the rendezvous/relay/api server and the public key (trust anchor); rustdesk://password/<pw> sets the permanent password — both default-OFF behind compile-time flags but with no install+root gate (unlike the CLI), and both are the same trust-anchor / credential-injection class as R-X4. The fork MUST remove the config/server/key/password-writing deep-link authorities entirely. The third path, rustdesk://<id>?password=…&/r@<server>, silently auto-initiates an outbound session to an attacker-chosen peer/relay on all platforms; the fork MUST require explicit user confirmation for any deep-link-initiated connection and MUST NOT honor an embedded password/relay/key (incl. the ?key= trust-anchor injected into the outbound session) without it. The strip MUST be enforced in both layers: the Dart parser (flutter/lib/common.dart handleUriLink) and the Rust core, where the embedded ?key= is adopted into other_server (client.rs:1818) and where the raw URI also reaches the core via bind.sendUrlScheme (the desktop_multi_window child-window path) — a Dart-only fix is bypassed, since the trust-anchor adoption lives in the Rust initialize (client.rs:1798-1818). (flutter/lib/common.dart:2377-2473; client.rs:1798-1818.) The Linux delivery transport is a D-Bus session-bus method (src/server/dbus.rs: bus org.rustdesk.rustdesk, method NewConnection(id)) that ignores the caller (no peer check) and forwards the URI verbatim into the same handler — any co-installed same-session app can invoke it, and it claims the bus name with replace_existing=true (a name-hijack that lets a malicious process intercept legitimate links). The fork MUST remove or caller-restrict NewConnection and drop replace_existing (CI-grep org.rustdesk.rustdesk · NewConnection · start_dbus_server). The Windows and macOS delivery transports are the same un-authenticated local-IPC class — Windows relays the URI by a WM_USER+2 window message (core_main.rs:864 → send_message_to_hnwd), macOS by the _url IPC socket (ipc::send_url_scheme) — and neither authenticates the sender; the fork MUST apply the same caller-restriction where the OS permits and route all of them through the explicit-confirmation gate above, which is the load-bearing control across all four transports. On Android the deep-link is the manifest's rustdesk:// VIEW/BROWSABLE intent-filter on an exported MainActivity, and the manifest also ships an exported BootReceiver with a DEBUG_BOOT_COMPLETED broadcast any installed app can fire (AndroidManifest.xml: MainActivity :61, BootReceiver :34); the fork MUST remove the debug broadcast, set those components exported=false where possible, scope the intent-filter, set android:allowBackup="false" (else the UUID-obfuscated config holding the password-equivalent PRS is exfiltratable via adb backup) + a cleartext-deny network-security-config, and minimize the permission set to the dual viewer+controlled minimum (the app is one binary in both roles, R-R2, so it cannot be pared to "viewer-only"). Drop the legacy READ_EXTERNAL_STORAGE/WRITE_EXTERNAL_STORAGE (superseded by MANAGE_EXTERNAL_STORAGE on the pinned targetSdk 33) and requestLegacyExternalStorage. Keep (each maps to a preserved feature) INTERNET, ACCESS_NETWORK_STATE, FOREGROUND_SERVICE (+ the typed FOREGROUND_SERVICE_MEDIA_PROJECTION if targetSdk is bumped to ≥ 34), BIND_ACCESSIBILITY_SERVICE (controlled-side input injection), RECORD_AUDIO, MANAGE_EXTERNAL_STORAGE, POST_NOTIFICATIONS, WAKE_LOCK, REQUEST_IGNORE_BATTERY_OPTIMIZATIONS and RECEIVE_BOOT_COMPLETED (controlled auto-start). SYSTEM_ALERT_WINDOW is the one high-risk permission to drop, and this profile decides it: cut the FloatingWindowService and drop SYSTEM_ALERT_WINDOW — the controlled-status surface is already carried by the mandatory foreground-service notification (MainService.kt), and a TYPE_APPLICATION_OVERLAY window is an overlay/tapjacking primitive a single-purpose box does not need. The inherited floating window is moreover default-on (disable-floating-window is opt-out, == "Y", MainActivity.kt:404) with cosmetic size/transparency/untouchable/SVG sub-options — all removed with it. Cutting the permission MUST sever, not silently disable, the boot-auto-start dependency that currently requires it (BootReceiver.kt; canStartOnBoot). (If a status affordance is ever retained, it MUST be a single non-configurable minimal form — no runtime variability.)
R-X7MUST Excise the one-time-password mechanism; pin to the permanent password
RustDesk carries two credentials: the permanent password and an in-memory, rotating one-time password (TEMPORARY_PASSWORD, hbb_common/src/password_security.rs:6), and the default verification-method accepts either (UseBothPasswords, :49; validate_password tries temporary first, then permanent, connection.rs:2178). A balanced PAKE authenticates from one shared secret, so a second live credential is dead weight at best and a non-PAKE bypass at worst; the one-time password is also attended-only (screen-shown, rotating) and useless for the unattended access this fork targets. The fork MUST excise the temporary-password code (TEMPORARY_PASSWORD, get_auto_password, update_temporary_password, check_update_temporary_password) and hard-pin verification-method = use-permanent-password, making the permanent password the sole credential and sole PAKE PRS (R-P1, R-S9). This also deletes the consecutive-failure temporary-password rotation (connection.rs:2131) — a remotely-triggerable lockout where 10 failed attempts rotate the password out from under a legitimate user, already gated on temporary_enabled() (:2143), so it vanishes with the excision. Enforcement MUST be fail-closed and immutable: verification_method()/approve_mode() today read a runtime-writable option and fail OPEN — an unset/unknown value yields UseBothPasswords/Both, re-enabling OTP and click-to-accept (password_security.rs:49,:84) — and a DEFAULT pin can be shadowed by a written option (the get_or read-precedence at config.rs:2745 returns a set value over the DEFAULT; set_option writes at :1259/:2228). The fork MUST make these return the pinned constant unconditionally (not read the option) and CI-grep that they don't (R-S12).
R-X7aMUST Excise the Android approve-mode / verification-method selector UI — not merely fix it off
R-X7/R-S9 pin verification-method = use-permanent-password and approve-mode = password in the core resolver (with R-A6's AST check that verification_method()/approve_mode() never read Config::get_option). But the Android server-settings UI still presents and writes the multiplicity — selectors for password / click / both and one-time / permanent / both, plus the one-time-password length and the numeric-OTP toggle, via bind.mainSetOption (flutter/lib/mobile/pages/server_page.dart:71-173; writers :148,:166-170) — gated only by the soft isOptionFixed managed-config lock (config.rs:76), the exact "defaulted off, not removed" pattern R-S12 forbids. Once R-X7's pin lands the resolver ignores these writes, so the UI is a second, divergent way to express the controlled-side policy. The fork MUST remove these selectors (leaving only Set permanent password), not disable them — the same toggle-and-surface removal R-X8 applies to the terminal.
R-X8MUST Excise the terminal root-PTY from the direct-only build — not merely default it off
Post-PAKE the Terminal session type spawns a shell as the service user — root on the exposed box (R-D3) — with no second credential on Linux: the OS-login/PAM re-auth (should_use_terminal_os_login_scope, connection.rs:130) is Windows-only by construction (cfg!(target_os = "windows")), and the Linux fill_terminal_user_token (:3509) is a stub that yields a root PTY via spawn_command (terminal_service.rs:1220). 10. The PAKE specification

The handshake is small but unforgiving. Every step below is normative to the byte — each is a place real PAKE deployments have failed — and the construction follows the CPace draft and its published security proof, not intuition.

10.1 Primitive, suite & implementation

Primitive — balanced CPace

CPace, a balanced composable PAKE that authenticates from the shared password alone, with a published formal proof (Abdalla–Haase–Hesse 2021, adaptive UC). Suite: ristretto255 + SHA-512. This is a deliberate, documented deviation: the CFRG draft's recommended suites are CPACE-X25519-SHA512 (first) and the NIST-P curves; ristretto255 is a permitted "can also be implemented" option, chosen here because it is a prime-order group — eliminating by construction the cofactor / small-subgroup / twist pitfalls the draft enumerates for X25519.

Implementation reality — implement the draft; the crate is non-conformant

Settled empirically (and independently re-verified). A from-scratch draft-21 CPACE-RISTR255-SHA512 reference passes all of the CFRG draft's published ristretto255 test vectors (from github.com/cfrg/draft-irtf-cfrg-cpace): 18/18 protocol checks + 13/13 encoding checks, both abort cases included. Those published values are the derived CPace generator g = a6fc82c3… (not the ristretto255 base point) and the symmetric-mode ISK ISK_SY = 980dcc5a… — the value our order-independent o_cat mode reproduces (R-P6). Computing jedisct1's pake-cpace on the same inputs diverges from both: the crate is on-the-wire non-conformant — it implements the 2019 draft-haase-cpace-01, unaudited, last released 2023. Its security-relevant deltas (no sid in the ISK, initiator-chosen sid, no key confirmation) live inside the hashed transcript and cannot be patched at the application layer — fixing them means forking lib.rs, i.e. rewriting draft-21 minus the vectors. There is no viable "use the crate" path.

Therefore the fork MUST implement draft-21 CPACE-RISTR255-SHA512 in-tree over curve25519-dalek — ~250 lines of encoding glue, zero novel curve math (dalek supplies the audited, constant-time from_uniform_bytes map, scalar×point, decompress, is_identity; you write lv_cat/o_cat, the 64-byte generator string, the ISK transcript, and the R-P3 MACs). CI MUST gate on the draft's published test vectors — they catch the real failure modes (encoding off-by-ones, the zero-pad length prefix, 32-vs-64 hash length, transcript ordering, the aborts). Pin curve25519-dalek ≥ 4.1.3 (RUSTSEC-2024-0344). This adds no new external crate: every primitive the handshake needs is already in the 1.4.7 lock — curve25519-dalek 4.1.3, sha2 0.10.8, hkdf 0.12.4, hmac 0.12.1, subtle 2.6.1, zeroize 1.8.1, rand 0.8.5 (CSPRNG; getrandom 0.2.15 also present), and the inherited session layer is sodiumoxide 0.2.7 (libsodium: secretbox + the Ed25519 device key to be removed per R-P5 — there is no ed25519-dalek) — so the PAKE is ~250 lines of our own code over crates already compiled in (exact lock versions in §3.2). The §11 review then scopes to those ~250 lines plus the two things the protocol vectors don't directly exercise: constant-time scalar sampling, and the R-P3 confirmation-MAC composition — though its inputs (sid, ISK, lv_cat(Y,AD)) are all vector-covered and it is the draft's own recommended construction over an RFC-4231-vectored primitive (HMAC-SHA-512), so it earns a self-consistency KAT rather than review alone.

No registry, no stored keys — the most sovereign anchor there is

Because a balanced PAKE authenticates from the shared password, this fork uses no long-term public keys, no signatures, no certificates, and no key registry of any kind. TLS anchors trust in CA registries; SSH anchors it in known_hosts (trust-on-first-use — a local registry). The PAKE anchors it in the password you already share — nothing to store, distribute, pin, or look up. One credential, one primitive, one path; no second PAKE to select, negotiate, or fall back to. (The per-connection ephemeral group elements on the wire are key-exchange values, not identities, and are kept nowhere.)

10.2 The handshake

R-P1MUST Inputs & channel binding
PRS = the RFC 8265 OpaqueString-normalized permanent password (NFC, no case-folding; §10.4) used directly — the sole credential, since the one-time-password path is excised (R-X7); no memory-hard pre-hash (R-S6 explains why one would add nothing here). Because CPace is balanced, the controlled side must hold this PRS — a password-equivalent value — to run the handshake; protect it as R-S9 requires. The channel identifier CI MUST carry the protocol/version label and any per-connection channel binding, and MUST be identical and pre-agreed on both sides (CI is never sent on the wire). The session id sid MUST be a unique per-connection value agreed by both parties — e.g. a fresh 16-byte random nonce contributed by each side, concatenated initiator-then-responder (a 32-byte sid). That contribution exchange rides the still-unkeyed wire pre-PAKE, so its integrity rests solely on the R-P3 confirmation abort: sid MUST enter the ISK transcript (draft-21 symmetric — the ISK_SY vector), not merely the generator, so a tampered nonce yields a divergent ISK/mac_key and aborts. CI MUST be a fixed both-sides-identical string (fork tag · suite · version · port binding) — the per-side role instead lives in AD (ADa=b"viewer" / ADb=b"server"), since the draft requires both sides to hold the same CI; no per-connection value that would need a tamperable exchange. The port folded into CI MUST be the pinned compile-time direct-access port (21118, R-F4) — the same constant compiled into both roles, never a runtime option and never a value read from the live socket (NAT/port-forwarding would otherwise desync CI between peers and silently abort the handshake) — and its byte-encoding MUST be pinned (§10.4); since no published draft vector exercises a fork-specific CI, the build MUST add a project KAT over CI and the derived generator (R-A10). No identity keys (R-P5).
R-P2MUST Exchange & key derivation
Each side sends its CPace message (ephemeral group element); both derive the Intermediate Session Key ISK. The raw DH output K MUST NOT be exposed, logged, or used directly (a leaked K re-enables offline attack). From ISK, derive two directional secretbox keys (controlled→viewer and viewer→controlled) by HKDF-SHA-512 with distinct info labels; derive the confirmation key separately by the draft's own construction (mac_key = H("CPaceMac" ‖ sid ‖ ISK), R-P3). All three are domain-separated by construction — MUST NOT slice one hash for several keys, nor reuse one key across both directions (R-P10).
R-P3MUST Explicit mutual key confirmation (the proven construction, not a transcript MAC)
After ISK: mac_key = H("CPaceMac" ‖ sid ‖ ISK); each party sends MAC(mac_key, lv_cat(its-own element, its-AD)); each MUST verify the peer's tag in constant time and abort before authorizing. The MAC MUST be HMAC-SHA-512 (the draft offers HMAC or CMAC; HMAC-SHA-512 matches the suite hash and carries its own RFC 4231 test vectors). This is the CPace draft's own Key confirmation construction, verbatim — specifically the variant it defines for the parallel setting without message ordering, which is exactly our symmetric mode (R-P6); it is not an ad-hoc transcript MAC. The reply MUST NOT be a constant — it is a function of ISK by construction (the fix for the prior fork's b"direct-ok"). Do not MAC "the whole transcript and both identities" — there are no identities, and that diverges from the construction the proof covers. Explicit confirmation also upgrades CPace's forward secrecy from weak to perfect. In the symmetric mode these tags are sender-bound only once R-P5's role/AD discipline is applied (otherwise a tag is reflectable in a loopback) — R-P3 MUST NOT be implemented without R-P5.
R-P4MUST One handshake at the choke point, before the type is parsed
Run once where the single stream is created (server.rs:190), keying all six transports and all five types before any application message.
R-P5MUST No PKI/registry; defend the loopback
No identity keys, signatures, certificates, host-key pinning, or registry; the existing SignedId/PublicKey/Ed25519 session-key bootstrap in the connection handshake is removed (§8) — but that is the device-identity keypair only: the rendezvous trust anchor (RS_PUB_KEY, R-X4) stays, and in any server-assisted build the register_pk/key_confirmed registration-liveness gate (rendezvous_mediator.rs:804) MUST be preserved or the box silently goes unreachable; the direct-only deployment (R-D4) sidesteps both. Consequence of having no identities: CPace will key with anyone presenting the same PRS+sid — including the host looping back to itself. The fork MUST add a loopback/relay defense: assign fixed initiator vs responder roles per (PRS,sid) and forbid one host playing both, or enforce unique ADa/ADb and reject a received AD that matches a locally-active session. Resolved (§10.4): pin fixed roles — the controlled side is always responder, the viewer always initiator — and carry the role as the per-side associated data ADa = b"viewer" (initiator) / ADb = b"server" (responder), with only the shared suite/version/port in CI (the draft requires both sides to hold the same CI, so the role cannot live there). Because AD is per-side, sent, and folded into both the transcript and the R-P3 tag, each tag is sender-bound and a host cannot run both ends for one (PRS,sid); a loopback/reflection fails by role/AD mismatch. The AD check MUST be explicit: the responder MUST verify the received ADa == b"viewer" and the initiator MUST verify ADb == b"server", aborting on mismatch — folding the received AD into the transcript is necessary but not sufficient; the literal equality check is what makes each confirmation tag sender-bound.

10.3 Construction details (each is a place real PAKEs have failed)

R-P6MUST Symmetric-mode ordering
Both ends are peers, so message order is undefined ⇒ the symmetric setting: combine with the draft's lexicographic o_cat(lv_cat(Ya,ADa), lv_cat(Yb,ADb)), OR impose fixed initiator/responder roles (which also satisfies R-P5). Never a fixed Ya‖Yb order without roles. Note: the fork pins fixed roles and uses the symmetric o_cat transcript — a deliberate superset of the draft's initiator-responder transcript_ir (safe: o_cat binds both elements + ADs). The ISK_SY vector is therefore the gating KAT; an implementer MUST NOT substitute transcript_ir (it fails the vector and breaks interoperability with a conformant fork peer).
R-P7MUST Abort on invalid / identity point
If the shared point is the group identity, or a received element fails ristretto255 decode(), abort — never continue with a degenerate key. The trap: the all-zeros encoding decodes successfully to the identity, so decode-success alone is insufficient — the K.is_identity() test after the multiply is what catches a malicious identity input (§10.4).
R-P8MUST sid uniqueness; no secret reuse
sid unique per connection (e.g. random_a ‖ random_b); ephemeral secret scalars and sid MUST NOT be reused.
R-P9MUST Pinned domain separation
Pin the exact DSI strings for the chosen draft version and a distinct HKDF label per key; a DSI mismatch MUST make keys not agree, never silently cross-confuse.
R-P10MUST secretbox nonce discipline
The 24-byte XSalsa20-Poly1305 nonce MUST be unique per key and never reused — including across directions. The inherited layer derives the nonce from a per-direction sequence counter alone (tcp.rs:203 get_nonce = LE64(seq), no direction tag) and installs one key with both counters at 0 (:199 set_key; :297 Encrypt(key,0,0)) — so a single shared key makes each side's frame n seal a different plaintext under the same (key, nonce): catastrophic XSalsa20-Poly1305 reuse. Per-direction keys (R-P2) close it — the invariant is key-uniqueness, not nonce-uniqueness, so identical per-direction counters from 0 are safe and one MUST NOT "optimize" back to a single key; also specify a derived-counter generation and an abort/rekey policy before counter wraparound. A rekey MUST install fresh per-direction keys — never reset the counters under the existing keys, which would re-create the very (key, nonce) reuse this requirement closes.
R-P11MUST No downgrade, no suite negotiation
Hard-pin one suite, one draft version, one DSI set; carry the version label in CI so any mismatch yields non-agreeing keys. No negotiation handshake exists to downgrade. The pinned draft is draft-irtf-cfrg-cpace-21 — the current CFRG revision as of 2026-06 (latest revision 2026-04-22, in IESG review; not yet an RFC), and its published ristretto255 vectors are exactly the ones §10.4 reproduces — which MUST be taken from the draft-irtf-cfrg-cpace-21 tag (or vendored), not the live CFRG master, whose later example-CI change produces different vector values (R-A10). If CPace later reaches RFC, adopt it only as a deliberate, re-vectored change — never an automatic version bump.
R-P12MUST K-secrecy & constant-time
K MUST NOT be exposed or logged (only ISK leaves the PAKE layer). Generator computation, scalar sampling, scalar multiplication, and every tag comparison MUST be constant-time (curve25519-dalek provides this for the curve ops — assert it). Scalar sampling MUST use Scalar::from_bytes_mod_order_wide / Scalar::random, never the deprecated Edwards-only Scalar::from_bits (§10.4); and the tag comparison is the one CT op not inside dalek — use a constant-time comparator (subtle / Mac::verify), never ==. Constant-timeness MUST be machine-checked — a dudect-style statistical timing test in CI over the sampling and the tag compare, plus the type-level guarantees of subtle/curve25519-dalek — so it is verified, not reviewed by eye (this is what lets §11 hold without a cryptographer in the loop).
R-P13MUST Quantum-annoying, not PQ — record it
CPace is "quantum-annoying" (a quantum attacker solves one CDH per password guess), not post-quantum. Role/port already fold into CI (R-P5/§10.4), raising per-guess work; the fork MUST document that long-term confidentiality rests on password entropy — a recorded transcript is offline-recoverable by a future quantum attacker against a weak password. A PQ-hybrid is out of scope for a single-user box unless that threat is.

Estimated change size: ~200–400 lines (handshake + confirmation + the §10.3 invariants + the §8 excisions + removing the SignedId/Ed25519 exchange). The code diff is small; the work around it is not — every line in §10.3 is a place a real PAKE deployment has failed, and the verification (§11), independent audit (R-V3), and per-platform build pipeline (§12) dwarf the diff. That asymmetry is the point, not a reassurance.

R-P14MUST On-wire choreography: where the handshake runs and the messages it replaces
The CPace exchange runs inside create_tcp_connection (server.rs:190) on the unkeyed Stream, replacing the current SignedId ↔ PublicKey key bootstrap verbatim (controlled sends SignedId at server.rs:206; peer replies PublicKey{symmetric_value,asymmetric_value}; set_key(Encrypt::decode(…)) at :227). That block (server.rs:204-248), its client counterpart, the box_ seal/open path (tcp.rs:326-335), and the SignedId/PublicKey protobuf messages are removed (R-P5, §8). The new flow is pinned:
  • Message — a dedicated top-level type, not a Message-union variant. Add Cpace as its own top-level protobuf message in libs/hbb_common/protos/message.proto (a oneof over the steps below), framed through the existing bytes_codec but parsed and serialized on its own on the unkeyed stream — parse_from_bytes::<Cpace>, never Message::parse_from_bytes. Routing it through the application Message union (the obvious shortcut) would make the pre-auth decoder reach the entire Message schema — every VideoFrame/file/clipboard field an attacker could populate — whereas the dedicated type keeps the reachable pre-key decoder surface to the handshake's ~4 fixed fields (R-S7). Both roles change: the responder (controlled side) parses only Cpace pre-key, and the initiator (viewer) sends/parses only Cpace pre-key — which also protects a connecting viewer from a malicious server's malformed pre-key frames. Both sides switch to the normal Message framing only after keying. No other message type is accepted, or even decoded, on the unkeyed stream.
  • Fixed roles & sequence (viewer = initiator, controlled = responder). The generator g needs the full sid = sid_a‖sid_b and CI, so the sid contributions precede the group elements: ① initiator → responder { sid_a(16B), ADa=b"viewer" }; ② responder → initiator { sid_b(16B), ADb=b"server", Yb(32B) }; ③ initiator → responder { Ya(32B), Ta }; ④ responder → initiator { Tb }. Ta/Tb are the R-P3 HMAC-SHA-512 tags (64 B). Two round-trips.
  • Gate. Each side verifies the peer's tag in constant time, derives the two directional keys (R-P2), and calls set_key only on success; no set_key, no Connection::start, no application message, no authorized before both tags verify (R-A1/R-A2). Any decode/length/timeout/tag failure aborts fail-closed.
  • Framing. set_max_packet_length is capped small before the first byte (R-S7); each CPace message is ≤ ~120 B. With KCP dropped (R-S7/R-D4), this dedicated, fixed-shape Cpace message (parsed on its own, never via the Message union — R-P14) is the only attacker-reachable parser before keying.
This choreography is in scope for the §11 adversarial review (R-V1).
R-P14aMUST Strict step-ordering and no-duplicate handshake state machine
§10.2 gives the message sequence; this pins the state machine that enforces it. The handshake MUST be driven by an explicit four-state machine per role (responder: WAIT_1 → INIT2 → WAIT_3 → SEND_4 → KEYED; initiator: INIT → WAIT_2 → SEND_3 → WAIT_4 → KEYED). In each receiving state the implementation MUST accept exactly one Cpace oneof variant and MUST abort fail-closed on any other — a future-step variant, an own-side variant, or a repeat of an already-consumed step (no peer may send two step-① frames). A step arriving before its prerequisites (e.g. step-③ before step-① at the responder, which then holds no sid_b/g/yb) MUST be a wrong-variant abort, never buffered; no state may silently ignore an unexpected frame and keep waiting (that re-opens the half-open / duplicate surface). Each step is consumed exactly once, and set_key is reachable only from the tag-verify success edge (R-P14, R-A1).
R-P14bMUST Bounded reads — the timeout is the half-open resource bound
set_max_packet_length(≤ 4 KiB) MUST be set before the first read (R-S7), and every inbound handshake step MUST be read under a finite timeout (per-step, or one whole-handshake deadline). Verified: the framing codec returns Ok(None) on a partial/dribbled frame (bytes_codec.rs:50,:78) — it never completes and never errors — so the timeout is the sole bound on a peer that opens a connection and stalls (the inherited code wrapped only its single next(): server.rs:221 / client.rs:792, CONNECT_TIMEOUT/READ_TIMEOUT = 18 s; four steps need four bounded waits, or one deadline). A timeout, a peer EOF (Stream::next() → None, tcp.rs:178), an oversize frame (bytes_codec.rs:64 "Too big packet"), or a parse_from_bytes::<Cpace> decode error MUST abort fail-closed. The timeout (stall bound), the ≤ 4 KiB cap (size bound, R-S7), and the per-source limiter (CPU-flood shedding, R-S10) together bound half-open exhaustion to ≤ 4 KiB + one task per connection until the deadline; no per-connection state beyond the in-flight handshake is created.
R-P14cMUST Only key-confirmation failure feeds the limiter; replay needs no state; nothing on the wire to downgrade
Of all pre-key aborts, only an R-P3 key-confirmation tag mismatch MUST increment the per-source limiter (R-S10) — it is the sole event that is an online password guess; decode / length / out-of-order / duplicate / decompress / identity / AD-mismatch / timeout / EOF aborts MUST NOT increment it, or a malformed-frame flood would trip the owner's own block (R-S10's cardinal principle; the limiter is checked before the scalar-mult). Whole-handshake replay is harmless and MUST NOT be defended by sid-tracking state: each side contributes a fresh 16-byte sid nonce and a fresh ephemeral scalar per connection (R-P8), so a replayed sid_a/Ya cannot yield a tag valid under the responder's fresh sid_b/Yb (the mac_key/ISK diverge) and aborts at R-P3. sid freshness is enforced unilaterally and locally by CSPRNG sampling; a "seen-sid" cache is forbidden — unnecessary, and itself a memory-DoS / stateful-config surface (R-S12). The Cpace oneof carries no version / suite / capability field — those live only in the never-sent CI (R-P1, R-P11) — so there is no on-wire field to downgrade, and an unknown/unset oneof case is a decode abort, not a fallback.

Three traps a vector-blind implementer hits

(1) Scalar sampling: use Scalar::from_bytes_mod_order_wide (what Scalar::random calls) — a 64-byte wide reduction with no bit-masking. Honest caveat — stated every time

The in-tree CPace implementation (§10.1) is not yet independently security-audited. R-V1MUST Adversarial multi-lens review
The handshake design + implementation undergo structured review attacking from independent angles (first-contact MITM, replay, downgrade, timing, nonce/transcript reuse, the five-type coverage, the set_raw path, curve25519-dalek usage). Findings resolved or risk-accepted in writing.
R-V2MUST Spec conformance & implementation audit
Checked against the chosen CPace draft and its published test vectors (group, encoding, generator derivation, ISK derivation, KDF labels — the published vectors stop at ISK; no vector covers the R-P3 key-confirmation MAC, so that step is checked by a self-consistency + negative KAT instead, R-A10). The in-tree implementation and its curve25519-dalek dependency are read for correct usage and rev-pinned; the implementation's un-audited status is disclosed in the release notes.
R-V3MUST Assurance basis: full byte-level spec + published vectors + adversarial tests

Because this code guards an internet-exposed, root-capable service and replaces the two most serious inherited findings (Appendix C #1–#2), an independent expert audit of the §10.4 construction, the constant-time paths, and the KDF parameters MUST be obtained before exposed operation and published — the ~250-line scope makes this tractable. The vectors + KATs + adversarial tests above remain the day-to-day CI assurance basis (and what makes the audit checkable); until the audit is obtained, the §11 "not independently audited" disclosure stands and the box MUST be treated as pre-audit. What remains forbidden is the inverse: shipping a "clean bill of health" the tests do not actually back — every security claim here is reproducible against the vectors and the file:line citations.

12. Build, release & version pinning

R-B1MUST Build & publish the full client matrix
From the one forked codebase, the flutter-build.yml workflow MUST produce Android aarch64 (.apk), Windows x86_64 (.exe + .msi), and Debian x86_64 (.deb) — each usable as viewer and, via --server, controlled side. It MUST run to completion off GitHub-hosted runners (locally or a self-hosted runner; R-B2); macOS/iOS are dropped (R-R2).
R-B2MUST No code-signing; the build cannot live on GitHub-hosted CI
Upstream's release model signs artifacts with secrets injected into GitHub Actions — unusable here on two counts. (1) GitHub-hosted runners are too unreliable to be the build path: the heavy RustDesk matrix (vcpkg built from source, multi-GB toolchains, long Windows/Android jobs) fails on the order of one run in five, so you cannot depend on getting a green build out of it — flutter-build.yml MUST be runnable to completion off GitHub-hosted runners (locally or a self-hosted runner; R-B1). (2) Artifacts ship unsigned: code-signing is a CA/registry dependency (the model R-P5 already rejects), it adds nothing once the binary's hash is verified directly, and with the build off GitHub there are no CI secrets to hold a signing key anyway. Exception — Android: the OS refuses to install an APK with no signature, so it uses a self-generated local key (apksigner/keytool — no CA, no Play Store), a forced local keypair, not CA code-signing — Android refuses to install an unsigned APK (the v2 scheme is mandatory since Android 11), so this signature is not optional. Pin it: one stable RSA-4096 keystore (SHA256withRSA; validity ≥ 10000 days; fixed alias), generated once and reused for every release — Android ties app identity to the signing key, so a stable key gives clean in-place upgrades whereas a changed/absent key forces uninstall+reinstall (wiping app data). The keystore is a secret: kept out of the repo and the build image, fed to build-android.sh only at sign time (R-B11) — mounted read-only, never baked into an image layer, with its password supplied via file/agent, not env or argv (both leak via /proc and container inspect). Because one stable key signs every release for clean in-place upgrades, its theft lets an attacker ship malicious upgrades and rotation forces a data-wiping reinstall, so the fork MUST document a key-compromise / break-glass procedure and hold the keystore offline (ideally HSM-backed) — the residual is real and currently unaddressed. The signature is Android's install gate, not the trust anchor — the pinned SHA-256 still is. Integrity from build to install is a pinned SHA-256 per artifact, verified on the target over the operator's existing trusted out-of-band channel — the same pinned-SHA check used for any third-party .deb — over the reproducible build from the committed lockfiles (R-R1). Tamper-evidence without a signature. One determinism prerequisite the recorded-SHA bar depends on: upstream bakes a wall-clock, local-timezone build timestamp into every binary (hbb_common::gen_version emits BUILD_DATE via chrono::Local::now(), hbb_common/src/lib.rs:241, also writing into the source tree at ./src/version.rs:231), so two builds of identical source on the identical pinned image otherwise yield different bytes and a different SHA-256 — turning the operator's rebuild-and-compare into a guaranteed mismatch and making a malicious build host indistinguishable from a benign rebuild. The fork MUST neutralize it — honor SOURCE_DATE_EPOCH (patch gen_version to read it, or hard-pin BUILD_DATE to the release commit's date) — and CI MUST assert it by a double build: build the same source twice and require byte-identical SHA-256. Without that assertion the recorded-SHA bar is unfalsifiable.
R-B3MUST Publish AGPLv3 source per binary
Each release links the exact source revision (both repos + submodule commit) it was built from.
R-B4MUST Smoke-test every platform; assume nothing builds until watched
Per platform, connect + authenticate + drive each session type must pass before tagging. (Upstream's fork pipeline emits nothing until upload-artifact is wired and the build is watched to green — real work, the more so on the flaky runners R-B2 keeps the build off of.) Plus a socket-surface smoke-test on the running --service: ss -lntup + strace -f -e trace=bind,listen,connect MUST show exactly one v4 TCP listener on the direct port, zero UDP listeners, and no unsolicited outbound connect — the build-time empirical confirmation that the binary binds what Appendix B claims (R-D3/R-D5/R-D6), with R-A4 the permanent runtime backstop.
R-B6MUST Drop the Sciter UI
The repo carries two UI stacks (Flutter + legacy Sciter). Sciter is why Rust is pinned to 1.75 (it breaks on the 1.78 i128-layout change — flutter-build.yml:22), why a 32-bit i686-pc-windows-msvc job exists, and the source of a large HTML/JS (res/inline-sciter.py, src/ui/*.tis) surface. Building Flutter-only drops the sciter-rs fork, the i686 job, and the .tis tree — and unblocks a modern Rust toolchain, removing a whole class of build/maintenance and attack surface. Decision: the fork nonetheless keeps Rust 1.75 (edition 2021) for now — the deps are pinned, so a bump buys no security and would force re-validating the entire lock; moving to a newer stable is optional later work, gated on a green --locked build + audit (R-R3).

(Version pinning: R-B5a/R-B5b in §3.1 — build from the committed Cargo.lock with the pinned toolchain; track/mirror the 34 rustdesk-org git deps.)

R-B7MUST Reproduce upstream 1.4.7's official build first; the fork is an auditable delta on it

All three thus run on one Linux x86_64 host with no GitHub-hosted runner in the path (R-B2), each as a disposable instance of a pinned, immutable template (Docker image or golden VM image) provisioned to the pinned toolchain (R-B5a) and nothing more, and torn down when the build completes (§12.1).
R-B9MUST One build script per target — environment-validated, fail-loud, no fallbacks, one mode
The build is driven by checked-in scripts, one per target (build-debian.sh, build-android.sh, build-windows.ps1), held to the same engineering bar as the rest of this spec — the "one option, the good option" discipline applied to the build:
  • Validate the environment, then abort. Assert the exact pinned versions — not mere presence — of Rust / Flutter / NDK / the vcpkg baseline / LLVM / JDK, plus the OS + arch, the required system libraries, and the submodule + lockfile state, before compiling a byte.
  • Fail loud; no fallbacks. set -euo pipefail (PowerShell $ErrorActionPreference = 'Stop'); every check dies with a precise message; no "install latest if missing", "retry another way", or silent default — a single mode of operation, the correct one.
  • Pin from one manifest. Every version comes from a single committed pins file; no command resolves "latest" or a moving channel (R-B5a, R-R1).
  • Self-verifying + idempotent. Re-running is safe; on success the script verifies the artifact (it exists, is the right type, and matches its recorded SHA-256, R-B2) and prints that — it never reports success it has not proven (the full per-script behavioral contract is §12.3).
R-B10MUST One networked step: online-fetch.sh → a git-ignored, SHA-256-verified ./online
The repository is build-oriented and offline-by-construction. A single checked-in scripts/online-fetch.sh is the only script permitted to touch the network; it materializes every resource the repo does not embed — cargo vendor of the committed lockfile (incl. the 44 git deps), the pub/Gradle caches, vcpkg @120deac3 and its asset cache, the NDK r28c zip, the Flutter SDK, LLVM 15.0.6, the Android SDK (cmdline-tools / platform-34 / build-tools-34.0.0), the pinned base-image digests, and the Windows installers/ISOs — into ./online/, which is git-ignored. Every artifact is checked against its pinned SHA-256 in the single scripts/pins.env manifest (the machine-readable §3.2); any mismatch MUST abort fail-closed. This reconciles R-R1's "pinning ≠ vendoring" with the offline build: the bulky pinned world is cached, not committed — re-creatable from pins.env and re-verifiable, never trusted from the network at build time. The build scripts (R-B8/B9, §12.1–§12.2) then run with the network namespace removed (--network=none) and MUST refuse to run — never silently fetch — if ./online is incomplete or any SHA fails. The isolation MUST be of the compile container itself, not merely "no published ports": its own network namespace removed, no Docker socket bind-mounted in, and the warm caches mounted read-only — with a canary build.rs in CI that attempts an outbound connect and MUST fail the compile stage, so "no fetch at compile time" is proven, not trusted. Run order: host-provision.sh (once) → online-fetch.sh (once, or on a pins.env change) → build-* (per build, offline) → cleanup.sh.
R-B11MUST Additive host provisioning; reversible-only cleanup that never touches what pre-existed
scripts/host-provision.sh installs only the host-level runtimes the ephemeral envs need — the container engine and, for the Windows VM, qemu-kvm + libvirt + swtpm + OVMF — plus the few tools the scripts call. The pinned toolchains never land on the host; they live in the ephemeral image/VM (R-B8). Provisioning is additive and idempotent: for each package it checks whether the host already has it and installs only what is absent, recording exactly what it added to ./online/.provisioned. scripts/cleanup.sh is reversible-only: by default it removes just the artifacts the harness created — the ephemeral containers, qcow2 overlays and transient libvirt domains, and the built images — which is always safe because we made them. Reversing the host install is a separate, explicit invocation that removes only the packages recorded in ./online/.provisioned as installed-by-us; it MUST NOT remove or downgrade any package that pre-existed on the build machine, and MUST refuse (fail-closed) if that manifest is absent rather than guess. No --force, no heuristics — the cleanup surface is exactly the inverse of what the scripts created. (The .provisioned manifest is build state, not cache: it MUST live outside the disposable ./online so a cache wipe cannot strand the reversal.)
R-B12MUST Pin provenance, not just a hash — close the bootstrap TOFU and the codegen/distfile gaps
R-B10 verifies every ./online artifact against a SHA-256 in pins.env, but a hash is only as trustworthy as its origin. The fork MUST establish each first pin by an audited, dual-sourced bootstrap — cross-checking the publisher's own published hash/signature (Rustup, Flutter, LLVM, the NDK, and the Windows toolchain all publish them) against a second path — and record per-pin provenance, so a compromised mirror/CDN at first-fetch cannot silently poison the anchor (the distinction is pinned vs pinned-and-provenance-verified). Three inputs R-B10's enumeration does not currently reach MUST come under the same regime: (a) the vcpkg distfiles — the aom and libyuv overlay ports fetch by bare git REF with no SHA512 (res/vcpkg/aom/portfile.cmake:25, libyuv/portfile.cmake:6, from googlesource.com) while the other four overlays pin SHA512 — add it to those two portfiles or capture their tarballs into ./online SHA-pinned (they are C codec parsers on the media path); (b) the flutter_rust_bridge_codegen binary (R-B7); and (c) the Windows toolchain installers, which are partly evergreen (the VS Build Tools bootstrapper and the Win11 ISO are not stably SHA-addressable) — for these the fork MUST pin the captured offline layout by SHA-256 and document which inputs are publisher-hash-verified vs evergreen-best-effort. Finally, the 44 Cargo + 8 pub git deps and the hbb_common submodule are anchored only by git's SHA-1 commit hash (no content checksum, unlike the SHA-256-pinned crates.io/pub deps) — a residual bounded by git's collision-detection and the R-B5b diff-audit that MUST be documented as accepted; and the three ref: HEAD Dart fork pins (R-B5b) MUST be rewritten to their resolved commit so a re-resolve cannot float.

12.1 The containerized build flow — Debian x86_64 and Android aarch64, in turn

This is the concrete realization of R-B7–R-B9 for the two artifacts that build on the Linux x86_64 host in Docker — the Debian x86_64 .deb and the Android aarch64 .apk, built in turn; the Windows .exe/.msi follows the same discipline in a KVM virtual machine on the same box (§12.2). Both Docker builds run inside a digest-pinned image — the image is the versioned build environment R-B8 demands ("provisioned to the pinned toolchain and nothing more"). This does not contradict §17's "no Docker": that rule governs the controlled host's runtime, never the build. Every toolchain version, vcpkg triplet, feature flag and build command below is taken verbatim from upstream's 1.4.7 build (R-B7, R-B5a) — the fork makes no independent version choices; its only deltas are to pin what upstream uses, run off GitHub on one box, drop code-signing (R-B2), build offline (R-B5b), and bind loopback-only (R-D3).

Build environments are ephemeral instances of an immutable, pinned template

Every target follows the same disposable-instance discipline ("cattle, not pets"). The persistent, version-controlled artifact is the template — the Docker image pinned by digest (Debian/Android) or the golden Windows VM image (§12.2). Each build spins up a fresh, throwaway instance of that template (a docker run --rm container; a transient libvirt domain over a copy-on-write qcow2 overlay), runs the validated flow, copies out the artifact and its SHA-256, and is then stopped and deleted — no instance survives the task, and no state carries between builds. That is what makes R-B9's "one mode, no drift" structural rather than aspirational: every build starts from the byte-identical pinned baseline, so the recorded SHA-256 (R-B2) is reproducible, and nothing mutable accumulates on the host.

Why Docker is the faithful path, not a deviation (R-B7)

Upstream already builds the .deb inside a container: flutter-build.yml's build-rustdesk-linux runs every step through rustdesk-org/run-on-arch-action in an ubuntu18.04 container — a deliberately old glibc baseline that keeps the .deb portable. The Android job cross-compiles natively on ubuntu-24.04. So a pinned ubuntu:18.04 image for the .deb and a pinned ubuntu:24.04 image for the .apk reproduce upstream's own environments off GitHub-hosted runners — the "exactly two deltas (no signing, no GitHub runner) and no others" bar of R-B7.

The shared two-stage shape. Each script MUST split a network-enabled fetch stage from a network-isolated compile stage, so "no build.rs / Gradle / vcpkg fetch at compile time" (R-B5b) is proven structurally, not trusted:

  1. Validate (R-B9) — assert the exact pinned versions, the base-image digest, and the submodule + lockfile state; abort on any mismatch before a byte compiles.
  2. Fetch (network on) — cargo vendor the lockfile (incl. the 34 rustdesk-org git deps, R-B5b), build/restore the vcpkg deps from the pinned baseline, warm the Flutter/pub/Gradle caches.
  3. Compile (--network=none) — build --locked --offline; with the network namespace gone, any forgotten fetch fails loudly here instead of silently succeeding.
  4. Package + self-verify (R-B2, R-B9) — produce the artifact, record its SHA-256, re-check it; never claim success unproven.

No new exposure — loopback only. No build stage opens an externally-bound (0.0.0.0) listener: dockerd is reached over its unix socket with no published ports, the network-on fetch stage is outbound only, and the Windows VM (§12.2) is driven over a loopback-bound SSH forward (127.0.0.1). The build adds nothing to the host's audited ports (R-D3) and is safe to run even where only loopback binding is permitted.

Docker supplies the environmental determinism that keeps the SHA-256 stable across rebuilds on this one host; the committed lockfiles plus --locked --offline under --network=none supply the rest.

12.2 Windows x86_64 — the ephemeral KVM Windows 11 VM

Windows is the one target that cannot be cross-built from Linux: flutter build windows runs only on a Windows host (the desktop embedder needs Visual Studio's MSVC "Desktop development with C++"), the vcpkg x64-windows-static codecs need cl.exe, and the installer needs WiX — all Windows-only. So the .exe/.msi build in a Windows 11 guest, realized here as an ephemeral KVM virtual machine on the same bare-metal Linux x86_64 box (any x86_64 host with hardware-virtualization / KVM support) — so all three targets live on one machine with no second physical host and no GitHub-hosted runner. (Where the build shares the §17 host, all build tooling binds loopback only — no 0.0.0.0 listener beyond R-D3's audited ports — and runs only during a build window, never during exposed operation.)

The template — built once. Provision a Windows 11 guest with the pinned toolchain and nothing more (R-B5a): VS 2022 Build Tools + "Desktop development with C++", Windows 11 SDK, LLVM 15.0.6, Flutter 3.24.5, Rust 1.75 (x86_64-pc-windows-msvc), vcpkg @120deac3 (x64-windows-static), WiX, and OpenSSH Server for headless control — then freeze it read-only as the golden backing image. Windows 11's requirements are routine on KVM today: TPM 2.0 via swtpm (in the default Ubuntu repos), Secure Boot via OVMF, virtio storage/net. This golden image is the Windows analog of a digest-pinned Docker image — version-stamped, and the reproducibility anchor. Three integrity steps are mandatory: the golden image MUST be verified by the SHA-256 of its qcow2 before each build (a version-stamp string is forgeable; a content hash is not); the VM's SSH host key MUST be pinned in known_hosts (no first-connect TOFU, even on loopback — the spec rejects TOFU everywhere else); and the Windows toolchain installers that seed the image are partly evergreen, so the fork MUST apply R-B12's captured-layout pinning rather than assume a stable upstream URL. swtpm/OVMF here only satisfy Win11's installer — they are not part of the integrity story (no guest attestation is claimed).

# TEMPLATE (built once): golden Win11 image, pinned toolchain, frozen read-only
golden=win11-rustdesk-golden.qcow2           # the pinned "image" == read-only backing file

# PER BUILD: a disposable instance — the exact analog of `docker run --rm`
qemu-img create -f qcow2 -b $golden -F qcow2 build.qcow2      # copy-on-write overlay, throwaway
virsh create build-transient.xml                             # TRANSIENT domain; QEMU user-mode (SLIRP) net — only listener is a loopback SSH forward (hostfwd tcp:127.0.0.1:2222-:22); build-vm = that alias
ssh build-vm "powershell -File C:\rustdesk_fork\build-windows.ps1"   # validate -> vcpkg -> cargo -> flutter -> WiX
scp build-vm:C:/out/rustdesk-${VERSION}-x86_64.exe ./        # pull .exe + .msi + SHA256SUMS
scp build-vm:C:/out/rustdesk-${VERSION}-x86_64.msi ./
scp build-vm:C:/out/SHA256SUMS ./
sha256sum -c SHA256SUMS                                       # verify on the way out (R-B2)

# TEARDOWN: stop + delete — no instance survives; the golden image is untouched
virsh destroy build-transient

build-windows.ps1 mirrors upstream's build-for-windows-flutter — $ErrorActionPreference = 'Stop', assert the pinned versions, then vcpkg install --triplet x64-windows-static → build.py --portable --hwcodec --flutter --vram (cargo features hwcodec,vram,flutter; runs flutter build windows --release) → WiX .msi → record the SHA-256 (R-B9). No cross-compile, no code-signing — integrity is the recorded SHA-256, verified on the target over the trusted channel (R-B2).

12.3 The script contract — idempotent, defensive, fail-closed

Every script of §12 (R-B9/B10/B11) obeys one behavioral contract, so the harness is predictable regardless of the host's prior state. Each is a convergent, idempotent step: run it when the system already matches pins.env and it is a no-op (it verifies, says "already satisfied," exits 0); run it on a fresh host and it establishes only what is absent. One rule makes that safe.

Defensive means: complain loudly and die — not "deal with it"

A script may create what is missing; it MUST NOT mutate what is present-but-wrong to force a match. A wrong-version tool, a SHA that doesn't match the pin, a conflicting artifact it did not create — each is an anomaly, and the correct response is a precise, non-zero, loud abort naming expected vs found — never a silent upgrade, downgrade, overwrite, retry, or skip. Reconciling a genuine change (a bumped pin) is an explicit operator action — cleanup.sh the stale state, then re-run — not something a script does behind your back. Every component distrusts both the environment and the outputs of its sibling scripts and re-verifies them before acting (set -euo pipefail / $ErrorActionPreference = 'Stop'; every value from pins.env; success proven, never assumed). "Establish-the-absent" is the only mutation any script performs.

14. Non-goals

  • The rendezvous/relay servers (hbbs/hbbr) — serverless by design. (Self-hosting that pair is the upstream-sanctioned road to sovereign-and-encrypted, but it adds internet-facing ports and key-managed services and surrenders the "connect by address, nothing pre-configured" property the PAKE fork keeps.)
  • A new GUI / protocol redesign — keep the engine; change the handshake, the invariants, and excise the dangerous paths, nothing gratuitous.
  • Defending a compromised endpoint or a password-knowing adversary.
  • Host network segmentation — the fork must not worsen pivot risk but does not implement the host firewall/VLAN.

15. Acceptance criteria

  1. The unconditional PAKE (§10) is the only connection path in both roles — the responder's accept choke point and the initiator's connect path (the unkeyed viewer direct-IP early-returns are deleted, R-S13); the secure parameter, the plaintext path, the trust escape-hatch, and the fingerprint ceremony are deleted; CI proves their absence (R-A6).
  2. All six transports are keyed and all five types + auxiliary channels covered (four reachable in the shipped build; the terminal arm is refused, R-X8); the reconnect, switch-sides, and trusted-device paths are re-gated behind the PAKE — proven by R-A1/A2/A8.
  3. The port-forward set_raw escape is sealed and wire-capture proves the RDP tunnel is ciphertext (R-A9); no transcript yields offline-crackable material (R-S6).
  4. Pre-auth frame length is capped (R-S7); file writes use no-follow opens, with file transfer a single full-filesystem mode and no confinement toggle (R-S8); the whitelist is default-deny and a strong permanent password + password approve-mode are enforced, with the one-time-password path excised so the permanent password is the sole credential and sole PAKE PRS (R-S9, R-X7).
  5. The updater, plugin loader, ConfigureUpdate body, key/server overrides, and the terminal root-PTY are excised from the shipped default; CI proves absence (§8, R-A6, R-X8); the trust anchor equals the pinned value (R-A4).
  6. Crypto verification R-V1–V2 complete, findings resolved; release notes state exactly what review was done and that the in-tree CPace implementation is not independently audited (R-V3).
  7. Unsigned (SHA-256-pinned), source-published, smoke-tested binaries for Android, Windows, and Linux in both roles — produced off GitHub-hosted runners by env-validating per-host scripts (Debian + Android on the Linux box, Windows on a Windows 11 machine; R-B8/B9), having first reproduced upstream 1.4.7's official build (R-B7), --locked from the pinned toolchain (R-B5a), Sciter dropped (R-B6).
  8. The deployed surface is understood port-by-port: the shipped direct-only build exposes exactly one inbound v4 TCP port and zero UDP, makes no unsolicited egress, and has the rendezvous mediator, LAN discovery, and terminal root-PTY absent — on every shipped target (R-D3–R-D7, R-X5, R-X8); a startup self-check asserts the live socket surface (R-A4), so the box is defensible even with no firewall — the firewall is a redundant tripwire, not the gate.
The one-sentence bar

Connecting by address is encrypted and mutually password-authenticated across every transport, type, and content channel, with no plaintext escape and no fetch-and-run or override path left in the building; the program refuses to run if any of that is untrue; and a build exists for every target device — making it, on the wire, as defensible as SSH.

17. Deployment: serverless direct-IP on Linux (no central server, no Docker)

The controlled Ubuntu box runs the fork's own binary as a native systemd service and listens on one TCP port for inbound direct-IP connections, each gated by the PAKE (§10). There is no rendezvous account, no hbbs/hbbr, no relay, and no Docker — the "connect by address, nothing pre-configured" property (R-F3) with the sovereignty of a single owned port.

R-D1MUST Native systemd service; no container, no rendezvous dependency
Ship the verified unit (res/rustdesk.service: ExecStart=/usr/bin/rustdesk --service, Type=simple); the .deb postinst already systemctl enables and starts it. The controlled side MUST NOT require any container runtime or reachable external server to accept a direct connection.
R-D2MUST One port, permanent-password direct-server — the exact headless config
Configure the box through the option CLI (not --config, the excised key/server path, R-X4): 
rustdesk --option direct-server Y                              # always-on direct listener (config.rs:2916)
rustdesk --option verification-method use-permanent-password  # no one-time password (R-X7)
rustdesk --option approve-mode password                       # no click-to-accept (R-S9)
rustdesk --password <strong-permanent-password>               # the sole credential = PAKE PRS (R-P1)
The direct port is not set here — it is a pinned compile-time constant (21118, R-F4), removed from the runtime option set (and hence from the R-S11 IPC-writable keys) because it folds into the PAKE channel binding CI (R-P1). A client then reaches <static-IPv4>:21118 with that password → the mutual PAKE → a keyed session. No second credential exists (R-X7); per-IP rate-limiting is in its clean, real-address case (R-S10).
R-D3MUST Account for every open port — one inbound TCP port, zero UDP, understood not merely allowed
The hardened box's entire inbound surface MUST be exactly one TCP port — the direct-server port (21118, R-D2) — and zero UDP; there is no LAN-discovery (21119, R-X5), no relay range, no ephemeral UDP (R-D4), and no second listener of any kind. Every socket the process binds MUST be enumerable and individually justified (asserted at startup, R-A4). "Secure even on a DMZ" is not a firewall claim: it holds because the binary itself cannot open anything else — R-D4 removes the rendezvous/LAN/KCP sockets, R-D5 binds the listener v4-only, R-D6 silences egress — so an operator's firewall merely agrees with a surface that is already minimal, rather than being the thing that makes it safe. The service runs as root (input injection, virtual display, and login-screen capture require it), so R-S11 least-privilege is bounded by that reality — which is precisely why the failure-mode-C excisions (§8: updater, plugin loader, config rewrite, deep-link writes, and the root-PTY terminal R-X8) are load-bearing: a privileged, internet-exposed service must carry no fetch-and-run, re-home, or shell-spawning path. The host firewall/VLAN and any out-of-band management channel are the environment (a non-goal, §14); this requirement governs the fork's own surface, which must be self-evidently minimal whether or not a firewall is present. The complete socket-by-socket enumeration — every bind site, its original purpose, and the exact mechanism that neutralizes it — is Appendix B.
R-D3aSHOULD Confine the root service with the systemd sandbox the unit currently lacks
R-D3 accepts that the service runs as root (input injection, virtual display, login-screen capture need it) and that least-privilege is "bounded by that reality" — but the shipped unit (res/rustdesk.service) today adds zero kernel-level confinement, so a memory-corruption bug in the root service has the whole machine. The §8 excisions remove the reachable escalation paths; this shrinks the blast radius of anything missed, without touching the root requirement. NoNewPrivileges cannot be set (it would break the sudo -u drop to the --server child), but the unit SHOULD add: a CapabilityBoundingSet dropping caps the service never uses (CAP_SYS_MODULE, CAP_SYS_BOOT, CAP_NET_ADMIN, CAP_AUDIT_*, …); RestrictAddressFamilies=AF_UNIX AF_INET — dropping AF_INET6 (the kernel-policy twin of R-D5's v4-only bind) and AF_NETLINK; ProtectKernelModules, ProtectKernelTunables, RestrictRealtime, LockPersonality, SystemCallArchitectures=native, and SystemCallFilter=@system-service minus @mount/@reboot/@swap; and MemoryDenyWriteExecute if validated against the JIT-free codec path. Each is compatible with the root + input/display/capture reality and is the realistic least-privilege available beyond the excisions.
R-D4MUST The direct-only build excises the rendezvous mediator — no phoning home, no pre-PAKE signaling, no ephemeral UDP
Verified gap: the R-D2 config does not by itself make the box serverless. With direct-server=Y and no stop-service, RendezvousMediator::start_all (rendezvous_mediator.rs:106) still runs its registration loop (:145), calls test_nat_type() (:107) and lan::start_listening() (:133), and get_rendezvous_servers() falls back to the hardcoded rs-ny.rustdesk.com (config.rs:120,:960); stop-service disables the direct listener too, so no config makes the box serverless. The fork MUST ship a direct-only build as the default that compiles the mediator out, so this surface is absent, not firewalled. This also removes the register_pk/key_confirmed liveness dependency, making R-P5's identity-key removal clean.
R-D5MUST Bind the direct listener v4-only; make IPv6 unreachability structural, not configorial
Upstream's tcp::listen_any (hbb_common/src/tcp.rs:226) builds a dual-stack socket (TcpSocket::new_v6() + set_only_v6(false) → binds [::]:21118, accepting v4 and v6), falling back to a v4 socket only if the v6 bind fails (:257). On a static-IPv4 box that v6 face is attack surface with no purpose. The fork MUST bind the direct listener v4-only — add a listen_any_v4 (the existing v4-fallback body, ~6 lines) and call it from the lifted direct_server (R-D4) — so IPv6 unreachability is a property of the binary, not of a host sysctl or an ip6tables rule that can drift. This is "structural > config" applied to address families. Disabling host IPv6 and an ip6tables default-DROP remain sensible outer layers (environment, §14), but the innermost guarantee is the bind. Bonus: a v4-only box makes R-S10's entire IPv6-prefix limiter (connection.rs:3753) dead code — retiring its hardest, highest-collateral-lockout subsystem and its /64 memory-amplification vector by construction.
R-D6MUST Outbound-silent by construction; the firewall is a tripwire, not the gate
A correct direct-only build makes zero unsolicited outbound connections: the rendezvous registration/heartbeat, NAT/STUN probes, and hbbs_http::sync go with the mediator (R-D4); the fetch-and-run updater and its version-check go with R-X1; ConfigureUpdate/custom_server with R-X3/R-X4; and there is no live telemetry (Firebase is compiled out). Steady state: accept inbound on 21118, dial nobody. This inverts the usual posture — the binary is the primary control (it cannot phone home because the code is gone), and an operator's egress firewall is a defense-in-depth tripwire: a uid/cgroup-scoped OUTPUT default-DROP (v4 and v6) on the service account makes any outbound packet from the RustDesk process an IOC — a regression, or a supply-chain dependency that phoned home (the §3.1/§13 chain is large and not cargo audit-clean today, so the backstop is earned). Verified hole to close so the invariant is unconditional: four audit/record POSTs survive R-D4 because they are config-gated, not compiled out — post_conn_audit (connection.rs:1337, fired in on_open pre-authorization: after the fork keys the stream via the PAKE but before any password check, so it triggers on the mere keyed reach of any peer), post_alarm_audit (connection.rs:1421, on whitelist-reject / rate-limit), post_file_audit (:1386), and session-record upload (record_upload.rs:103) — each silent only while the api-server option is empty/public, and each routable through an attacker set_socks proxy were api-server ever written locally (R-S11). The fork MUST hard-pin the api-server/audit/record endpoints empty (or compile these POSTs out) on the controlled build, so "the process makes no egress" is a structural fact, not a configuration accident.

Fit with the hardened host

This is the rebuilt DMZ box: a static IPv4, the fork's RustDesk on a single v4-only TCP port (21118), zero UDP, IPv6 off, no Docker, no rendezvous, and no unsolicited egress (R-D6).
